## 四、关键异常分析

### 4.1 DAG2：并发只有 3

```
Map 1: 0(+3)/487  持续了整个 DAG2 的生命周期
```

487 个 Map 只给了 3 个 Container 并发 → 资源严重不足 → 队列 `dw` 在凌晨 04:12 时非常繁忙（82 个 active apps）。

### 4.2 DAG2：Reducer 从 2000 被缩减为 500

```
初始: Reducer 2: 0/2000
后来: Reducer 2: 0(+500)/500
```

Tez `ShuffleVertexManager` 的 auto parallelism 介入，根据 Map 的实际输出量将 Reducer 从 2000 缩减到 500。

### 4.3 DAG3：487 个 Map 全部 RUNNING 但 0 个完成

```
05:24:20 ~ 07:06:36 → 1小时42分钟
Map 1: 0(+487)/487 → 没有一个 Map 完成
```

**这是最大的异常**。可能原因：
1. **全部 487 个 Map 都卡在读取同一个损坏的文件**（task_000485 对应的文件已知 EOFException）
2. **COS/HDFS 读取卡住**（网络问题或存储后端问题）
3. **所有 Map 都在做长时间 I/O，但还没完成**（数据量大 + I/O 慢）

但考虑到 DAG2 中其他 486 个 Map 都正常完成（每个 1-2 分钟），DAG3 **1 小时 42 分钟一个都没完成**极不正常，高度怀疑是**存储层（COS）在 05:24 之后出现了问题**。

### 4.4 DAG3：Reducer 只有 2 个

DAG2 时 Reducer 是 500（auto 缩减后），DAG3 重试时变成了 2。说明 Hive 在重试时重新计算了 Reducer 数量（可能使用了 DAG2 的统计信息来优化）。
