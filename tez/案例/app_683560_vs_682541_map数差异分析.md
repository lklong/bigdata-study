## 二、关键发现：Map 数差异原因

### 2.1 核心数据

```
682541 Map task = 246
683560 Map task = 487
比值 = 487 / 246 = 1.98x ≈ 正好 2 倍
```

### 2.2 根因分析

两个 app 执行**完全相同的 SQL**，读**同一张表、同一个分区**，且分区数据是 4 月 7 号生产的，**两次执行之间文件数和大小完全一样**。

数据量不变 → Map 数不同 → **唯一的变量是 `availableSlots`**。

根据 `HiveSplitGenerator.initialize()` 源码：

```java
availableSlots = totalResource / taskResource;
// totalResource = getContext().getTotalAvailableResource().getMemory()  ← 队列当前可用总内存
// taskResource = getContext().getVertexTaskResource().getMemory()       ← 单个 task 的内存
```

然后在 `SplitGrouper.estimateBucketSizes()` 中：
```java
numEstimatedTasks = (int)(availableSlots * waves * bucketSize / totalSize);
```

再由 `TezSplitGrouper.getGroupedSplits()` 做边界检查：
- `lengthPerGroup > maxLengthPerGroup` → 被 max-size 约束重算（split 变多）
- `lengthPerGroup < minLengthPerGroup` → 被 min-size 约束重算（split 变少）

**两次执行的差异推断**：

```
682541（00:20，凌晨高峰）:
  队列 dw 同时运行的应用多 → 队列可用资源 totalResource 较少
  → availableSlots 较小（比如 ~30）
  → desiredNumSplits = availableSlots * waves ≈ 30 * 1.7 = 51
  → lengthPerGroup = totalData / 51 > max-size(500MB)
  → 触发 max-size 约束重算 → Map = totalData / max-size ≈ 246

683560（03:51，凌晨低谷）:
  队列 dw 的应用已完成部分 → 队列可用资源 totalResource 较多
  → availableSlots 较大（比如 ~60+）
  → desiredNumSplits = availableSlots * waves ≈ 60 * 1.7 = 102+
  → 可能仍触发 max-size 约束但阈值计算不同
  → 或 min-size 约束介入
  → 最终 Map = 487
```

**但更可能的场景**是：两次执行的 `tez.grouping.max-size` 或 `tez.grouping.min-size` 参数值不同（session 级别设置），或者 `preferredSplitSize` 计算结果不同（受 `blockSize` 和 `minGrouping` 影响）。

### 2.3 精确定位需要 AM 日志

要确认具体是哪个参数导致差异，需要两个 app 的 **AM Container 日志**（Tez DAGAppMaster 日志），其中包含：

```
The preferred split size is XXXXX        ← preferredSplitSize 是否不同
Number of input splits: N. X available slots, Y waves  ← availableSlots 是否不同
Desired splits: N too small/large        ← 是否触发 max/min-size 约束
Desired numSplits: N lengthPerGroup: M   ← 最终分组参数
Number of split groups: K                ← 最终 Map 数
```

**当前日志（RM 端 + HiveServer2 端）不包含这些分片计算参数。**

### 2.4 其他可能因素（概率较低）

| 因素 | 可能性 | 说明 |
|------|--------|------|
| **availableSlots 不同** | 低 | 不同时段集群资源使用率不同，导致 `availableSlots * waves` 不同 → `desiredNumSplits` 不同。但如果 `desiredNumSplits` 太小触发 `maxLengthPerGroup` 约束，最终 Map 数主要由数据量决定 |
| **tez.grouping 参数不同** | 极低 | 同一集群、同一用户、同一 session 名，参数应一致 |
| **小文件数量差异** | 极低 | 数据 7 号已产好不会变，文件数两次执行完全一致 |
| **RM Failover 影响** | 低 | RM failover 不直接影响 Map 数（Map 数在 AM 端计算）|

## 四、682541 失败的根因

### EOFException 分析

```
task_000245 连续 4 次 attempt 全部失败:
java.io.EOFException: Unexpected end of input stream
```

- **task_000245** 是 Map 1 的第 246 个 task（从 0 开始）
- 说明这是**最后一个 task**（Map 1 总共 246 个 task，编号 0~245）
- **EOFException** = 读到了意外的文件结尾
- **根因**：该 task 对应的输入文件在 COS 对象存储上**尾部不完整或读取过程中连接异常**
- 数据 7 号已产好不会变，非 Kafka 写入竞争导致
- 可能原因：①ORC 文件 footer 损坏 ②COS 连接中断导致不完整读 ③文件生产时写入异常导致尾部截断

### 为什么总是 task_000245 失败？

因为它恰好分配到了**包含该损坏/不完整文件**的 split。不管重试多少次，只要读到同一个文件的同一段数据就会 EOFException。

## 五、结论

### Map 数翻倍的根本原因

**数据是 4 月 7 号生产的，两次执行之间文件数和大小完全一样。唯一的变量是 `availableSlots`。**

`availableSlots` 取决于 AM 第一次心跳时 RM 返回的 headroom（当前可用资源快照），由 `YarnTaskSchedulerService.getProgress()` 在 `if(totalResources.getMemory()==0)` 时一次性赋值，之后不再更新。

- **00:26**（682541 AM 启动）：凌晨高峰，dmp 用户的其他 app 占用多 → headroom 小 → availableSlots = a
- **04:11**（683560 AM 启动）：凌晨低谷，部分 app 已结束释放资源 → headroom 大 → availableSlots = b ≈ 2a

availableSlots 同时影响三层：
1. **原始 split 数**：`getSplits(numSplits=(int)(availableSlots*waves))` → numSplits 越大 goalSize 越小 → 原始 splits 越多
2. **desiredNumSplits**：`estimateBucketSizes(availableSlots, waves, ...)` → 线性受 availableSlots 影响
3. **TezSplitGrouper 分组路径**：682541 可能走路径A（max-size 兜底 → Map=246），683560 可能走路径C（desiredNumSplits >= 原始splits数 → 不分组 → Map=487）

**但实际调度时**，683560 的 DAG2 只拿到 3 个并发（队列 82 个 active apps 竞争），分片计算用的是理论资源快照，实际调度用的是竞争后的真实分配，两者脱钩。

### 建议

1. **用 `tez.grouping.max-size` + `min-size` 夹逼** — 让 Map 数由数据量决定，不受 availableSlots 瞬时波动影响
2. **EOFException** — 可能是 COS 对象存储读取异常或文件本身尾部不完整（数据 7 号已产好不会变，非写入竞争导致）
3. **683560 被手动 kill** — 用户发现任务跑了 3 小时没完就手动终止了
