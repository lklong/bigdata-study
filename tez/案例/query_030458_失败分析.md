## 三、根因分析

### 3.1 直接原因：客户端主动取消

```
03:35:45.237  operation.OperationManager: Closing operation
03:35:45.237  operation.Operation: The running operation has been successfully interrupted
```

这是客户端（JDBC driver / 调度系统）发送了 `CloseOperation` RPC 到 HS2，HS2 中断了正在执行的 query。

### 3.2 为什么被取消？调度超时

从提交到取消：`03:04:59` → `03:35:45` = **30 分 46 秒**。

这个时间很可能触发了调度系统（DolphinScheduler/Airflow）的**任务超时**阈值。

### 3.3 为什么跑了 30 分钟还没开始执行实际 DAG？

```
03:04:59.998  App 提交到 YARN
03:30:27.785  PreWarm DAG 才提交（25.5 分钟后！）
```

**AM Container 在 YARN 队列排了 25.5 分钟**才拿到资源启动。

之后 PreWarm DAG（3 个 container）还在跑，实际查询 DAG 还没提交，客户端就取消了。

### 3.4 队列资源紧张证据

从日志中看到同时段有大量其他查询在跑（多个 Map/Reducer 数千级别的 DAG），说明 `dw` 队列在 03:00-03:30 这个时段非常繁忙。

PreWarm DAG 被 kill 时状态：`TotalTasks: 3, Succeeded: 0, Killed: 3` — 3 个 PreWarm container 一个都没跑完。

## 四、异常栈解读

```java
ERROR ql.Driver: FAILED: Hive Internal Error: java.lang.InterruptedException(null)
  at java.lang.Object.wait(Native Method)
  at o.a.hadoop.hive.ql.DriverContext.pollFinished(DriverContext.java:118)
  at o.a.hadoop.hive.ql.Driver.execute(Driver.java:2372)
```

这不是 Bug，是**正常的取消流程**：
1. 客户端调 `CloseOperation`
2. HS2 对 Background Thread 调 `Thread.interrupt()`
3. Background Thread 正在 `DriverContext.pollFinished()` 的 `Object.wait()` 等待 task 完成
4. `wait()` 被 interrupt → 抛 `InterruptedException`
5. Driver 捕获后报 `Operation cancelled`

```
ERROR exec.Task: Failed to execute tez graph.
  HiveException: Operation cancelled
  at TezTask.execute(TezTask.java:214)
```

这是 TezTask 在准备提交实际 DAG 时发现 query 已被取消，抛出 `Operation cancelled`。
