## 四、资源变化核心事件深度分析

### 4.1 RM Failover 影响

**关键证据**：`container_e40_` 中的 `e40` 是 RM epoch，表示当前 RM 是第 40 次启动。

**推断的事件序列**：

1. 应用在某个时间点之前提交并运行（提交日志不在本文件中）
2. **RM 发生过 failover**（从 epoch 39 → 40），导致：
   - RM 丢失了 container 002~751 的 RMContainer 状态
   - AM 的心跳中仍然携带这些旧 container 的释放请求
   - RM 收到释放请求后发现 container 不属于该应用 → 865 条 FAILURE
3. RM failover 后，container 752+ 被重新注册或新分配

### 4.2 AM Released Container FAILURE 详解

```
时间: 05:14:20 ~ 07:06:36（持续 1小时52分钟）
数量: 865 条
涉及: container 002 ~ 943
```

**为什么持续 1.5 小时？**

AM 通过心跳（allocate RPC）向 RM 发送释放请求。每次心跳间隔 1~3 秒，但：
- AM 在每次心跳中会**批量发送**多个释放请求
- RM 逐个处理后逐个返回 FAILURE
- AM 可能会**重试**释放（因为收到 FAILURE 后不确定是否真的释放了）
- 直到应用结束时仍在尝试

### 4.3 批量终止事件（07:06:36）

在 **0.1 秒内**（07:06:36.861 ~ 07:06:36.955），488 个 container 状态发生变化：

```
07:06:36.861  ─ 开始批量 COMPLETED（NM 汇报的正常完成）
07:06:36.949  ─ 开始批量 RELEASED（AM 主动释放的）
07:06:36.949  ─ 同时出现 487 条 "RMContainer doesn't exist"
07:06:36.955  ─ 批量状态变化结束
```

**0.1秒内 488 个 container 同时变化的原因**：
- 这不是 container 在 0.1 秒内完成的
- 而是 **AM 在一次心跳中批量发送了所有 container 的完成/释放通知**
- RM 在一个心跳处理周期内批量处理了这些事件

### 4.4 RMContainer doesn't exist（487 条）

```
"Container container_xxx completed with event RELEASED, 
 but corresponding RMContainer doesn't exist."
```

**根因**：同一个 container 被**双重处理**：
1. NM 先汇报 container COMPLETED → RM 将 RMContainer 标记为 COMPLETED 并清理
2. AM 随后发送 RELEASE 请求 → RM 找不到 RMContainer → 打印此日志

这 487 个 container（752~1275）和前面 157 个 COMPLETED + 331 个 RELEASED 有重叠，说明部分 container 同时收到了 NM 的完成通知和 AM 的释放请求。
