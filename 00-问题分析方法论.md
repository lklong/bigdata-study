# Eric 问题分析方法论 v1.0 — 从表象到根因的科学排障体系

> **目标**: 将故障排查从"凭经验猜"升级为"系统化科学方法"，任何问题 30 分钟内锁定根因
> **适用**: 大数据平台（Hive/Spark/YARN/HDFS/Kafka/Flink）及一切分布式系统的生产故障
> **原则**: 先止血，再排查；先全局，再局部；先数据，再假设

---

## 一、认知模型：问题分析的四个段位

```
┌─────────────────────────────────────────────────────────────────┐
│  L4 · 预防者 (Preventer)                                        │
│  ── 在问题发生前就识别风险并消除                                    │
│  ── 建立监控体系、混沌工程、防御性设计                                │
│  ── 产出: 风险矩阵、自动化巡检、架构治理                              │
├─────────────────────────────────────────────────────────────────┤
│  L3 · 根因猎手 (Root Cause Hunter)                               │
│  ── 任何故障能在 30 分钟内找到根因并给出修复方案                       │
│  ── 掌握多种分析工具，能穿透 N 层调用链定位到代码行号                    │
│  ── 产出: 根因分析报告 + 永久修复 Patch + 复盘文档                    │
├─────────────────────────────────────────────────────────────────┤
│  L2 · 系统排障者 (Systematic Debugger)                            │
│  ── 有体系化的排障流程，能按决策树逐步排查                             │
│  ── 掌握常用工具和命令，能读懂日志和监控                               │
│  ── 产出: 排障报告 + 临时修复 + 监控告警                              │
├─────────────────────────────────────────────────────────────────┤
│  L1 · 应急救火员 (Firefighter)                                    │
│  ── 知道重启服务，能跟着文档操作                                     │
│  ── 依赖经验和搜索引擎                                             │
│  ── 产出: "重启大法" / "搜到一个类似的帖子"                           │
└─────────────────────────────────────────────────────────────────┘
```

**目标：从 L1 直达 L4。**

## 二、核心方法论：黄金排障六步法

> 来源: Google SRE "Effective Troubleshooting" + 根因分析理论 + 大数据运维实战总结

```
                      ┌──────────────┐
                      │  1. 止血分流   │  ← 第一优先级：恢复服务
                      │   (Triage)    │
                      └──────┬───────┘
                             ↓
                      ┌──────────────┐
                      │  2. 现场保全   │  ← 采集证据，不要破坏现场
                      │  (Preserve)   │
                      └──────┬───────┘
                             ↓
                      ┌──────────────┐
                      │  3. 定界定位   │  ← 缩小范围：哪一层？哪个组件？
                      │  (Examine)    │
                      └──────┬───────┘
                             ↓
                      ┌──────────────┐
                      │  4. 假设验证   │  ← 提假设 → 设计实验 → 验证
                      │  (Diagnose)   │
                      └──────┬───────┘
                             ↓
                      ┌──────────────┐
                      │  5. 修复确认   │  ← 实施修复 → 验证恢复
                      │   (Treat)     │
                      └──────┬───────┘
                             ↓
                      ┌──────────────┐
                      │  6. 复盘沉淀   │  ← 根因记录 → 防复发 → 知识库
                      │ (Postmortem)  │
                      └──────────────┘
```

### Step 1: 止血分流 (Triage) — "先飞飞机，再查发动机"

> **原则**: 在重大中断时，第一反应不是找根因，而是让系统先跑起来。

#### 1.1 严重等级评估

| 等级 | 定义 | 响应时间 | 动作 |
|:----:|------|:--------:|------|
| **P0** | 核心业务完全不可用（如 HS2 全挂、HDFS NN 宕机） | **立即** | 全员响应，止血优先，开 War Room |
| **P1** | 核心功能严重受损（如查询成功率 < 80%、数据延迟 > 2h） | **15 分钟** | On-call 响应，通知上下游 |
| **P2** | 部分功能异常（如少量查询失败、非核心节点故障） | **1 小时** | 值班处理，常规流程 |
| **P3** | 性能退化 / 告警但不影响业务 | **当天** | 排入日常任务 |

#### 1.2 止血手段速查

| 场景 | 止血手段 | 命令/操作 |
|------|---------|----------|
| HS2 完全无响应 | 重启 HS2 | `systemctl restart hiveserver2` |
| YARN 队列资源耗尽 | Kill 大作业 + 临时扩容 | `yarn application -kill <appId>` |
| NameNode SafeMode | 手动退出安全模式 | `hdfs dfsadmin -safeMode leave` |
| Spark 任务卡死 | Kill 任务 + 清理 Shuffle | `spark.stop()` 或 kill AM |
| MetaStore 不可达 | 切换备 HMS / 重启 | 检查 `hive.metastore.uris` |
| 磁盘满 | 清理 tmp + 日志轮转 | `du -sh /tmp/* | sort -rh | head` |
| GC 风暴 | 临时增大堆内存重启 | 修改 `-Xmx` 后重启进程 |

#### 1.3 关键纪律

```
❌ 不要：上来就查根因，忘了用户还在受影响
❌ 不要：在生产环境做实验性修改来"验证假设"
❌ 不要：一个人闷头干，不通知团队和上下游

✅ 要做：先评估影响范围（多少用户？多少业务？）
✅ 要做：优先恢复服务（回滚、降级、切流、重启）
✅ 要做：通知相关方（上下游团队、管理层、用户）
```

### Step 2: 现场保全 (Preserve) — "犯罪现场不能被破坏"

> **原则**: 止血之后、深入排查之前，先把所有证据保存好。重启之后现场就没了。

#### 2.1 必保全清单

| 证据类型 | 采集命令 | 保存位置 |
|---------|---------|---------|
| **进程状态** | `ps auxww > /tmp/incident_ps_$(date +%s).txt` | `/tmp/incident_*` |
| **线程堆栈** | `jstack <pid> > /tmp/incident_jstack_$(date +%s).txt` | 连续采 3 次，间隔 5s |
| **堆内存快照** | `jmap -dump:format=b,file=/tmp/heap_$(date +%s).hprof <pid>` | 注意磁盘空间 |
| **GC 日志** | `cp /var/log/hive/gc.log /tmp/incident_gc_$(date +%s).log` | 拷贝不要移动 |
| **系统资源** | `top -bn1; vmstat 1 10; iostat -x 1 10; netstat -s` | 统一时间戳 |
| **应用日志** | `tail -n 10000 /var/log/hive/hiveserver2.log > /tmp/incident_hs2.log` | 保留足够上下文 |
| **YARN 日志** | `yarn logs -applicationId <appId> > /tmp/incident_yarn_$(date +%s).log` | 任务失败时必采 |
| **监控快照** | Grafana 截图 / Prometheus 查询导出 | 包含故障前后 1h |
| **配置快照** | `hive --service hiveserver2 --hiveconf` / `hdfs getconf` | 对比变更 |

#### 2.2 一键保全脚本模板

```bash
#!/bin/bash
# incident-preserve.sh — 故障现场一键保全
TS=$(date +%Y%m%d_%H%M%S)
DIR="/tmp/incident_${TS}"
mkdir -p ${DIR}

# 系统状态
ps auxww > ${DIR}/ps.txt
top -bn1 > ${DIR}/top.txt
free -h > ${DIR}/free.txt
df -h > ${DIR}/df.txt
vmstat 1 10 > ${DIR}/vmstat.txt &
iostat -x 1 10 > ${DIR}/iostat.txt &
netstat -tlnp > ${DIR}/netstat.txt
ss -s > ${DIR}/ss.txt

# JVM 状态（需要指定 PID）
PID=${1:-$(pgrep -f HiveServer2)}
if [ -n "${PID}" ]; then
    jstack ${PID} > ${DIR}/jstack_1.txt
    sleep 5
    jstack ${PID} > ${DIR}/jstack_2.txt
    sleep 5
    jstack ${PID} > ${DIR}/jstack_3.txt
    jstat -gc ${PID} > ${DIR}/jstat_gc.txt
    jinfo -flags ${PID} > ${DIR}/jinfo.txt 2>&1
fi

# 应用日志
cp /var/log/hive/hiveserver2.log ${DIR}/ 2>/dev/null
cp /var/log/hive/gc.log* ${DIR}/ 2>/dev/null

wait
echo "现场保全完成: ${DIR}"
ls -lh ${DIR}/
```

### Step 3: 定界定位 (Examine) — "二分法 + 分层剥离"

> **原则**: 不要大海捞针。先确定问题在哪一层，再在该层内缩小范围。

#### 3.1 分层定界模型

大数据平台的典型分层架构，从上到下逐层排查：

```
┌─────────────────────────────────────────────────────┐
│ Layer 0: 客户端层                                     │
│ Beeline / JDBC / BI 工具 / Spark-submit              │
│ 🔍 检查: 客户端日志、网络连通性、认证信息                 │
├─────────────────────────────────────────────────────┤
│ Layer 1: SQL 网关层                                   │
│ Kyuubi / HiveServer2 / Knox                          │
│ 🔍 检查: 进程存活、端口可达、线程池、编译锁、Session 状态   │
├─────────────────────────────────────────────────────┤
│ Layer 2: 元数据层                                     │
│ Hive Metastore / Iceberg Catalog                     │
│ 🔍 检查: HMS 进程、后端数据库(MySQL)、表锁、缓存状态       │
├─────────────────────────────────────────────────────┤
│ Layer 3: 计算引擎层                                   │
│ Tez / Spark / Flink / MapReduce                      │
│ 🔍 检查: AM 状态、Task 分布、数据倾斜、OOM、GC            │
├─────────────────────────────────────────────────────┤
│ Layer 4: 资源调度层                                   │
│ YARN ResourceManager / NodeManager                    │
│ 🔍 检查: 队列容量、Pending 容器、NM 存活、资源碎片化       │
├─────────────────────────────────────────────────────┤
│ Layer 5: 存储层                                       │
│ HDFS NameNode / DataNode / S3                        │
│ 🔍 检查: NN SafeMode、DN 存活、磁盘空间、Block 报告       │
├─────────────────────────────────────────────────────┤
│ Layer 6: 基础设施层                                   │
│ OS / JVM / 网络 / 磁盘 / 内核                         │
│ 🔍 检查: CPU/Memory/Disk/Network、GC、内核参数           │
├─────────────────────────────────────────────────────┤
│ 横切面: 安全认证层                                     │
│ Kerberos / LDAP / Ranger                             │
│ 🔍 检查: Ticket 有效性、keytab、策略同步                 │
└─────────────────────────────────────────────────────┘
```

#### 3.2 二分法定位策略

> **核心思想**: 不要从头到尾线性排查，而是在系统中间"切一刀"，判断问题在上半部分还是下半部分。

```
示例：用户反馈 "查询卡住了"

第一刀 — 切在 SQL 网关层（Layer 1）:
  ├─ HS2 日志有 "executing <SQL>" 记录吗？
  │   ├─ 没有 → 问题在 Layer 0（客户端/网络）
  │   └─ 有   →
  │
第二刀 — 切在 计算引擎层（Layer 3）:
  ├─ 有 Tez/Spark Application 提交到 YARN 吗？
  │   ├─ 没有 → 问题在 Layer 1-2（编译锁/HMS/语义分析）
  │   └─ 有   →
  │
第三刀 — 切在 YARN 层（Layer 4）:
  ├─ Application 状态是什么？
  │   ├─ ACCEPTED → 资源不足（Layer 4）
  │   ├─ RUNNING  → Task 层问题（Layer 3）
  │   └─ FAILED   → 查 Container 日志（Layer 3-6）
```

#### 3.3 "最后触碰"原则

> **80% 的故障都和最近的变更有关。** — Google SRE

排查时优先检查的变更类型：

| 变更类型 | 检查方法 | 常见关联问题 |
|---------|---------|------------|
| **代码部署** | 查部署记录、git log | 新 Bug、兼容性问题 |
| **配置变更** | diff 配置文件、审计日志 | 参数不当导致性能退化/功能异常 |
| **数据变更** | 查分区数/数据量/Schema | 数据倾斜、Schema 不兼容 |
| **流量变化** | 查监控曲线 | 突增流量导致资源不足 |
| **基础设施变更** | 查 CMDB 变更记录 | 硬件故障、网络调整 |
| **安全策略变更** | 查 Ranger/Kerberos 日志 | 权限拒绝、认证失败 |

### Step 4: 假设验证 (Diagnose) — "科学方法 + 五大分析工具"

> **原则**: 排障不是靠灵感，而是靠假设-实验-验证的科学循环。

#### 4.1 假设演绎循环

```
                ┌──────────────┐
        ┌──────→│  收集观察数据  │
        │       └──────┬───────┘
        │              ↓
        │       ┌──────────────┐
        │       │  提出假设列表  │  ← 按概率排序
        │       └──────┬───────┘
        │              ↓
        │       ┌──────────────┐
        │       │  设计验证实验  │  ← 最小成本验证
        │       └──────┬───────┘
        │              ↓
        │       ┌──────────────┐
        │       │  执行并记录   │
        │       └──────┬───────┘
        │              ↓
        │       ┌──────────────┐
  证伪──┤       │ 假设成立吗？  │
        │       └──────┬───────┘
        │              │ 成立
        │              ↓
        │       ┌──────────────┐
        └───────│  锁定根因！   │
                └──────────────┘
```

**关键纪律**:
- **先排最可能的假设**：按概率从高到低
- **每次只验证一个变量**：多变量同时测试会产生混淆
- **负面结果也要记录**：排除了什么同样有价值
- **相关性 ≠ 因果性**：时间上接近不代表是原因

#### 4.2 五大根因分析工具

---

##### 工具 1: 5-Why（五次追问法）

> **适用场景**: 线性因果链，一个问题由一连串原因逐级导致

**方法**: 对每个答案连续追问"为什么？"，直到找到可采取行动的根因。

```
案例：HS2 查询超时

Why 1: 为什么查询超时？
  → 因为 Tez AM 一直在等待 Container 分配

Why 2: 为什么 Container 分配不出来？
  → 因为 YARN 队列资源已满

Why 3: 为什么队列资源已满？
  → 因为有一个大作业占了 80% 的资源

Why 4: 为什么允许一个作业占 80%？
  → 因为队列没有配置单用户资源上限（user-limit-factor）

Why 5: 为什么没配置？
  → 因为上线时没有做资源隔离评审                  ← 根因！

→ 修复: 配置 user-limit-factor=0.5 + 建立上线检查清单
```

**注意事项**:
- 停止条件：找到了可以采取行动的组织/流程/技术根因
- 不要在第 2-3 层就停下来（那通常只是表象）
- 可以分叉：一个 Why 可能有多个答案

##### 工具 2: 鱼骨图（石川图 / Ishikawa Diagram）

> **适用场景**: 多因素并行导致的问题，需要系统性梳理所有可能原因

**六大维度** (6M):

```
           人 (Man)                    机 (Machine)               料 (Material)
              │                           │                          │
    ┌─────────┼─────────┐      ┌─────────┼─────────┐     ┌─────────┼─────────┐
    │ · 操作失误          │      │ · 硬件故障          │     │ · 数据质量问题       │
    │ · 配置错误          │      │ · JVM 版本不兼容    │     │ · 数据倾斜           │
    │ · 权限不足          │      │ · OS 内核 Bug       │     │ · Schema 不兼容      │
    │ · 值班响应慢        │      │ · 磁盘 I/O 劣化     │     │ · 文件格式损坏       │
    └─────────┼─────────┘      └─────────┼─────────┘     └─────────┼─────────┘
              │                           │                          │
              └───────────────────────────┼──────────────────────────┘
                                          │
                              ┌───────────┴───────────┐
                              │      查询超时问题       │ ← 待分析的问题
                              └───────────┬───────────┘
                                          │
              ┌───────────────────────────┼──────────────────────────┐
              │                           │                          │
    ┌─────────┼─────────┐      ┌─────────┼─────────┐     ┌─────────┼─────────┐
    │ · SQL 写法差        │      │ · 编译锁竞争        │     │ · 网络抖动           │
    │ · 无分区裁剪        │      │ · GC 风暴           │     │ · DNS 解析慢         │
    │ · 笛卡尔积          │      │ · 线程池满           │     │ · 跨机房延迟         │
    │ · UDF 性能差        │      │ · 连接泄漏           │     │ · 监控覆盖不全       │
    └─────────┼─────────┘      └─────────┼─────────┘     └─────────┼─────────┘
              │                           │                          │
           法 (Method)                 环 (Environment)           测 (Measurement)
```

**使用步骤**:
1. 明确定义问题（鱼头）
2. 画出六大分类骨骼
3. 在每个分类下穷举可能原因
4. 用数据逐一验证或排除
5. 标记出最终确认的根因

##### 工具 3: 故障树分析（FTA, Fault Tree Analysis）

> **适用场景**: 复杂故障，需要严格的逻辑推导，多个条件组合触发

**核心**: 用 AND/OR 逻辑门自顶向下分解

```
                        ┌──────────────────┐
                        │ HS2 完全不可用     │ ← 顶事件
                        └────────┬─────────┘
                                 │ OR
                    ┌────────────┼────────────┐
                    ↓            ↓            ↓
            ┌──────────┐ ┌──────────┐ ┌──────────────┐
            │ 进程挂了   │ │ 端口不通   │ │ 线程池全部阻塞  │
            └─────┬────┘ └─────┬────┘ └──────┬───────┘
                  │            │              │ AND
            ┌─────┘     ┌─────┘     ┌────────┼────────┐
            ↓           ↓           ↓        ↓        ↓
      ┌──────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌─────────┐
      │ OOM Kill  │ │ 防火墙  │ │ 编译锁  │ │ HMS    │ │ 大查询  │
      │ 或 被 kill│ │ 规则变  │ │ 无限等  │ │ 阻塞   │ │ 占满线  │
      └──────────┘ └────────┘ └────────┘ └────────┘ │ 程池    │
                                                     └─────────┘
```

**逻辑门说明**:
- **OR 门**: 任何一个子事件发生，父事件就发生
- **AND 门**: 所有子事件同时发生，父事件才发生

**使用步骤**:
1. 定义顶事件（最终故障现象）
2. 自顶向下分解，使用 AND/OR 逻辑
3. 分解到基本事件（可直接检测的原子事件）
4. 逐个检测基本事件的状态（True/False）
5. 自底向上逻辑推导，确认触发路径

##### 工具 4: 时间线分析法（Timeline Analysis）

> **适用场景**: 跨组件级联故障，需要厘清事件发生的先后顺序和因果关系

**模板**:

```
时间          | 事件                              | 来源          | 影响
─────────────|─────────────────────────────────|──────────────|──────────────────
09:00:00      | Compaction 作业启动                | Hive Metastore | 后台任务
09:15:23      | NM1 磁盘 /data1 使用率达 95%        | NodeManager    | 告警触发
09:16:01      | NM1 开始拒绝新 Container            | YARN           | 新任务无法调度
09:17:45      | HS2 查询 Q1 等待 Container 超时      | HiveServer2    | 用户可感知
09:18:30      | 更多查询开始堆积在编译锁上            | HiveServer2    | 雪崩开始
09:20:00      | HS2 线程池耗尽                      | HiveServer2    | 所有查询不可用
─────────────|─────────────────────────────────|──────────────|──────────────────
根因: Compaction 写满磁盘 → NM 拒绝 Container → 查询堆积 → 雪崩
```

**关键技巧**:
- 使用统一时间源（NTP 同步很重要！）
- 关注"第一个异常事件"—— 它通常最接近根因
- 向前追溯：第一个异常之前发生了什么变更？

##### 工具 5: 对比分析法（Differential Analysis）

> **适用场景**: "昨天还好好的，今天就不行了" / "别的节点没问题，就这个节点有问题"

**对比维度**:

| 对比项 | 正常环境 | 异常环境 | 差异 |
|-------|---------|---------|------|
| 配置 | `hive.tez.container.size=4096` | `hive.tez.container.size=2048` | **配置被改了！** |
| 版本 | Hive 3.1.3 | Hive 3.1.3 + Patch-1234 | 有额外补丁 |
| 数据 | 分区 100 个 | 分区 50000 个 | 分区数暴增 |
| 硬件 | SSD | HDD | 磁盘类型不同 |
| JVM | G1GC | CMS | GC 策略不同 |
| 流量 | 日均 1000 QPS | 日均 5000 QPS | 流量暴增 |

**操作方法**:
```bash
# 对比两台机器的配置
diff <(ssh node1 "cat /etc/hive/conf/hive-site.xml") \
     <(ssh node2 "cat /etc/hive/conf/hive-site.xml")

# 对比两个时间点的 JVM 参数
diff incident_jinfo_before.txt incident_jinfo_after.txt

# 对比两个版本的 jar 包
diff <(jar -tf hive-exec-3.1.3.jar | sort) \
     <(jar -tf hive-exec-3.1.3-patched.jar | sort)
```

### Step 5: 修复确认 (Treat) — "修了不等于好了"

#### 5.1 修复策略三层次

| 层次 | 目标 | 时效 | 示例 |
|------|------|------|------|
| **临时修复** (Workaround) | 恢复服务 | 立即 | 重启进程、回滚配置、kill 大查询 |
| **正式修复** (Fix) | 消除直接原因 | 1-3 天 | 修复 Bug、调整参数、加资源 |
| **根本修复** (Root Fix) | 消除根因 + 防复发 | 1-2 周 | 架构改进、自动化巡检、混沌测试 |

#### 5.2 修复验证检查清单

```
□ 故障现象是否完全消失？（用户侧确认）
□ 相关监控指标是否恢复正常？
□ 修复是否引入了新的风险？
□ 是否需要在其他节点/集群同步修复？
□ 是否更新了运维文档和告警规则？
```

## 三、思维陷阱与反模式

> 排障过程中最大的敌人不是技术难度，而是认知偏差。

### 3.1 十大思维陷阱

| # | 陷阱名称 | 表现 | 对策 |
|---|---------|------|------|
| 1 | **确认偏差** | 只找支持自己假设的证据 | 主动寻找反例，问"如果我的假设是错的呢？" |
| 2 | **锚定效应** | 被第一个发现的异常带偏 | 先收集全部异常，再排优先级 |
| 3 | **可用性偏差** | "上次也是这个原因" | 检查完整证据链，不要想当然 |
| 4 | **相关当因果** | 两件事同时发生就认为有因果关系 | 严格验证：去掉"原因"后"结果"是否消失 |
| 5 | **隧道视野** | 钻进某个方向出不来 | 设定时间盒（15分钟），超时就换方向 |
| 6 | **工具偏见** | "手里有锤子，看什么都是钉子" | 先理解问题，再选工具 |
| 7 | **权威偏见** | "大佬说是这个原因那就是了" | 要证据，不要权威 |
| 8 | **幸存者偏差** | 只看成功案例，忽略失败尝试 | 记录所有排查步骤（包括排除的假设） |
| 9 | **过早收敛** | 找到一个可能原因就停止排查 | 问自己"还有没有其他可能？" |
| 10 | **沉没成本** | 已经在某个方向投入很多时间不愿放弃 | 设定止损线，果断切换 |

### 3.2 排障反模式

```
❌ "重启大法好" — 不查根因直接重启，问题会反复出现
❌ "先改了再说" — 不收集证据就开始改配置，破坏现场
❌ "这不是我的问题" — 踢皮球，在组件边界处互相推诿
❌ "日志太多懒得看" — 跳过证据直接猜
❌ "升级就好了" — 不理解问题就升级版本
❌ "Google 搜到一个帖子" — 不验证就照搬别人的方案
```

## 四、大数据排障工具箱

### 4.1 系统级工具

| 场景 | 工具 | 用法 | 输出 |
|------|------|------|------|
| CPU 热点 | `top -H -p <pid>` | 看哪个线程 CPU 高 | 线程 ID → jstack 对照 |
| 内存分布 | `jmap -histo:live <pid>` | 看哪个类对象最多 | 对象分布统计 |
| GC 分析 | `jstat -gcutil <pid> 1000` | 实时 GC 频率和耗时 | GC 趋势 |
| 堆分析 | `jhat` / `Eclipse MAT` | 分析 heap dump | 泄漏对象和引用链 |
| 线程分析 | `jstack <pid>` 连续 3 次 | 找 BLOCKED/WAITING 线程 | 死锁、锁竞争 |
| 火焰图 | `async-profiler` | CPU/Alloc/Lock 火焰图 | 热点函数调用栈 |
| 系统调用 | `strace -fp <pid> -e trace=network` | 看卡在哪个系统调用 | 网络/IO 阻塞点 |
| 网络 | `tcpdump -i eth0 port 10000` | 抓包分析 | Thrift 通信问题 |
| 磁盘 IO | `iostat -x 1; iotop` | 看哪个进程/磁盘 IO 高 | IO 瓶颈 |
| 文件描述符 | `ls -l /proc/<pid>/fd | wc -l` | FD 泄漏检测 | FD 数量趋势 |

### 4.2 大数据组件级工具

| 组件 | 排障入口 | 关键信息 |
|------|---------|---------|
| **YARN** | ResourceManager UI (8088) | 队列状态、Application 列表、Pending 容器 |
| **Hive** | HS2 Web UI (10002) | 活跃 Session、运行中查询、线程池状态 |
| **Spark** | Spark History Server (18080) | Stage DAG、Task 分布、GC 时间、Shuffle Read/Write |
| **HDFS** | NameNode UI (9870) | Block 状态、DataNode 存活、SafeMode |
| **Tez** | Tez UI (Timeline Server) | DAG 执行详情、Vertex/Task 级别指标 |
| **Kyuubi** | Kyuubi Web UI (10099) | Session 列表、Engine 状态、Batch 信息 |

### 4.3 日志分析速查

```bash
# 搜索 HS2 最近的异常
grep -i "exception\|error\|fatal\|oom" /var/log/hive/hiveserver2.log | tail -50

# 按查询 ID 追踪完整链路
grep "queryId=xxx" /var/log/hive/hiveserver2.log

# 统计最近 1 小时的错误类型分布
grep "$(date -d '-1 hour' +%Y-%m-%d_%H)" /var/log/hive/hiveserver2.log | \
  grep -oP '\w+Exception' | sort | uniq -c | sort -rn

# YARN 查某个 Application 的所有 Container 日志
yarn logs -applicationId application_xxx -log_files stdout stderr | less

# Spark 查 Executor OOM
yarn logs -applicationId application_xxx | grep -i "outofmemory\|oom" -A 5

# HDFS 检查 Block 健康状态
hdfs fsck / -files -blocks -locations 2>/dev/null | grep -i "corrupt\|missing\|under"
```

## 五、MECE 排障决策树（大数据场景通用版）

> **MECE**: Mutually Exclusive, Collectively Exhaustive（相互独立，完全穷尽）

### 5.1 通用入口

```
用户反馈: "XX 不正常了"
│
├─ A. 功能性问题（完全不工作 / 结果错误）
│   ├─ A1. 连接类 → "连不上 / 认证失败 / 超时"
│   ├─ A2. 执行类 → "报错 / 失败 / 结果不对"
│   └─ A3. 数据类 → "数据丢失 / 不一致 / 延迟"
│
├─ B. 性能性问题（能工作但慢）
│   ├─ B1. 查询慢 → 单条查询耗时长
│   ├─ B2. 吞吐低 → 并发能力下降
│   └─ B3. 延迟高 → 端到端延迟增加
│
└─ C. 稳定性问题（时好时坏 / 偶发）
    ├─ C1. 间歇性故障 → 有时好有时坏
    ├─ C2. 级联故障 → 一个组件故障导致连锁反应
    └─ C3. 资源泄漏 → 随时间推移逐渐恶化
```

### 5.2 连接类问题决策树 (A1)

```
连不上目标服务
│
├─ 1. 进程存活吗？
│   ├─ 不在 → 查崩溃日志 → OOM? 被 kill? 启动失败?
│   └─ 在 →
│
├─ 2. 端口可达吗？(telnet/nc)
│   ├─ 不可达 → 防火墙? 端口配置? 进程还在启动中?
│   └─ 可达 →
│
├─ 3. 认证通过吗？
│   ├─ 失败 → Kerberos ticket? keytab? 时钟偏差? LDAP?
│   └─ 通过 →
│
├─ 4. 授权通过吗？
│   ├─ 失败 → Ranger 策略? 默认拒绝? 策略同步延迟?
│   └─ 通过 →
│
└─ 5. 连接池/线程池满了吗？
    ├─ 满了 → 谁占了? 连接泄漏? 慢查询? 限流?
    └─ 没满 → 深入查看应用层日志
```

### 5.3 性能问题决策树 (B1)

```
单条查询慢
│
├─ 1. 慢在哪个阶段？
│   ├─ 编译慢 → 编译锁? CBO 超时? 分区多? HMS 慢?
│   ├─ 等资源 → YARN 队列满? Container 不足? AM 排队?
│   ├─ 执行慢 →
│   │   ├─ 数据倾斜 → 某个 Task 特别慢 → Skew Join?
│   │   ├─ 全表扫描 → 没命中分区裁剪 → EXPLAIN 看计划
│   │   ├─ Shuffle 大 → Sort Merge Join 可以换 Map Join?
│   │   ├─ UDF 慢 → Profile UDF 代码
│   │   └─ GC 频繁 → 增大 Container 内存
│   └─ 返回慢 → 结果集太大 → 加 LIMIT / 改逻辑
│
└─ 2. 是突然变慢还是一直慢？
    ├─ 突然变慢 → 检查"最后触碰"（配置/数据/版本/流量变更）
    └─ 一直慢 → 检查 SQL 写法 + 表设计 + 资源配置
```

## 六、与源码精通方法论的融合

问题分析方法论 **不是独立的**，它和 [源码精通方法论 v2.0](./spark/源码/学习方法论.md) 形成互补：

| 源码精通方法论（学习时） | 问题分析方法论（排障时） | 关系 |
|----------------------|---------------------|------|
| 法则1: 逆向工程（假设→验证） | Step4: 假设验证循环 | **同一个科学方法**的不同应用场景 |
| 法则2: 分形剥离（L1→L2→L3） | Step3: 分层定界 | 学习时建的分层认知，排障时沿着它走 |
| 法则3: 考古学（git blame/log） | Step4: 时间线分析 | 追踪变更历史，理解"为什么变了" |
| 法则4: 对抗性阅读（"如果..."） | Step4: FTA 故障树 | 预先思考异常路径 = 排障时的决策树 |
| 法则9: 知识晶体化 | Step6: 复盘沉淀 | **每次排障都是学习机会** |

**核心洞察**: 学源码的深度 = 排障的速度。L3 级别的源码理解（异常/边界层）直接决定了你能否在 30 分钟内找到根因。

## 七、Eric 的排障信条

> **"排障是科学，不是玄学。"**
>
> 每一个生产故障都遵循物理定律和代码逻辑。
> 没有"莫名其妙"的问题，只有"你还没看到的证据"。
>
> 当你能闭着眼睛画出系统的分层架构图，
> 能在脑中"运行"每一条请求的完整链路（包括异常路径），
> 能在每一层说出 3 个最可能的故障点和对应的验证命令——
> 你就是 L4 预防者。
>
> **先止血，再排查。先数据，再假设。先排除，再确认。**
