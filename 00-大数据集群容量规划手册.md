# 大数据集群容量规划手册

> 怎么算需要多少机器、多少内存、多少磁盘

---

## 一、HDFS 容量规划

### 存储容量
```
公式: 需要的裸存储 = 数据量 × 副本数 × 1.3(预留30%余量)

例: 100TB 数据, 3副本
需要: 100 × 3 × 1.3 = 390TB 裸存储

如果每台 DN 12块 × 4TB = 48TB
需要: 390 / 48 ≈ 9 台 DataNode
```

### NameNode 内存
```
公式: NN堆内存 = 文件数 × 300B + Block数 × 200B + 2GB(基础开销)

例: 1亿文件, blocksize=128MB, 平均文件200MB → 约1.6亿Block
需要: 1亿 × 300 + 1.6亿 × 200 + 2GB ≈ 62GB
建议: -Xmx80g (留余量)
```

## 二、YARN 容量规划

### 计算资源
```
每台 NodeManager 可分配:
  Memory = 物理内存 × 0.8 (留20%给OS)
  vCores = 物理核数 × 0.8

例: 128GB内存, 32核
  NM Memory = 128 × 0.8 = 102GB
  NM vCores = 32 × 0.8 = 25

Spark Executor 配置: 4core, 8GB
  每台可跑: min(102/8, 25/4) = min(12, 6) = 6个Executor
  10台 = 60个Executor并行

需要多少台:
  目标并行Executor数 / 每台Executor数 = 机器台数
  100 Executor / 6 = 17台
```

## 三、Spark 任务资源估算

```
输入数据量: D (GB)
目标完成时间: T (秒)

估算 Executor 数:
  1. 单 Executor 处理速度 ≈ 50-200 MB/s (取决于任务复杂度)
  2. 需要的总吞吐 = D × 1024 / T (MB/s)
  3. Executor 数 = 总吞吐 / 单Executor速度

例: 1TB数据, 30分钟完成, 单Executor 100MB/s
  总吞吐 = 1024 × 1024 / 1800 ≈ 583 MB/s
  Executor 数 = 583 / 100 ≈ 6个 (但实际加 Shuffle 开销 × 2 = 12个)

Shuffle 磁盘:
  Shuffle 数据量 ≈ 输入数据 × 0.5 ~ 2 (取决于 Join/GroupBy)
  需要的本地磁盘 = Shuffle数据 / Executor数 + 余量
```

## 四、Flink 容量规划

```
输入 QPS: Q (条/秒)
单条大小: S (bytes)
处理逻辑复杂度系数: C (简单=1, 中等=2, 复杂=4)

TaskManager 数:
  单 TM Slot 处理能力 ≈ 10,000 ~ 50,000 条/s (简单逻辑)
  需要 Slot 数 = Q × C / 单Slot能力
  TM 数 = Slot数 / numberOfTaskSlots

State 存储:
  如果用 RocksDB, 每个 Key 约 100-500 bytes state
  总 State = 活跃 Key 数 × 单 Key State × 安全系数(2x)
  TM 磁盘 = 总State / TM数

Checkpoint 存储 (HDFS):
  保留 N 个 CP × 总State × 增量比例(0.1~0.3)
```

## 五、ZooKeeper 容量规划

```
ZK 节点数: 3(最小) 或 5(推荐)
ZK 堆内存: 数据量 × 3 + 1GB
  数据量 = znode数 × 平均数据大小

例: 10万 znode, 平均 1KB
  数据量 = 100MB
  堆内存 = 300MB + 1GB ≈ 2GB (-Xmx2g)

磁盘: 事务日志用 SSD (写入密集)
```

## 六、扩容信号

| 指标 | 阈值 | 动作 |
|------|------|------|
| HDFS 使用率 | > 75% | 扩 DN 或开 EC |
| NN 堆使用率 | > 80% | 扩 NN 内存或 Federation |
| YARN 队列使用率 | > 90% | 扩 NM 或调整队列 |
| NM 内存利用率 | > 85% | 扩 NM |
| ETL 耗时增长 | > 上月 20% | 查性能瓶颈或扩容 |
| Flink 消费延迟 | 持续增长 | 加并行度/TM |
