# 模块一：认证体系（Authentication）

## 1.1 白话版（费曼 L1）

> 认证 = **验明正身**。就像进入公司大楼：
>
> - **NONE**：大门敞开，谁都能进（开发环境）
> - **KERBEROS**：刷工牌（Kerberos ticket），最严格
> - **LDAP**：报工号+密码，去人事系统验证
> - **JDBC**：报工号+密码，去数据库查验
> - **CUSTOM**：公司自己的安检系统（自定义认证器）
>
> Kyuubi 支持**多种认证同时启用**（`kyuubi.authentication=KERBEROS,LDAP`），就像大楼同时接受刷工牌和报工号两种方式。

## 1.2 架构版（费曼 L2）

### 1.2.1 认证方式枚举

```scala
// 配置: kyuubi.authentication=KERBEROS,LDAP (逗号分隔，多选)
enum AuthTypes {
  NOSASL      // 不走 SASL 协议（纯 Thrift，不安全）
  NONE        // 走 SASL 但不校验密码（接受任何用户名）
  KERBEROS    // Kerberos 强认证 (SASL/GSSAPI)
  LDAP        // LDAP 目录服务认证 (SASL/PLAIN)
  JDBC        // SQL 查库认证 (SASL/PLAIN)
  CUSTOM      // 自定义认证提供者 (SASL/PLAIN)
}
```

### 1.2.2 认证链架构

```
客户端连接
  │
  ├── Thrift Binary → SASL 协商
  │   ├── GSSAPI (Kerberos) → KerberosAuthenticationProvider
  │   │     验证 Kerberos ticket → 获取 principal → 提取用户名
  │   │
  │   └── PLAIN (用户名+密码) → 按配置的认证链依次尝试
  │         ├── LdapAuthenticationProvider
  │         │     连接 LDAP Server → bind(user, password)
  │         ├── JdbcAuthenticationProvider
  │         │     执行 SQL 查询 → 比对密码 hash
  │         ├── CustomAuthenticationProvider
  │         │     加载自定义类 → authenticate(user, password)
  │         └── NoneAuthenticationProvider
  │               直接通过（不校验密码）
  │
  ├── REST API → HTTP 认证
  │   ├── 基于 Servlet Filter
  │   ├── Kerberos → SPNEGO 协商
  │   └── LDAP/JDBC → Basic Auth header
  │
  └── MySQL Protocol → 密码认证
      └── MySQL 握手 → password 校验
```

### 1.2.3 Kerberos 认证详解

```
前提：
  - Kyuubi Server 有 keytab: /etc/security/keytabs/kyuubi.keytab
  - Principal: kyuubi/_HOST@REALM

认证流程：
  1. 客户端 (beeline):
     kinit alice@REALM → 获取 TGT
     beeline -u "jdbc:hive2://host:10009/;principal=kyuubi/host@REALM"

  2. JDBC Driver:
     SASL 协商 → 选择 GSSAPI 机制
     发送 Kerberos Service Ticket (对 kyuubi/host@REALM 的票据)

  3. Kyuubi Server:
     用 kyuubi.keytab 验证 Service Ticket
     提取 client principal: alice@REALM
     用户名: alice (去掉 @REALM)

  4. 代理用户:
     Server 以 kyuubi 账号运行
     启动引擎时: spark-submit --proxy-user alice
     引擎以 alice 身份运行 → 访问 alice 有权限的数据
```

### 1.2.4 LDAP 认证详解

```scala
class LdapAuthenticationProvider(conf: KyuubiConf) {
  val ldapUrl = conf.get(AUTHENTICATION_LDAP_URL)       // ldap://ldap.company.com:389
  val baseDN  = conf.get(AUTHENTICATION_LDAP_BASE_DN)   // dc=company,dc=com
  val domain  = conf.get(AUTHENTICATION_LDAP_DOMAIN)    // company.com

  def authenticate(user: String, password: String): Unit = {
    val bindDN = s"$user@$domain"  // 或 "uid=$user,$baseDN"
    val env = new Hashtable[String, String]
    env.put(PROVIDER_URL, ldapUrl)
    env.put(SECURITY_PRINCIPAL, bindDN)
    env.put(SECURITY_CREDENTIALS, password)
    
    try {
      new InitialDirContext(env)  // LDAP bind 操作
      // 成功 → 认证通过
    } catch {
      case _: AuthenticationException =>
        throw new AuthenticationException(s"LDAP auth failed for $user")
    }
  }
}
```

### 1.2.5 JDBC 认证

```scala
class JdbcAuthenticationProvider(conf: KyuubiConf) {
  val jdbcUrl  = conf.get(AUTHENTICATION_JDBC_URL)
  val driver   = conf.get(AUTHENTICATION_JDBC_DRIVER)
  val query    = conf.get(AUTHENTICATION_JDBC_QUERY)
  // 例: "SELECT password_hash FROM users WHERE username = ?"

  def authenticate(user: String, password: String): Unit = {
    val conn = DriverManager.getConnection(jdbcUrl)
    val stmt = conn.prepareStatement(query)
    stmt.setString(1, user)
    val rs = stmt.executeQuery()
    if (rs.next()) {
      val storedHash = rs.getString(1)
      if (!BCrypt.checkpw(password, storedHash))
        throw new AuthenticationException("Password mismatch")
    } else {
      throw new AuthenticationException("User not found")
    }
  }
}
```

# 模块二：授权体系（Authorization）

## 2.1 白话版

> 认证 = "你是谁"，授权 = "你能做什么"。
>
> Kyuubi 集成 **Apache Ranger** 做细粒度授权：
> - 用户 alice 可以 SELECT 表 A，但不能 DROP 表 A
> - 用户 bob 只能看表 B 的 col1、col2，不能看 col3（列级授权）

## 2.2 架构版

### 2.2.1 Ranger 授权链路

```
用户提交 SQL: SELECT col1, col2 FROM db.table WHERE id > 100
  │
  ▼
Spark Catalyst 解析 → LogicalPlan
  │
  ▼
RangerSparkExtension 拦截 (Spark SQL Extension)
  │
  ├── PrivilegesBuilder.build(plan)
  │     提取: 访问了哪些表、哪些列、什么操作
  │     → PrivilegeObject(db.table, [col1,col2], SELECT)
  │
  ├── RangerSparkPlugin.isAccessAllowed(privilegeObject)
  │     查询 Ranger Admin (或本地缓存策略)
  │     → 检查 alice 是否有 SELECT db.table.col1,col2 的权限
  │
  ├── 授权通过 → 继续执行
  └── 授权拒绝 → 抛出 AccessControlException
```

### 2.2.2 配置方式

```properties
# 在 kyuubi-defaults.conf 或 spark-defaults.conf 中
spark.sql.extensions=org.apache.kyuubi.plugin.spark.authz.ranger.RangerSparkExtension

# ranger-spark-security.xml
ranger.plugin.spark.service.name=kyuubi_spark
ranger.plugin.spark.policy.rest.url=http://ranger-admin:6080
ranger.plugin.spark.policy.cache.dir=/tmp/ranger-cache
```

### 2.2.3 支持的权限粒度

| 操作类型 | 示例 | 控制粒度 |
|---------|------|---------|
| 数据读取 | SELECT | 表级 / 列级 |
| 数据写入 | INSERT / OVERWRITE | 表级 |
| DDL | CREATE / DROP / ALTER | 库级 / 表级 |
| 元数据 | SHOW / DESCRIBE | 库级 / 表级 |
| 函数 | UDF 调用 | 函数级 |
| URL | 外部路径访问 | 路径级 |

# 模块三：代理用户（Proxy User）

## 3.1 架构版

### 3.1.1 代理用户模型

```
Kyuubi Server (运行用户: kyuubi)
  │
  │ 认证后获知: 客户端是 alice
  │
  ├── spark-submit --proxy-user alice
  │     Spark Driver 以 alice 身份运行
  │     HDFS/HMS 请求带 alice 的身份
  │
  └── 代理授权检查 (Hadoop core-site.xml):
      hadoop.proxyuser.kyuubi.hosts=*
      hadoop.proxyuser.kyuubi.groups=*
      hadoop.proxyuser.kyuubi.users=*
```

### 3.1.2 凭证管理（CredentialManager）

```
问题：USER/GROUP/SERVER 级别下，Engine 长期运行，
      Kerberos TGT 和 Hadoop DelegationToken 会过期。

解决方案：CredentialManager 定期刷新

Kyuubi Server (有 keytab)
  → CredentialManager 定期获取新 DelegationToken
  → 写入 ZK 的 secret node
  → Engine 端 CredentialManager 监听 ZK
  → 获取新 Token → 更新 SparkContext.hadoopConfiguration
  → 后续 HDFS/HMS 请求使用新 Token

配置:
  kyuubi.credentials.hadoopfs.enabled=true
  kyuubi.credentials.hive.enabled=true
  kyuubi.credentials.renewal.interval=PT1H
```

# 对抗性分析

## 4.1 认证绕过风险

| 场景 | 风险 | 缓解 |
|------|------|------|
| `authentication=NONE` | 任何人可冒充任何用户 | 生产必须配置认证 |
| REST API 无独立认证 | curl 直接调用 | REST 走全局认证链 |
| MySQL 协议明文密码 | 密码被嗅探 | 启用 SSL/TLS |

## 4.2 代理用户滥用

**场景**：如果 `hadoop.proxyuser.kyuubi.users=*`，任何通过 Kyuubi 认证的用户都可以代理任何其他用户。

**缓解**：限制代理范围：
```xml
<property>
  <name>hadoop.proxyuser.kyuubi.groups</name>
  <value>data-team,analytics-team</value>
</property>
```

## 4.3 Kerberos Ticket Replay 攻击

**场景**：攻击者截获 Kerberos Service Ticket 后重放。

**防护**：Kerberos 协议本身有 replay 保护（timestamp + authenticator），在 ticket lifetime 内有效。ZK 传输的 secret 应启用 ACL 保护。
