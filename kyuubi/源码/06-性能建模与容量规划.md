# 一、引擎启动延迟模型

## 1.1 延迟分解

一个引擎从请求到可用的总延迟：

\[
T_{engine} = T_{lock} + T_{double\_check} + T_{semaphore} + T_{submit} + T_{yarn} + T_{spark} + T_{zk\_reg}
\]

| 阶段 | 符号 | 典型值 | 瓶颈因素 |
|------|------|--------|---------|
| 等待分布式锁 | \( T_{lock} \) | 0 ~ 180s | 前一个引擎是否在创建中 |
| 二次检查 ZK | \( T_{double\_check} \) | 1-10ms | ZK 响应延迟 |
| 等待信号量 | \( T_{semaphore} \) | 0 ~ 60s | 并发启动引擎数 |
| spark-submit 提交 | \( T_{submit} \) | 1-5s | 命令解析 + 进程创建 |
| YARN 资源调度 | \( T_{yarn} \) | 5-120s | **主要瓶颈** |
| SparkContext 初始化 | \( T_{spark} \) | 10-30s | JVM 启动 + 类加载 |
| ZK 注册 | \( T_{zk\_reg} \) | 10-500ms | ZK 写入延迟 |

**典型场景**：
- 引擎已存在（复用）：\( T \approx 10-100ms \)（仅 ZK 查询）
- 新建引擎（资源充足）：\( T \approx 30-60s \)
- 新建引擎（资源紧张）：\( T \approx 60-180s \)
- 新建引擎（锁等待）：\( T \approx 180-360s \)

## 1.2 热启动 vs 冷启动

| 类型 | 条件 | 延迟 | 占比 |
|------|------|------|------|
| 热启动 | engineSpace 在 ZK 中已有引擎 | < 100ms | ~80% |
| 温启动 | 引擎池中有空闲引擎 | < 100ms | ~10% |
| 冷启动 | 需要新建引擎 | 30-180s | ~10% |

**优化建议**：
1. 使用 USER 级别共享（热启动率最高）
2. 配置引擎池（`ENGINE_POOL_SIZE=3`）增加温启动机会
3. 预热引擎（定时脚本创建 Session 触发引擎创建）

# 二、Session 并发容量模型

## 2.1 容量公式

Kyuubi Server 的最大并发 Session 数受以下因素限制：

\[
C_{max} = \min(C_{thrift}, C_{exec}, C_{semaphore}, C_{limiter}, C_{memory})
\]

| 因素 | 配置 | 默认值 | 说明 |
|------|------|--------|------|
| \( C_{thrift} \) | `frontend.thrift.max.worker.threads` | 999 | Thrift 工作线程 |
| \( C_{exec} \) | `backend.server.exec.pool.size` | 100 | 后台操作线程池 |
| \( C_{semaphore} \) | `engine.startup.maxConcurrent` | ∞ | 引擎启动并发限制 |
| \( C_{limiter} \) | `limit.connections.per.user` × N_users | ∞ | 连接限流 |
| \( C_{memory} \) | JVM heap / per_session_overhead | ~5000 | 内存限制 |

**关键瓶颈分析**：

默认配置下，`exec.pool.size = 100` 是最小值。每个 Session 打开时至少占用一个线程（LaunchEngine 操作），因此：

\[
C_{practical} \approx 100 \text{（默认配置下）}
\]

**生产建议**：将 `exec.pool.size` 调到 500-1000。

## 2.2 Little's Law 应用

排队论中的 Little's Law：\( L = \lambda \cdot W \)

- \( L \) = 系统中的平均请求数（在途 Session 数）
- \( \lambda \) = 请求到达率（Session/秒）
- \( W \) = 平均服务时间（Session 生命周期）

**示例**：
- 到达率：100 个用户，每人每小时创建 1 个 Session → \( \lambda = 100/3600 \approx 0.028 \) Session/s
- 平均 Session 生命周期：1 小时 → \( W = 3600s \)
- 系统中平均 Session 数：\( L = 0.028 \times 3600 = 100 \)

**结论**：100 个用户，每人每小时 1 个 Session，需要同时容纳约 100 个活跃 Session。

## 2.3 引擎数量估算

对于 USER 级别共享：

\[
N_{engines} = N_{active\_users} \times (1 + \frac{pool\_size - 1}{pool\_size})
\]

简化为：
- 无引擎池：\( N_{engines} = N_{active\_users} \)（每个活跃用户一个引擎）
- 引擎池大小 K：\( N_{engines} = N_{active\_users} \times K \)

**YARN 资源需求**：

每个引擎需要：
\[
R_{engine} = (driver\_mem + driver\_overhead) + N_{exec} \times (exec\_mem + exec\_overhead)
\]

典型值：Driver 4G+1G=5G，Executor 8G+2G=10G，动态分配 1-20 个 Executor。

稳态资源：\( R_{total} = N_{engines} \times (5G + 5 \times 10G) = N_{engines} \times 55G \)

# 三、ZK 压力模型

## 3.1 ZK 节点数估算

```
总节点数 = Server 注册节点 + Engine 注册节点 + 锁节点 + Secret 节点

Server 节点: N_servers (通常 3-10)
Engine 节点: N_active_users × pool_size (可能 100-1000)
锁节点: 瞬时 (引擎创建期间存在)
Secret 节点: N_active_users

总计: ~N_servers + N_active_users × (pool_size + 1) + transient_locks
```

## 3.2 ZK 请求频率

| 操作 | 频率 | QPS 估算 |
|------|------|---------|
| 服务发现（getServerHost） | 每个新 Session | N_sessions / T_session |
| 引擎发现（getEngineByRefId） | 每次冷启动 | N_cold_starts × 10 (轮询) |
| 分布式锁 | 每次冷启动 | N_cold_starts |
| Session 心跳 | 无（ZK 自身心跳） | — |
| 引擎池计数器 | 每次有池的 Session | N_pool_sessions |

**示例**（1000 用户，20% 冷启动率）：
- 服务发现：1000 / 3600 ≈ 0.3 QPS
- 引擎发现（轮询）：200 × 10 / 3600 ≈ 0.6 QPS
- 总 ZK QPS：< 5 QPS

**结论**：Kyuubi 对 ZK 的压力很小。一个 3 节点 ZK 集群足以支撑数千用户。

# 四、Amdahl 定律在引擎池中的应用

## 4.1 引擎池的加速比

引擎池将一个用户的查询分散到 K 个引擎。如果查询间有依赖（如共享临时视图），并行度受限。

设 P = 可并行的查询比例（通常 80-95%），K = 引擎池大小：

\[
Speedup = \frac{1}{(1-P) + \frac{P}{K}}
\]

| K | P=80% 加速比 | P=95% 加速比 |
|---|-------------|-------------|
| 1 | 1.0x | 1.0x |
| 2 | 1.67x | 1.90x |
| 3 | 2.14x | 2.73x |
| 5 | 2.78x | 4.17x |
| 10 | 3.57x | 6.90x |

**结论**：对于 95% 可并行的工作负载，引擎池大小 5 能获得 4.17x 加速。超过 10 收益递减。

# 五、容量规划模板

## 5.1 输入参数

| 参数 | 值 | 说明 |
|------|---|------|
| 总用户数 | N_users | — |
| 日活用户占比 | R_active | 通常 30-60% |
| 每用户每小时查询数 | Q_per_hour | — |
| 平均查询时间 | T_query | 秒 |
| 引擎共享级别 | USER | — |
| 引擎池大小 | K | — |

## 5.2 计算公式

```
# 同时活跃用户数
N_active = N_users × R_active

# 同时活跃引擎数
N_engines = N_active × K

# 每引擎 YARN 内存（Driver + Executor 稳态）
R_engine = driver_mem + driver_overhead + avg_executors × (exec_mem + exec_overhead)

# 总 YARN 内存需求
R_total = N_engines × R_engine

# Server 线程池需求
exec_pool = N_active × avg_concurrent_queries_per_user

# ZK 节点数
zk_nodes = N_servers + N_engines + N_active (secrets)
```

## 5.3 示例计算

**场景**：500 用户，50% 日活，USER 级别，池大小 2

```
N_active = 500 × 0.5 = 250
N_engines = 250 × 2 = 500
R_engine = 4G + 1G + 5 × (8G + 2G) = 55G
R_total = 500 × 55G = 27.5 TB
exec_pool = 250 × 2 = 500
zk_nodes = 5 + 500 + 250 = 755
```

**结论**：
- YARN 集群需要 **27.5 TB 内存**（动态分配下可能只需 30-50%）
- Server `exec.pool.size` 设为 **500+**
- ZK 集群 3 节点足够
