## 一、概述

Kyuubi 的核心竞争力之一是**多引擎架构**——通过统一的 SQL 网关，支持 Spark、Flink、Trino、Hive、JDBC、Chat 六种引擎类型。本篇从源码层面深度剖析：

1. **EngineType 枚举体系** — 引擎类型的定义与扩展点
2. **EngineRef 引擎引用** — 引擎的发现、创建、锁定全流程
3. **ProcBuilder 进程构建器体系** — 各引擎的进程启动差异
4. **ShareLevel 共享级别** — CONNECTION/USER/GROUP/SERVER 四级复用策略
5. **引擎池（Engine Pool）** — 连接级负载均衡的 POLLING/RANDOM 策略
6. **引擎生命周期** — 从创建到销毁的完整生命周期管理

## 二、EngineType 枚举体系

### 2.1 源码定位

**文件**：`kyuubi-common/.../engine/EngineType.scala`

```scala
object EngineType extends Enumeration {
  type EngineType = Value
  val SPARK_SQL, FLINK_SQL, CHAT, TRINO, HIVE_SQL, JDBC = Value
}
```

**设计要点**：
- 使用 Scala `Enumeration` 实现，简洁但**不支持运行时动态扩展**
- 新增引擎类型需修改此枚举 + 添加对应 `ProcBuilder` + 修改 `EngineRef.create()` 的 match 分支
- 引擎类型通过配置 `kyuubi.engine.type` 指定，默认 `SPARK_SQL`

### 2.2 引擎类型与对应 ProcessBuilder

| EngineType | ProcessBuilder | 进程启动方式 |
|------------|---------------|-------------|
| `SPARK_SQL` | `SparkProcessBuilder` | `spark-submit` CLI |
| `FLINK_SQL` | `FlinkProcessBuilder` | Flink session/application |
| `TRINO` | `TrinoProcessBuilder` | Trino worker 进程 |
| `HIVE_SQL` | `HiveProcessBuilder` | HiveServer2 嵌入式 |
| `JDBC` | `JdbcProcessBuilder` | JDBC 连接引擎 |
| `CHAT` | `ChatProcessBuilder` | Chat 对话引擎 |

## 三、EngineRef — 引擎引用核心源码解析

### 3.1 类定义与构造参数

**文件**：`kyuubi-server/.../engine/EngineRef.scala`

```scala
private[kyuubi] class EngineRef(
    conf: KyuubiConf,
    sessionUser: String,         // 会话用户
    doAsEnabled: Boolean,        // 是否启用 doAs 代理
    groupProvider: GroupProvider, // 用户组提供者
    engineRefId: String,         // 对应 session 的唯一 ID
    engineManager: KyuubiApplicationManager,
    startupProcessSemaphore: Option[Semaphore] = None)  // 启动并发控制
```

### 3.2 核心属性计算

#### ShareLevel 路由用户

```scala
private[kyuubi] val routingUser: String = shareLevel match {
  case SERVER => Utils.currentUser           // SERVER 级：Kyuubi 服务用户
  case GROUP  => groupProvider.primaryGroup(sessionUser, conf)  // GROUP 级：主用户组
  case _      => sessionUser                 // USER/CONNECTION：会话用户
}
```

**关键认知**：`routingUser` 决定了引擎在 ZooKeeper 中的命名空间路径，直接影响引擎复用策略。

#### 引擎池选择策略

```scala
val seqNum = enginePoolSelectPolicy match {
  case "POLLING" =>
    // 通过 ZK 原子递增计数器实现轮询
    DiscoveryClientProvider.withDiscoveryClient(conf) { client =>
      client.getAndIncrement(snPath)
    }
  case "RANDOM" =>
    Random.nextInt(poolSize)
}
subdomain = s"$clientPoolName-${seqNum % poolSize}"
```

**POLLING 策略的原子性保证**：依赖 ZooKeeper 的 `getAndIncrement` 操作（CAS 语义），确保多 Kyuubi Server 实例间的全局轮询不重复。

### 3.3 EngineSpace — ZK 命名空间

```
对于 CONNECTION 共享级别:
  /{serverSpace}_{version}_CONNECTION_{engineType}/{routingUser}/{engineRefId}

对于 USER/GROUP/SERVER 共享级别:
  /{serverSpace}_{version}_{shareLevel}_{engineType}/{routingUser}/{subdomain}
```

**设计意图**：通过 ZK 路径的层级结构，天然实现了引擎的**命名空间隔离**和**服务发现**。

### 3.4 getOrCreate — 引擎获取或创建

```scala
def getOrCreate(discoveryClient, extraEngineLog): (String, Int) = {
  discoveryClient.getServerHost(engineSpace)   // 1. 先尝试从 ZK 发现已有引擎
    .getOrElse {
      create(discoveryClient, extraEngineLog)  // 2. 不存在则创建新引擎
    }
}
```

### 3.5 create — 引擎创建全流程（核心方法）

```
create() 流程:
┌────────────────────────────────┐
│ 1. tryWithLock(ZK 分布式锁)     │  非 CONNECTION 级需要加锁
│    ↓                           │
│ 2. 再次检查 ZK 是否已有引擎      │  double-check 避免重复创建
│    ↓                           │
│ 3. 根据 engineType 创建 Builder │  SparkProcessBuilder / FlinkProcessBuilder / ...
│    ↓                           │
│ 4. 获取 startup semaphore      │  控制并发启动数量
│    ↓                           │
│ 5. builder.start 启动进程       │  fork 子进程
│    ↓                           │
│ 6. 轮询等待引擎注册到 ZK        │  每秒检查一次
│    ├─ 进程退出 exitCode=0       │  释放信号量，继续等待 ZK 注册
│    ├─ 进程退出 exitCode≠0       │  抛出异常（引擎启动失败）
│    ├─ 超时                      │  kill 应用 + 抛出超时异常
│    └─ 应用状态已终止             │  快速失败
│    ↓                           │
│ 7. 返回 (host, port)           │  引擎的 Thrift 服务地址
└────────────────────────────────┘
```

#### 分布式锁机制

```scala
private def tryWithLock[T](discoveryClient: DiscoveryClient)(f: => T): T =
  shareLevel match {
    case CONNECTION => f  // CONNECTION 级不需要锁（每个连接独享引擎）
    case _ =>
      val lockPath = s"${serverSpace}_${version}_${shareLevel}_${engineType}_lock/${routingUser}/${subdomain}"
      discoveryClient.tryWithLock(lockPath, timeout + lockSpan)(f)
  }
```

**关键设计**：
- `CONNECTION` 级别无需锁，因为每个连接创建独立引擎
- `USER`/`GROUP`/`SERVER` 级别需要 ZK 分布式锁，防止多个 Session 并发创建同一引擎
- 锁超时 = 引擎初始化超时 + 10% 缓冲（`LOCK_TIMEOUT_SPAN_FACTOR = 0.1`）

#### 引擎启动并发控制

```scala
if (!startupProcessSemaphore.forall(_.tryAcquire(timeout, TimeUnit.MILLISECONDS))) {
  throw KyuubiSQLException("Timeout to acquires a permit from engine builder semaphore.")
}
```

**设计意图**：通过 `Semaphore` 限制同时启动的引擎进程数量，防止 Kyuubi Server 在高并发场景下 fork 过多进程导致资源耗尽。

### 3.6 引擎类型匹配与 ProcessBuilder 创建

```scala
builder = engineType match {
  case SPARK_SQL =>
    conf.setIfMissing(SparkProcessBuilder.APP_KEY, defaultEngineName)
    new SparkProcessBuilder(appUser, doAsEnabled, conf, engineRefId, extraEngineLog)
  case FLINK_SQL =>
    conf.setIfMissing(FlinkProcessBuilder.APP_KEY, defaultEngineName)
    new FlinkProcessBuilder(appUser, doAsEnabled, conf, engineRefId, extraEngineLog)
  case TRINO =>
    new TrinoProcessBuilder(appUser, doAsEnabled, conf, engineRefId, extraEngineLog)
  case HIVE_SQL =>
    HiveProcessBuilder(appUser, doAsEnabled, conf, engineRefId, extraEngineLog, defaultEngineName)
  case JDBC =>
    new JdbcProcessBuilder(appUser, doAsEnabled, conf, engineRefId, extraEngineLog)
  case CHAT =>
    new ChatProcessBuilder(appUser, doAsEnabled, conf, engineRefId, extraEngineLog)
}
```

**扩展点分析**：新增引擎类型需要：
1. 在 `EngineType` 中添加枚举值
2. 实现对应的 `ProcBuilder` 子类
3. 在此 `match` 中添加分支

## 四、ShareLevel — 四级引擎共享策略

### 4.1 策略对比

| ShareLevel | 引擎复用范围 | ZK 路径 | 适用场景 |
|-----------|-------------|---------|---------|
| `CONNECTION` | 每个连接独享 | `.../{user}/{sessionId}` | 强隔离需求、短查询 |
| `USER` | 同用户共享 | `.../{user}/{subdomain}` | 默认模式，资源效率最优 |
| `GROUP` | 同用户组共享 | `.../{primaryGroup}/{subdomain}` | 部门级隔离 |
| `SERVER` | 全局共享 | `.../{kyuubiUser}/{subdomain}` | 超大规模集群、查询路由 |

### 4.2 引擎池（Engine Pool）

当 `kyuubi.engine.pool.size > 0` 时，同一 ShareLevel 下的引擎会形成**引擎池**：

```
USER 级 + poolSize=3:
  ZK: /{ns}/{user}/pool-0    → Engine A
  ZK: /{ns}/{user}/pool-1    → Engine B
  ZK: /{ns}/{user}/pool-2    → Engine C
```

新连接通过 POLLING（轮询）或 RANDOM（随机）策略选择引擎。

**池大小限制**：`poolThreshold`（服务端全局上限）会 clamp 客户端请求的 poolSize：

```scala
val poolSize = math.min(clientPoolSize, poolThreshold)
```

## 五、引擎生命周期

### 5.1 完整生命周期状态图

```
                    ┌──────────────┐
                    │   PENDING    │  Session 创建，准备启动引擎
                    └──────┬───────┘
                           │ getOrCreate()
                    ┌──────▼───────┐
                    │  STARTING    │  ProcessBuilder.start()，等待注册到 ZK
                    └──────┬───────┘
                           │ ZK 注册完成
                    ┌──────▼───────┐
                    │   RUNNING    │  引擎正常服务中
                    └──────┬───────┘
                           │ 空闲超时 / 主动关闭 / 异常退出
                    ┌──────▼───────┐
                    │  TERMINATED  │  引擎进程退出，ZK 临时节点自动删除
                    └──────────────┘
```

### 5.2 引擎空闲超时机制

引擎端（如 SparkSQLEngine）自身维护空闲检测：
- 配置：`kyuubi.session.engine.idle.timeout`（默认 30 分钟）
- 当引擎上没有活跃 Session 且超过空闲阈值时，引擎自行退出
- ZK 上的临时节点（Ephemeral Node）在引擎进程退出后自动删除
- 下次新连接进来时，`EngineRef.getOrCreate()` 发现 ZK 无注册信息，触发新引擎创建

### 5.3 KyuubiApplicationManager — 应用状态追踪

```scala
class KyuubiApplicationManager {
  def getApplicationInfo(appMgrInfo, engineRefId, user, startTime): Option[ApplicationInfo]
  def killApplication(appMgrInfo, engineRefId, user): String
}
```

在引擎创建过程中，如果 submit 进程已退出（exitCode=0，集群模式），但 ZK 尚未注册，`KyuubiApplicationManager` 会通过 YARN/K8s API 查询应用状态，实现**快速失败**：

```scala
if (ApplicationState.isTerminated(appInfo.state)) {
  throw new KyuubiSQLException("The engine application has been terminated.")
}
```

## 六、Batch 模式 — KyuubiBatchSession

### 6.1 与 Interactive 模式的区别

| 维度 | Interactive (KyuubiSessionImpl) | Batch (KyuubiBatchSession) |
|------|-------------------------------|---------------------------|
| 会话类型 | `SessionType.INTERACTIVE` | `SessionType.BATCH` |
| 引擎创建 | 通过 `EngineRef` 创建共享引擎 | 通过 `BatchJobSubmissionOperation` 提交独立作业 |
| 交互方式 | Thrift RPC 双向通信 | 异步提交 + 状态轮询 |
| 生命周期 | 连接断开时关闭 | 作业完成时关闭 |
| 元数据 | 不持久化 | 持久化到 MetadataStore（MySQL/内存） |

### 6.2 BatchSession 核心流程

```scala
class KyuubiBatchSession(
    user, password, ipAddress, conf,
    sessionManager, sessionConf,
    batchType: String,      // 引擎类型（通常为 SPARK）
    batchName: Option[String],
    resource: String,        // Jar/Py 资源路径
    className: String,       // 主类名
    batchArgs: Seq[String],  // 作业参数
    metadata: Option[Metadata],
    fromRecovery: Boolean)   // 是否从故障恢复
```

#### open() 流程

```
1. 创建/更新 Metadata（持久化到 MetadataStore）
   ├─ 新建（v2）：更新 requestConf + clusterManager
   ├─ 新建（v1）：插入完整 Metadata 记录
   └─ 恢复模式：跳过
2. checkSessionAccessPathURIs() — 校验资源路径访问权限
3. super.open() — 创建 OperationLog 目录
4. runOperation(batchJobSubmissionOp) — 异步提交批处理作业
```

### 6.3 故障恢复

`fromRecovery = true` 时，Kyuubi Server 重启后从 MetadataStore 恢复 Batch Session 状态，继续追踪已提交作业的执行进度。这是 Batch 模式的核心高可用能力。

## 八、关键设计模式总结

### 8.1 发现优先于创建（Discovery-First）

```
getOrCreate() → 先查 ZK → 没有再创建
```

避免重复创建引擎，节省集群资源。

### 8.2 分布式锁保护（Distributed Lock）

非 CONNECTION 级别通过 ZK 分布式锁保证同一命名空间下只创建一个引擎。

### 8.3 信号量限流（Semaphore Throttling）

`startupProcessSemaphore` 限制并发启动进程数，防止 Kyuubi Server 在瞬间大量连接涌入时 fork bomb。

### 8.4 ZK 临时节点自动清理

引擎使用 ZK 临时节点注册服务信息，进程退出后节点自动删除，无需额外清理逻辑。

## 十、踩坑记录

1. **引擎池计数器溢出**（KYUUBI-5941）：POLLING 策略的 ZK 计数器在长期运行后可能溢出，需要 `% poolSize` 取模
2. **SESSION 级别引擎不释放**（KYUUBI-5232）：CONNECTION ShareLevel 下，客户端异常断连导致 ZK Session 超时延迟，引擎孤儿进程残留
3. **多 Kyuubi Server 竞争创建**：分布式锁的超时必须大于引擎初始化超时，否则会出现锁过期后重复创建

> **认知更新**：Kyuubi 的多引擎管理本质上是一个**分布式对象池**——以 ZooKeeper 为注册中心，以分布式锁为并发控制，以命名空间为隔离边界。理解这个模型后，所有 ShareLevel/Pool/Discovery 的行为都可以推导出来。
