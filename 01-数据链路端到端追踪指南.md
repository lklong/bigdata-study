# 数据链路端到端追踪指南

> 从 SQL 提交到结果返回，每一步经过了什么组件、什么代码路径、可能在哪里出问题

## 一、Kyuubi + Spark SQL 查询全链路

```
用户: SELECT count(*) FROM db.events WHERE dt = '2026-04-08'

┌──────────────────────────────────────────────────────────────────────┐
│ Step 1: 连接建立                                                      │
│   Client → ZK (查 /kyuubi/serviceUri) → 获取 Kyuubi Server 地址       │
│   Client → Kyuubi Server (JDBC/Thrift)                               │
│   Kyuubi 认证: Kerberos kinit / LDAP bind                           │
│   耗时: ~100ms                                                       │
│   可能故障: ZK 挂 / Kyuubi 挂 / 认证失败                              │
└──────────────────────────────────────────────────────────────────────┘
                                    ↓
┌──────────────────────────────────────────────────────────────────────┐
│ Step 2: Engine 获取/创建                                               │
│   Kyuubi → 检查是否有该用户的 Spark Engine                            │
│   ├── 有 → 复用 (< 10ms)                                             │
│   └── 没有 → spark-submit 启动新 Engine                               │
│       → YARN: AM 启动 → 申请 Container → Executor 启动 (~30-60s)     │
│       → K8s: Driver Pod 创建 → Executor Pod 创建 (~30-90s)           │
│   耗时: 0ms (复用) ~ 90s (冷启动)                                     │
│   可能故障: YARN 队列满 / K8s 资源不够 / Spark 配置错误                │
└──────────────────────────────────────────────────────────────────────┘
                                    ↓
┌──────────────────────────────────────────────────────────────────────┐
│ Step 3: SQL 编译                                                      │
│   Spark Engine 收到 SQL                                              │
│   → SparkSqlParser.parsePlan(): SQL 文本 → 未解析逻辑计划             │
│   → Analyzer.resolve(): 查询 HMS 获取表 schema + 分区信息             │
│     → HMS (Thrift RPC): getTable("db", "events")                     │
│     → HMS (Thrift RPC): getPartitions("db", "events", "dt=2026-04-08")│
│   → Optimizer.optimize(): 逻辑优化 (谓词下推/列裁剪/常量折叠)         │
│   → SparkPlanner.plan(): 逻辑计划 → 物理计划                         │
│   → Ranger Plugin: 检查用户是否有权访问 db.events 表                  │
│   耗时: 100ms ~ 5s (取决于分区数量和 HMS 性能)                        │
│   可能故障: 表不存在 / HMS 超时 / 权限被拒 / SQL 语法错               │
└──────────────────────────────────────────────────────────────────────┘
                                    ↓
┌──────────────────────────────────────────────────────────────────────┐
│ Step 4: 任务调度                                                      │
│   DAGScheduler: 物理计划 → Stage DAG                                  │
│   TaskScheduler: Stage → TaskSet → 分配到 Executor                   │
│   │                                                                   │
│   │ Stage 0 (FileScan): 分区裁剪后确定需要扫描的文件列表               │
│   │   → 查询 Iceberg Manifest (如果是 Iceberg 表)                     │
│   │   → 或读取 HMS 的 partition location (如果是 Hive 表)             │
│   │   → 每个文件 → 一个 Task (可能合并小文件)                         │
│   │                                                                   │
│   │ Stage 1 (Partial Aggregate): 本地预聚合 count                     │
│   │ Stage 2 (Final Aggregate): Shuffle + 最终聚合                     │
│   │                                                                   │
│   AQE: Stage 0 完成后，根据实际 Shuffle 数据量动态调整 Stage 2 分区数  │
│   耗时: 10ms ~ 100ms (调度本身)                                       │
│   可能故障: Executor 不够 → 排队等待                                   │
└──────────────────────────────────────────────────────────────────────┘
                                    ↓
┌──────────────────────────────────────────────────────────────────────┐
│ Step 5: 数据读取 (每个 Task 在 Executor 中执行)                        │
│                                                                       │
│   5a. 文件定位                                                        │
│       HDFS: NameNode RPC → getBlockLocations() → 获取 Block 位置      │
│       COS: HTTP GET 请求 COS Endpoint                                │
│                                                                       │
│   5b. 数据读取                                                        │
│       HDFS: DataNode 直接读取 (Short-Circuit 本地读 / 远程读)          │
│       COS: HTTP GET → 通过 COS SDK 读取数据块                        │
│       K8s 额外开销: veth → CNI → iptables → 物理网络                  │
│                                                                       │
│   5c. 数据解码                                                        │
│       Parquet/ORC Reader → 列裁剪 → 解压 (zstd/snappy/zlib)          │
│       → 谓词下推 (RowGroup/Stripe 级别 min/max 过滤)                  │
│       → 返回 ColumnarBatch (向量化读取)                               │
│                                                                       │
│   5d. 计算 count(*)                                                   │
│       HashAggregateExec → 本地 partial count                          │
│                                                                       │
│   耗时: 占总时间 60-90% (I/O bound)                                   │
│   可能故障: HDFS Block 损坏 / COS 超时 / 解码错误 / OOM               │
│   性能关键: COS 带宽 / HDFS 本地性 / 文件格式 / 压缩算法              │
└──────────────────────────────────────────────────────────────────────┘
                                    ↓
┌──────────────────────────────────────────────────────────────────────┐
│ Step 6: Shuffle + 最终聚合                                            │
│                                                                       │
│   6a. Shuffle Write                                                   │
│       YARN + ESS: 写本地磁盘 → ESS 管理                              │
│       K8s + Celeborn: 写 Celeborn Worker (远程)                       │
│                                                                       │
│   6b. Shuffle Read                                                    │
│       从各 Executor 拉取 partial count 结果                           │
│       数据量很小 (只有 count 值) → Shuffle 开销忽略不计                │
│                                                                       │
│   6c. Final Aggregate                                                 │
│       sum(partial_counts) → 最终 count 值                             │
│                                                                       │
│   耗时: < 1s (这个查询的 Shuffle 数据量极小)                          │
│   可能故障: Shuffle 文件丢失 / 网络超时                               │
└──────────────────────────────────────────────────────────────────────┘
                                    ↓
┌──────────────────────────────────────────────────────────────────────┐
│ Step 7: 结果返回                                                      │
│                                                                       │
│   Executor → Driver: 返回最终结果 (1 行 1 列)                        │
│   Driver → Kyuubi: Thrift RPC 返回 ResultSet                        │
│   Kyuubi → Client: JDBC ResultSet                                    │
│                                                                       │
│   耗时: < 100ms                                                       │
│   可能故障: 结果集太大 Driver OOM → 用 INSERT 代替 SELECT             │
└──────────────────────────────────────────────────────────────────────┘
```

## 二、每一步的监控指标

| Step | 关键指标 | 在哪里看 |
|------|---------|---------|
| 1.连接 | 连接延迟, 认证成功率 | Kyuubi 日志 / ZK 指标 |
| 2.Engine | Engine 启动时间, 复用率 | Kyuubi Metrics / YARN App 列表 |
| 3.编译 | HMS RPC 延迟, 分区查询耗时 | Spark Driver 日志 / HMS JMX |
| 4.调度 | Task 数, Locality 比例 | Spark UI → Stages |
| 5.读取 | **Scan Time**, Input Size, GC Time | Spark UI → Task Metrics / **Event Log** |
| 6.Shuffle | Shuffle Read/Write 大小, Fetch Wait Time | Spark UI → Stage Details |
| 7.返回 | Result Size, Getting Result Time | Spark UI → Task Summary |

## 三、Flink 流处理全链路

```
Kafka Topic: events (实时写入)

Step 1: Source (KafkaSource)
  → Consumer Group 消费 → 分配 partition → 读取消息
  → 关键: 并行度 = Kafka partition 数

Step 2: Deserialize
  → JSON/Avro/Protobuf 反序列化
  → 关键: 反序列化失败处理 (侧输出 or 跳过)

Step 3: Watermark 生成
  → 从事件中提取时间戳 → 生成 Watermark
  → 关键: 乱序延迟设置

Step 4: KeyBy + Window
  → 按 key 分区 → 窗口聚合
  → 关键: State 大小 / key 分布是否倾斜

Step 5: Process / Aggregate
  → 计算逻辑 → 输出结果
  → 关键: UDF 性能 / 是否有外部调用

Step 6: Sink
  → 写 Iceberg / HBase / Kafka / ES
  → 关键: 批量写入 / 事务提交 / 背压

Checkpoint (贯穿全程):
  → Barrier 从 Source 传播到 Sink
  → 每个算子快照状态到 HDFS
  → 关键: Barrier 传播是否被背压阻塞
```

## 四、故障定位 — 到底慢在哪一步

### 4.1 快速定位方法

```bash
# 1. Spark — 看 Event Log 中每个 Stage 的耗时
grep "SparkListenerStageCompleted" eventlog | python3 -c "
import sys, json
for line in sys.stdin:
    d = json.loads(line)
    info = d['Stage Info']
    dur = (info['Completion Time'] - info['Submission Time']) / 1000
    print(f'Stage {info[\"Stage ID\"]}: {dur:.1f}s  ({info[\"Number of Tasks\"]} tasks)')
"

# 2. Spark — 看 Task 级时间分解
# Executor Deserialize Time → 反序列化慢 → 换 Kryo
# Executor Run Time → 计算/IO 慢 → 看 scan time
# Shuffle Read Blocked → Shuffle 慢 → 看网络/ESS
# JVM GC Time → GC 频繁 → 调内存/GC 参数
# Scheduler Delay → 调度慢 → Executor 不够

# 3. Flink — 看 Checkpoint 对齐时间
# Web UI → Checkpoints → Alignment Duration
# 如果对齐时间很长 → 背压严重

# 4. Hive — 看 Tez DAG
# Tez UI → DAG → Vertex → Task Attempts → Counters
```

### 4.2 端到端延迟分解公式

```
Spark 任务总耗时 = 
    Engine 启动时间 (0 ~ 90s)
  + SQL 编译时间 (0.1 ~ 5s)
  + Σ(Stage 耗时)
    = Σ(Task Duration × Task 数 / 并行度)
    = Σ((Scan Time + Compute Time + GC Time + Shuffle Time) × Task 数 / 并行度)
  + 结果返回时间 (< 1s)

Flink 端到端延迟 =
    Kafka 消费延迟
  + 反序列化
  + 处理逻辑
  + 窗口等待
  + Sink 写入
```
