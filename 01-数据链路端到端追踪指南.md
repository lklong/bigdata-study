# 数据链路端到端追踪指南

> 从 SQL 提交到结果返回，每一步经过什么组件、可能在哪出问题

---

## 一、Kyuubi + Spark SQL 查询全链路

```
Step 1: 连接 (~100ms)
  Client → ZK 查 Kyuubi 地址 → Kyuubi Server (认证: Kerberos/LDAP)
  故障点: ZK挂 / Kyuubi挂 / 认证失败

Step 2: Engine 获取 (0ms复用 ~ 90s冷启动)
  Kyuubi → 查该用户的 Spark Engine → 有则复用 / 无则 spark-submit 新建
  故障点: YARN队列满 / K8s资源不够 / Spark配置错

Step 3: SQL 编译 (100ms ~ 5s)
  Parser → Analyzer (查 HMS 获取 schema/分区) → Optimizer → Planner
  Ranger Plugin 检查权限
  故障点: 表不存在 / HMS超时 / 权限被拒

Step 4: 任务调度 (10ms ~ 100ms)
  DAGScheduler → Stage DAG → TaskScheduler → 分配到 Executor
  AQE: Stage 完成后动态调整后续 Stage
  故障点: Executor不够排队

Step 5: 数据读取 (占总时间 60-90%)
  HDFS: NN RPC → DN 读 Block → 解压(zstd) → 列裁剪 → 谓词过滤
  COS: HTTP GET → COS SDK → 解压 → 列裁剪
  K8s额外: veth → CNI → iptables → 物理网络
  故障点: Block损坏 / COS超时 / OOM / GC暂停

Step 6: Shuffle + 聚合 (视数据量)
  Shuffle Write → ESS(YARN) / Celeborn(K8s) → Shuffle Read → Final Aggregate
  故障点: Shuffle文件丢失 / 网络超时

Step 7: 结果返回 (<100ms)
  Executor → Driver → Kyuubi → Client
  故障点: 结果集太大 Driver OOM
```

## 二、每步的监控指标

| Step | 指标 | 在哪看 |
|------|------|-------|
| 连接 | 连接延迟 | Kyuubi日志 |
| Engine | 启动时间/复用率 | Kyuubi Metrics |
| 编译 | HMS RPC延迟 | Driver日志/HMS JMX |
| 调度 | Task数/Locality | Spark UI → Stages |
| 读取 | **Scan Time/GC Time/Input Size** | **Event Log** |
| Shuffle | Read/Write大小 | Spark UI → Stage |
| 返回 | Result Size | Spark UI → Task |

## 三、Flink 流处理全链路

```
Kafka → KafkaSource (消费) → Deserialize → Watermark
→ KeyBy + Window → Process/Aggregate → Sink (Iceberg/HBase/Kafka)

Checkpoint 贯穿全程:
  Barrier 从 Source → 每个算子快照 → HDFS
  背压时 Barrier 传播被阻塞 → CP 超时
```

## 四、快速定位慢在哪

```bash
# Spark — Event Log 中每个 Stage 耗时
grep "SparkListenerStageCompleted" eventlog | python3 -c "
import sys,json
for l in sys.stdin:
    d=json.loads(l); i=d['Stage Info']
    dur=(i['Completion Time']-i['Submission Time'])/1000
    print(f'Stage {i[\"Stage ID\"]}: {dur:.1f}s ({i[\"Number of Tasks\"]} tasks)')
"

# Task 级时间分解 (从 Event Log):
# - Executor Deserialize Time → 反序列化慢
# - Executor Run Time → 计算/IO慢 (看 scan time)
# - JVM GC Time → GC频繁
# - Shuffle Read Blocked → Shuffle慢
# - Scheduler Delay → 调度慢

# Flink — Checkpoint 对齐时间
# Web UI → Checkpoints → Alignment Duration
# 对齐时间长 = 背压严重
```

## 五、端到端延迟公式

```
Spark总耗时 = Engine启动(0~90s) + 编译(0.1~5s)
            + Σ(Task耗时 × Task数 / 并行度)
            + 结果返回(<1s)

Flink端到端延迟 = Kafka消费延迟 + 反序列化 + 处理逻辑
                + 窗口等待 + Sink写入
```
