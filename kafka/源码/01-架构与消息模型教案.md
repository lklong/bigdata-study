# Kafka 第一课：架构与消息模型 — Producer/Broker/Consumer 三角关系

> **教学目标**：学完本课后，你能画出 Kafka 的完整架构图，解释消息从 Producer 到 Consumer 的完整旅程，并理解为什么 Kafka 能做到百万级 TPS。

---

## 一、先讲个故事（白话层）

想象一个**超级邮局系统**：

- **Producer（寄件人）**：把信（消息）投进邮箱
- **Broker（邮局）**：存信、分拣信，Kafka 集群由多个邮局组成
- **Topic（邮箱编号）**：每种信件有一个专属邮箱，比如"订单邮箱"、"日志邮箱"
- **Partition（邮箱里的格子）**：一个邮箱里有多个格子，信件按顺序塞进格子
- **Consumer（收件人）**：从格子里按顺序取信
- **Consumer Group（收件人团队）**：多个收件人组成团队，每人负责不同的格子，**一封信只被团队中的一人取走**

**Kafka 的核心设计哲学**：信件（消息）**写进去就不删**，任何人随时可以从任何位置重新读取——就像一个只追加的日志（Append-Only Log）。

## 三、核心概念详解

### 3.1 Topic 与 Partition

```
Topic "orders" (3个分区):

Partition-0: [msg0] [msg3] [msg6] [msg9]  ...  → offset 递增
Partition-1: [msg1] [msg4] [msg7] [msg10] ...  → offset 递增
Partition-2: [msg2] [msg5] [msg8] [msg11] ...  → offset 递增

每个 Partition 是一个有序、不可变的消息序列（Commit Log）
消息在 Partition 内通过 offset（偏移量）定位
```

**为什么要分区？**
| 设计目的 | 说明 |
|---------|------|
| **并行度** | 多个 Consumer 并行消费不同分区 |
| **吞吐量** | 分区分散在不同 Broker，I/O 不成瓶颈 |
| **有序性** | 同一分区内消息严格有序（跨分区不保证） |

### 3.2 Producer 发送策略

```
Producer 发送消息时决定去哪个 Partition：

① 指定了 partition → 直接发到指定分区
② 指定了 key     → hash(key) % numPartitions（相同 key 永远去同一分区）
③ 都没指定       → Round-Robin 轮询（或 Sticky Partition 批量发）

实际应用：
  - 订单按 order_id 做 key → 同一订单的所有事件在同一分区 → 有序
  - 日志不指定 key → 均匀分布到所有分区 → 最大吞吐
```

### 3.3 Consumer Group 与 offset

```
Consumer Group "analytics-group":
  Consumer-1 负责: Partition-0, Partition-1
  Consumer-2 负责: Partition-2

规则：
  ① 一个 Partition 在同一个 Group 内只能被一个 Consumer 消费
  ② 一个 Consumer 可以消费多个 Partition
  ③ Consumer 数 > Partition 数时，多余的 Consumer 空闲

Offset（偏移量）：
  每个 (Group, Topic, Partition) 维护一个 offset
  表示"我读到哪里了"
  offset 提交到 __consumer_offsets 内部 Topic
```

## 四、Kafka 为什么这么快？（4 个核心技术）

| 技术 | 类比 | 效果 |
|------|------|------|
| **顺序写磁盘** | 像写日记一样只追加，不随机改 | 磁盘顺序写 ~600MB/s，接近内存速度 |
| **Page Cache** | 操作系统自动帮你缓存热数据到内存 | 读写都走 OS 缓存，JVM 零 GC |
| **零拷贝（sendfile）** | 数据从磁盘直接到网卡，不过用户空间 | 减少 2 次内存拷贝 |
| **批量压缩** | 多条消息打包压缩一起发 | 减少网络 I/O 次数和带宽 |

**类比**：普通数据库像"查字典"（随机读写），Kafka 像"写日记+拍照发送"（顺序追加+批量传输）。

## 五、Kafka vs 传统消息队列

| 维度 | Kafka | RabbitMQ | RocketMQ |
|------|-------|----------|----------|
| 定位 | 分布式日志系统 | 消息中间件 | 消息中间件 |
| 消息模型 | Pull（消费者主动拉） | Push（推给消费者） | Pull + Push |
| 消息保留 | **持久保留**（按时间/大小） | 消费后删除 | 可配置保留 |
| 吞吐量 | **百万级 TPS** | 万级 TPS | 十万级 TPS |
| 有序性 | 分区内有序 | 队列内有序 | 队列内有序 |
| 重复消费 | ✅ 支持（重置 offset） | ❌ 消费即删 | 部分支持 |
| 生态 | 大数据生态核心 | 微服务场景 | 电商/金融 |

## 六、课堂练习

### 练习 1：设计分区策略

> 场景：电商系统的订单事件流，要求同一个用户的所有订单事件有序到达。Topic 有 16 个分区，日均 1 亿条消息。如何配置 Producer？

答：
- key = `user_id`
- 同一 user_id 的所有消息 hash 到同一个 Partition
- 分区内有序 → 同一用户的事件有序
- 16 个分区 → 16 个 Consumer 并行消费 → 吞吐量足够

### 练习 2：举一反三

> "Kafka 的 Partition + Offset 模型与 HDFS 的 Block + Offset 有什么同构关系？"

答：
- Kafka Partition ≈ HDFS Block — 都是数据的物理分片单位
- Kafka Offset ≈ HDFS Block Offset — 都是在分片内的位置标识
- 区别：Kafka 只追加不修改（append-only），HDFS Block 写完就不变（write-once）

### 练习 3：容量估算

> 场景：日均 10 亿条消息，平均每条 1KB，保留 7 天，3 副本。需要多少存储？

答：10 亿 × 1KB × 7 天 × 3 副本 = 10^9 × 10^3 × 7 × 3 = **21 TB**

## 七、知识晶体

```
问题/场景: 为什么选 Kafka 作为大数据生态的消息枢纽？
核心设计: Append-Only Commit Log + Partition + Consumer Group
设计意图: 用"日志"思维替代"队列"思维，实现高吞吐、持久化、可重放
关键参数:
  num.partitions            → Topic 默认分区数
  replication.factor        → 副本数（建议 3）
  log.retention.hours       → 消息保留时间（默认 168h = 7天）
  log.segment.bytes         → 日志分段大小（默认 1GB）
踩坑记录:
  - 分区数设多了不能减少（只能增加），需提前规划
  - Consumer 数 > Partition 数时多余的 Consumer 空闲浪费
  - key 分布不均会导致分区数据倾斜（热点分区）
认知更新: Kafka 不是"消息队列"，它是"分布式提交日志"——理解这一点就理解了它的一切设计决策
```
