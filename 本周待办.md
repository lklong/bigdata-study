#### 背景与动机

超长 SQL（动辄数 MB 的拼接 SQL、IN 列表上万元素、机器生成的递归 CTE 等）会引发：

1. **HS2 OOM**：解析阶段 AST 树爆炸，Driver 端 JVM 直接 OOM
2. **Metastore 写不下**：`QUERY_STRING` 字段超长（Hive 默认 4000 字符），审计日志 / 血缘记录截断或写入失败
3. **编译耗时雪崩**：单条 SQL 编译耗时几十秒，HS2 编译锁排队，其他用户全部卡住
4. **日志爆盘**：每次打印 SQL 都几 MB，hive.log / driver.log 几小时打满磁盘
5. **YARN AM 启动慢**：超长 SQL 走到 Tez/Spark 时序列化到 AM 启动参数里，启动超时

#### 功能需求

在 Hive 侧增加一个**前置拦截参数**，在 SQL 进入编译前就拒绝：

| 参数名（建议） | 默认值 | 作用层 | 说明 |
|---------------|--------|--------|------|
| `hive.query.max.length.bytes` | 1048576（1MB） | HS2 / CLI / Driver | 单条 SQL 字节数硬上限，超过直接拒绝 |
| `hive.query.max.length.chars` | 524288（512K） | HS2 / CLI / Driver | 单条 SQL 字符数硬上限（兼容多字节） |
| `hive.query.length.warn.threshold` | 262144（256K） | HS2 / CLI / Driver | 软告警阈值，超过打 WARN 日志但放行 |
| `hive.query.length.check.enabled` | true | 全局开关 | 总开关，灰度阶段可关 |

拒绝时返回明确错误，例如：

```
Error: Query length 2.3MB exceeds the limit 1MB (hive.query.max.length.bytes).
Please split the query or contact admin to raise the limit.
```

#### 实现要点

1. **拦截点选择**（关键）：
   - Hive：`Driver.compile()` 入口处，在 `ParseDriver.parse()` **之前**判断 `command.length()`
   - 源码位置：`D:\bigdata\txproject\hive\ql\src\java\org\apache\hadoop\hive\ql\Driver.java`
   - **不要**放到 Parser 之后，那时 OOM 已经发生了

2. **参数注册**：
   - 走 `HiveConf.ConfVars` 注册四个新参数
   - 源码位置：`D:\bigdata\txproject\hive\common\src\java\org\apache\hadoop\hive\conf\HiveConf.java`

3. **多字节兼容**：
   - 同时校验字节数（`getBytes(StandardCharsets.UTF_8).length`）和字符数（`length()`）
   - 取**任一超限**即拒绝

4. **白名单机制**（可选 P2）：
   - 支持 per-user / per-queue 覆盖，例如 ETL 账号给 10MB 上限

5. **审计**：
   - 拒绝时打 WARN 日志：用户、IP、SQL 前 200 字符 hash、长度
   - 接入现有审计链路（Ranger / 审计表）

#### 验收标准

1. 普通 SQL（< 256K）零影响，编译耗时不变
2. 超长 SQL（> 1MB）在 100ms 内被拒绝，不进入 Parser
3. HS2 进程内存曲线稳定，不再因超长 SQL 抖动
4. 错误信息能让业务方一眼看懂怎么改

#### 风险与回滚

- **风险**：误伤合法的大 SQL（如 BI 工具自动生成的 IN 列表）
- **回滚**：参数 `hive.query.length.check.enabled=false` 即时关闭，无需重启
- **观察指标**：`hive_query_rejected_by_length` 计数器，正常应为 0 或个位数/天
