> **目标**: 从"能用"到"精通"到"创造"，成为源码级科学家
> **适用**: 任何大型开源项目（Spark、Kyuubi、Hadoop、Flink、Kafka、Linux Kernel 等）
> **原则**: 读源码不是目的，建立心智模型才是

## 二、核心方法论：九大法则

### 法则 1：逆向工程思维 (Reverse Engineering Mindset)

> **核心**: 不要"读"源码，要"逆向"源码。像破解一个系统一样去理解它。

**操作**:
1. **黑盒观察** — 先不看代码，只观察系统的输入/输出/行为
   - 跑 benchmark，观察 metrics
   - 故意制造异常，观察错误信息和堆栈
   - 用 `strace`/`perf`/`async-profiler` 观察系统调用和热点
2. **形成假设** — 基于观察，猜测内部实现
3. **验证假设** — 带着假设去读源码，证实或推翻
4. **修正心智模型** — 假设错了？好，更新你的理解

```
黑盒观察 → 形成假设 → 读码验证 → 修正模型 → 循环
```

**为什么有效**: 认知科学 Testing Effect，主动预测比被动阅读的记忆留存率高 3-5 倍。

---

### 法则 2：分形剥离法 (Fractal Decomposition)

> **核心**: 任何复杂系统都是分形的——整体结构在每一层都自相似。

**三层剥离**:

| 层级 | 关注点 | 产出物 | 时间投入 |
|------|--------|--------|---------|
| **L1 接口层** | 公开 API、核心抽象类、trait/interface | 模块依赖图 + 接口契约文档 | 20% |
| **L2 主链路层** | 正常执行路径的完整时序 | 时序图 + 调用链文档 | 50% |
| **L3 异常/边界层** | 异常处理、资源回收、并发控制、边界条件 | 异常场景清单 + 防御代码分析 | 30% |

**关键洞察**: L3 才是区分"会用"和"精通"的分水岭。所有生产事故都发生在 L3。

---

### 法则 3：考古学方法 (Archaeological Method)

> **核心**: 代码是地层，Git 历史是考古记录。理解"为什么变成这样"比"现在是什么样"重要 10 倍。

```bash
git log --oneline --graph --all -- <file>        # 文件演化时间线
git log --all --grep="SPARK-12345"               # 按 JIRA 号追踪
git blame -L 100,200 <file>                      # 行级追溯
git diff v3.2.0..v3.5.0 -- <module>/             # 跨版本对比
```

---

### 法则 4：对抗性阅读 (Adversarial Reading)

> **核心**: 不要崇拜源码。用批判性思维去读，假设每一段代码都可能有 bug。

读到每个关键路径时，问自己:
- **"如果这里抛异常会怎样？"** — 资源有没有释放？
- **"如果并发调用会怎样？"** — 有没有 race condition？
- **"如果输入是 null/空/超大会怎样？"** — 边界处理完备吗？
- **"如果网络断了/磁盘满了会怎样？"** — 容错设计到位吗？

---

### 法则 5：类比迁移法 (Analogical Transfer)

> **核心**: 不同项目/领域之间存在深层同构。发现同构关系，一通百通。

| 概念 | Spark | Flink | Kafka | 操作系统 |
|------|-------|-------|-------|---------|
| 任务调度 | DAGScheduler | JobGraph | Controller | Process Scheduler |
| 内存管理 | Tungsten MemoryManager | MemorySegment | BufferPool | MMU/VMM |
| 容错机制 | RDD Lineage | Checkpoint/Savepoint | ISR Replication | Journaling FS |
| 序列化 | UnsafeRow | MemorySegment | Record Batch | System V IPC |
| 背压 | 无(批) | Credit-Based | Quota | TCP Flow Control |

---

### 法则 6：量化驱动理解 (Quantitative Understanding)

> **核心**: 不只理解"怎么做"，还要理解"做到什么程度"。科学家用数字说话。

- **性能建模**: `Shuffle Read 延迟 = 网络RTT × 请求次数 + 数据量/带宽 + 反序列化时间`
- **Amdahl 定律**: `加速比上限 = 1 / (S + P/N)`
- **容量估算**: `NameNode 内存 = 文件数 × 150B + 块数 × 150B → 1亿文件 ≈ 30GB heap`
- **Benchmark 验证**: 自己跑基准测试验证理论分析

---

### 法则 7：费曼输出法 (Feynman Output Method)

> **核心**: 如果你不能用大白话给别人讲清楚，你就没真正理解。

**三层输出**:

| 层级 | 目标受众 | 输出形式 | 检验标准 |
|------|---------|---------|---------|
| **白话层** | 非技术人员 | 类比故事 | 你妈能听懂 |
| **架构层** | 同事/社区 | 技术博客+架构图 | 同行认可 |
| **论文层** | 领域专家 | 设计文档/SPIP | 能经受质疑 |

---

### 法则 8：刻意练习 (Deliberate Practice)

> **核心**: 读 100 遍不如改 1 次。动手改代码是最快的学习路径。

| 级别 | 练习 | 频率 |
|------|------|------|
| ⭐ | 给模块补单元测试 | 每周 3-5 个 |
| ⭐⭐ | 复现已关闭的 JIRA Bug | 每周 2-3 个 |
| ⭐⭐⭐ | 读 open PR，写 review 意见 | 每周 3-5 个 |
| ⭐⭐⭐⭐ | 挑选 Good First Issue 提 PR | 每月 2-3 个 |
| ⭐⭐⭐⭐⭐ | 提出架构改进方案 (SPIP/KEP) | 每季度 1 个 |

---

### 法则 9：知识晶体化 (Knowledge Crystallization)

> **核心**: 零散的理解是沙子，系统化的知识才是晶体。

**晶体化模板**:

```markdown
## [知识晶体] <标题>

### 问题/场景
什么情况下会遇到这个问题？

### 核心源码
- 类: `org.apache.spark.xxx.YYY`
- 方法: `methodName()` (L120-L180)
- 关键行: L145 — 这里是魔法发生的地方

### 调用链
A.method() → B.method() → C.method() → [关键决策点] → D.method()

### 设计意图
为什么这样设计？有什么 trade-off？

### 关键参数
| 参数 | 默认值 | 作用 | 调优建议 |

### 关联知识
- 上游: [什么调用了它]
- 下游: [它调用了什么]
- 同构: [其他系统中的类似设计]

### 踩坑记录 & 认知更新记录
```
