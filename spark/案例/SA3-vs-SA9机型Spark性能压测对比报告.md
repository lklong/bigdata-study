# SA3 vs SA9 机型 Spark 性能压测对比报告

> 测试时间：2026-05-19 17:14 ~ 21:17
> 测试场景：同一 Spark 作业（参数一致、输入数据字节级一致）在 SA3 / SA9 两批机器（各 4 台）上配对压测，3 轮独立复现
> 输入规模：21.79 GiB（hdfsBytesRead = 23,400,146,949 B，6 次完全一致）

## 一、核心结论

| 指标 | SA9 vs SA3 | 说明 |
|---|---|---|
| **作业耗时** | 快 38.1%（19min13s → 11min54s） | 加速比 1.62× |
| **总资源账（vcoreSec / memSec）** | 省 25.5% | 跨 3 轮稳定不变 |
| **单核处理能力** | 强 33-35% | 单 core 读吞吐 258 → 342 KB/s |
| **结论可信度** | 高 | 6 次实验离散度 ±2%~±6% |

**迁移决策**：SA9 单价相对 SA3 不超过 **1.34×** 即划算（按单作业总资源消耗折算）。

## 五、按轮次趋势的关键现象

### 现象 1：加速比单调上升（1.49 → 1.57 → 1.79）

3 轮加速比逐轮放大，主要由 SA9 拿到的瞬时 vCore 越来越多驱动（88.4 → 97.0 → 102.5）。

### 现象 2：vcoreSec 节省比稳定 ±1pp（25.6% → 24.6% → 26.2%）

跨 3 轮稳态指标，均值 25.5%。**这是 SA9 vs SA3 算力差的核心数字**，不论集群忙闲都成立，是性价比测算的稳态依据。

### 现象 3：SA3 已撞 vCore 配额上限，SA9 仍有余量

- SA3 三轮瞬时 vCore：79.5 / 81.7 / 77.5，均值 ~80，**几乎是 4 台 SA3 的算力天花板**
- SA9 三轮瞬时 vCore：88.4 / 97.0 / 102.5，**单调向上爬，第 3 轮接近 4 台 SA9 的上限**

### 现象 4：稳态加速比 ~1.5×，第 3 轮 1.79× 是集群空闲叠加效应

第 3 轮（21:00 时段）集群相对空闲，SA9 拿到最多 vCore（102.5）+ SA3 拿到最少 vCore（77.5），两端拉大放大了加速比。**真实稳态加速比应取保守值 1.5× 附近**。

## 六、决策建议

### 1. 性价比测算口径

按**单作业总资源消耗（vcoreSec + memSec）**折算迁移成本：

```
SA9 总资源账 / SA3 总资源账 = 0.745
```

**SA9 单价 / SA3 单价 ≤ 1 / 0.745 = 1.342** 时，迁移划算。

### 2. 作业类型适配性

本次压测属于 **IO + 中量计算** 型作业（输入 21.79 GiB，输出 4.7 GiB，写/读比 21.6%）。

- 纯 CPU 密集（聚合 / 复杂 join）：加速比可能更高
- 纯 IO 密集（扫描 / 写出，CPU 利用率低）：加速比可能更低
- → 全面迁移决策前，建议补 1-2 种代表性作业类型的压测

### 3. 单核吞吐偏低，作业本身有优化空间

- SA9 单核读吞吐 342 KB/s，仍远低于 Spark 扫 HDFS 正常水位（5-20 MB/s/core）
- 瓶颈不在 IO，**大概率在 shuffle / GC / 小文件 / 倾斜**
- **不论是否迁机型，都建议进 Spark History Server 看 Stage 级数据进一步优化**

### 4. SA3 的 vCore 配额已撞顶，SA9 还能继续压

如果继续在 SA3 上跑这类作业，加资源也不会更快（已在天花板）。SA9 上还能再加一档 executor 数测一次稳态上限。
