## 一、问题切入：什么是"分布式协调"？

大数据三大引擎都要解决一个核心问题：**N 个节点上的进程如何协同完成一个全局任务，且任一节点失败时能恢复？**

- **Spark**：N 个 Mapper 写 shuffle 文件 → M 个 Reducer 拉数据，怎么协调读写？
- **Tez**：上下游 Vertex 之间，下游什么时候启动？启动多少？
- **Flink**：长时间运行的流任务，如何在不停止业务的前提下做全局一致性快照？

三种问题，三种范式。

## 二、范式对比一图流

```
┌────────────────────────────────────────────────────────────────────┐
│                    三大引擎的协调范式                                  │
├────────────────────┬───────────────────────────────────────────────┤
│ Spark Shuffle      │  推-拉两阶段 (Push-Pull Two Stage)              │
│                    │  Mapper 写本地 → Reducer 拉                    │
│                    │  通过 MapOutputTracker 传递元信息               │
│                    │  无中央协调器实时介入                           │
├────────────────────┼───────────────────────────────────────────────┤
│ Tez DAG 调度        │  事件驱动 (Event-Driven Plugin)                │
│                    │  VertexManager 接收上游完成事件                 │
│                    │  动态决定下游何时启动、启动多少                  │
│                    │  AM 是中央协调器                                │
├────────────────────┼───────────────────────────────────────────────┤
│ Flink Checkpoint   │  Chandy-Lamport (Marker-based Snapshot)        │
│                    │  Coordinator 注入 Barrier                       │
│                    │  Barrier 沿数据流传播                            │
│                    │  全局节点 ACK 后 Coordinator 持久化              │
└────────────────────┴───────────────────────────────────────────────┘
```

## 三、五维度对比

| 维度 | Spark Shuffle | Tez ShuffleVertexManager | Flink Checkpoint |
|------|---------------|--------------------------|-------------------|
| **协调对象** | 数据传输（Mapper→Reducer）| 任务调度（Vertex 启动时机/并行度）| 状态一致性（分布式快照）|
| **协调时机** | 静态（编译期决定）+ 运行时小调整（AQE）| 运行时动态（每个上游 task 完成都重算）| 周期触发 + 异步 |
| **中央组件** | MapOutputTracker（元信息）+ ESS（数据可选）| AM 内的 VertexManagerPlugin | CheckpointCoordinator |
| **数据流** | 推后拉（推到本地，reducer 拉）| 不传数据，仅传调度信号 | Marker 沿数据流传播 |
| **失败处理** | Stage 级重算（FetchFailed 触发）| Task 重试 + 整 vertex 重算 | 从最近 ckpt 恢复 |

## 四、三种"动态调整"机制对比

三个组件都在解决"事先不知道精确数据量"的问题，但用了不同的策略：

### Spark AQE - 运行时计划重写
- **触发点**：Stage 完成后、下游 Stage 提交前
- **决策依据**：MapOutputTracker 拿到的真实 stats（每分区字节数）
- **决策内容**：
  - `OptimizeSkewedJoin`：把超大分区切成多个子分区
  - `CoalesceShufflePartitions`：把多个小分区合成一个
  - `LocalShuffleReader`：bucketed join 不用真 shuffle
- **范围**：Stage 边界

### Tez Auto-Parallelism - 运行时 Vertex 重配置
- **触发点**：上游 task 完成事件累计到一定比例（`min-src-fraction=0.25`）
- **决策依据**：累计已完成上游 task 的输出字节估算总输出
- **决策内容**：
  - `pluginContext.reconfigureVertex(newParallelism)` 改变下游并行度
  - 同时改 EdgeManager 的 partition 映射
- **范围**：Vertex 边界（粒度比 Spark Stage 更细）

### Flink Unaligned Checkpoint - 反压自适应
- **触发点**：aligned 等待超时（`aligned-checkpoint-timeout=30s`）
- **决策依据**：barrier 对齐时长 vs 阈值
- **决策内容**：自动从 aligned 切到 unaligned，把 in-flight buffer 一并 snapshot
- **范围**：单次 checkpoint

**共同点**：三者都用"**先做出 best-effort 决定，运行时根据实际再调整**"的范式。
**差异**：Spark 在 Stage 边界、Tez 在 Vertex 边界、Flink 在 Checkpoint 内。

## 五、容错机制对比

| 维度 | Spark | Tez | Flink |
|------|-------|-----|-------|
| **失败粒度** | Task / Stage | Task / Vertex | Operator / Task |
| **容错单元** | RDD lineage | DAG recovery + 单 task 重试 | Checkpoint state |
| **状态存储** | 无（重算）| 无（重算）| StateBackend (HDFS/RocksDB) |
| **恢复方式** | 重新计算缺失 partition | TaskAttempt 重试 + 必要时重跑上游 vertex | 从 latest ckpt 恢复全 job |
| **典型故障日志** | `FetchFailedException` | `Vertex failed, vertexName=...` | `Checkpoint xxx expired` / `Restoring from checkpoint` |

**核心差异**：
- Spark/Tez 是"**重算容错**" —— 不存中间状态，靠 lineage/DAG 重新执行
- Flink 是"**状态容错**" —— 周期 snapshot，恢复时不重跑历史

**这种差异源于业务模型**：
- Spark/Tez 是批处理：数据有限，重算可行
- Flink 是流处理：数据无限，重算意味着无限重跑，必须 snapshot

## 七、三个引擎共享的设计模式

### 7.1 异步 + 状态机
- Spark：DAGScheduler 是事件驱动 + Stage 状态机
- Tez：DAGImpl/VertexImpl/TaskImpl 多层状态机 + Hadoop AsyncDispatcher
- Flink：CheckpointCoordinator timer + executor 双线程 + BarrierHandlerState 状态机

→ **大型分布式系统几乎都用"异步事件 + 状态机"驱动**，避免命令式控制流的状态错乱。

### 7.2 元信息与数据分离
- Spark：MapOutputTracker（元信息）+ BlockManager（数据）独立
- Tez：VertexManagerPluginContext（元信息）+ ShuffleHandler（数据）独立
- Flink：CheckpointMetadata（元信息）+ StateBackend（数据）独立

→ **元信息走快路径（控制平面），数据走慢路径（数据平面）**，避免大数据阻塞控制流。

### 7.3 可插拔扩展点
- Spark：ShuffleManager 接口（默认 SortShuffleManager，可替换为 PushBased）
- Tez：VertexManagerPlugin / EdgeManagerPlugin 都是 SPI
- Flink：StateBackend / CheckpointStorage 都是 SPI

→ **核心算法稳定，外围用插件机制让用户/厂商扩展**。这是开源框架的生命力来源。

### 7.4 配置驱动 + 运行时决策
- 三者都有大量默认配置（参数清单）
- 但运行时都会根据实际数据/状态再做决策（AQE / auto-parallel / unaligned 自动切换）

→ **静态配置定义边界，动态决策追求最优**。这是大数据系统从 MR 时代到现代的最大演进。

## 八、生产价值：故障定位的统一思路

不管是哪个引擎，故障排查都遵循"**协调链断点**"思路：

```
现象 → 协调链哪一步断了 → 上下游哪个角色出问题 → 看对应日志
```

**Spark 例**：
- 现象：Task 失败，FetchFailedException
- 断点：Reducer 拉 Mapper 数据失败
- 角色：Mapper 端 ESS / 网络 / 文件
- 日志：DN/Executor host 的 ESS log，看 BlockManager metrics

**Tez 例**：
- 现象：Vertex 卡 INITED 不前进
- 断点：VertexManagerPlugin 没收到上游完成事件
- 角色：上游 Vertex 没产出 / 事件丢失 / Plugin bug
- 日志：AM log 的 VertexImpl 状态转移、`onSourceTaskCompleted` debug

**Flink 例**：
- 现象：Checkpoint 持续超时
- 断点：barrier 对齐 / snapshot 写存储
- 角色：反压（barrier 走不动）/ state backend / 网络
- 日志：JM log 的 ckpt completion stat、TM log 的 BarrierHandler debug、StateBackend 上传速度

## 九、本次综述的认知突破

通过把三大引擎放在同一个分析框架里看，得到几个新认知：

1. **"动态决策"不是 Flink/AQE 独创** —— Tez Auto-Parallelism 早在 2014 年就实现了类似机制，只是名字不叫 "Adaptive" 而已。
2. **三引擎共享的"控制平面 vs 数据平面"思想**，源自更早的 Hadoop MR ApplicationMaster 架构。MR 没把这两者分清楚（JobTracker 干所有事），所以扩展性差，Tez/Spark/Flink 都吸取了教训。
3. **状态机 + 异步事件**是大型分布式系统的"通用语"，三者都重度使用。源码里大量的 `case SUCCESS / DUPLICATE / UNKNOWN` 就是状态机思维的体现。
4. **可插拔接口的设计深度反映了系统的成熟度**：MR 几乎不可扩展 → Tez/Spark/Flink 都有完善的 SPI → Flink 1.17 引入 Sink V2 进一步抽象。
5. **生产中遇到的 90% 故障都集中在"动态调整边界"** —— Spark AQE 误调、Tez auto-parallel 估错、Flink unaligned 切换异常。**生产稳定性 = 边界条件覆盖率**。
