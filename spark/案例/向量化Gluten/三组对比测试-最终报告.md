## 五、组1 失败诊断（H2 + 社区版 Spark 3.5.8 + Gluten 1.6.0）

### 5.1 现象
- Native round1 **完全成功**（7Q 总耗时 397s，比组2 Native R1 还快）
- Gluten round1 / **q1_agg 启动后第一个 stage 都没跑完，3 小时 42 分钟 progress 始终 10%**
- Driver 完全没产生 task 日志（14:47:02 之后 driver log 空白直到 18:29 我手动 kill）
- YARN application 有 `Aggregate Resource Allocation: 308M MB-sec, 109K vcore-sec`（executor 在跑但任务不返回）

### 5.2 根因（driver 日志中 14:46:49 的关键警告）
```
WARN VeloxListenerApi: Memory overhead is set to 1024MiB which is smaller than 
                      the recommended size 2457MiB. This may cause OOM.
W MemoryArbitrator.cpp:84] Query memory capacity[768.00MB] is set for NOOP 
                          arbitrator which has no capacity enforcement
WARN SparkShimProvider: Spark runtime version 3.5.8 is not matched with 
                       Gluten's fully tested version 3.5.5
```

**真凶猜测（按可能性排序）**：

1. **🥇 Spark 3.5.8 vs Gluten 1.6.0 兼容性 bug** —— Gluten 官方 only fully tested with **Spark 3.5.5**，3.5.8 是上游较新版本，shim 对接可能有问题导致 task 无法序列化或心跳异常
2. **🥈 Velox NOOP MemoryArbitrator** —— offHeap 8g 但实际容量只有 768MB，且没有 capacity enforcement，可能导致内存死循环或挂起
3. **🥉 Memory overhead 1g 不够** —— Velox 推荐 2457MiB，但 Native 在同集群同配置完全没问题，所以这不是主因

### 5.3 排除项
- ❌ 不是集群问题：Native 在同集群正常跑通
- ❌ 不是 JDK 问题：driver 用 Kona JDK17，executor archive 透传正常
- ❌ 不是数据问题：同样数据组2/组3 都跑通了
- ❌ 不是网络问题：YARN application 一直在 RUNNING 状态，资源分配正常

### 5.4 结论
**当前组合（社区版 Spark 3.5.8 + Gluten 1.6.0）在 EMR Hadoop 2.8.5 集群上不可用**。建议：
- 短期：用 EMR Spark 3.5.3 + Gluten 1.5.0-SNAPSHOT（验证可用）
- 长期：等 Gluten 官方对 Spark 3.5.8/3.5.10 的 shim 完善，或退回 Spark 3.5.5 + Gluten 1.6.0

## 六、关键洞察 / 落地建议

### 6.1 Gluten 加速场景分级

| 加速档位 | 加速比 | 典型场景 | 占比 |
|---|---|---|---|
| 🟢 **强加速（推荐用 Gluten）** | 1.5× ~ 1.85× | 3+ 表 join、子查询、窗口函数、复杂聚合 | 3/7 ≈ 43% |
| 🟡 **中加速（可用）** | 1.2× ~ 1.5× | 2 表 join、5 表全 join、轻度窗口 | 3/7 ≈ 43% |
| 🟠 **弱加速（不推荐）** | 1.0× ~ 1.15× | 单表轻聚合、self-join | 2/7 ≈ 28% |
| 🔴 **反向加速（必须避免）** | < 1.0× | self-join + LAG（H2 仅出现） | 1/7 ≈ 14% |

### 6.2 Hadoop 版本对 Gluten 影响

- **整体加速比 H2 ≈ H3（1.29× vs 1.31×）**，Hadoop 版本本身**不是 Gluten 加速的瓶颈**
- Native 性能 H2 略快于 H3 ~5-15%（EMR Hadoop 2.8.5 优化更成熟）
- **q6_self_join 在 H2 上反向（0.79×）但在 H3 上正常（1.12×）** —— 同样 Gluten 1.5.0，集群差异显著影响 self-join 类查询

### 6.3 生产部署建议

| 项 | 建议 | 备注 |
|---|---|---|
| Spark 版本 | **3.5.3 EMR 版** | 已验证生产可用 |
| Gluten 版本 | **1.5.0-SNAPSHOT** | 与 Spark 3.5.3 + EMR HashAggregateExec 兼容 |
| JDK | **Kona JDK 11**（driver / executor 一致） | 避免 ByteBuffer.flip 错误 |
| OffHeap | `spark.memory.offHeap.size=8g`（5g heap + 8g offHeap + 1g overhead = 14g/exec） | 与 Native 11g 总内存对齐 |
| Velox spill | **必须关闭**（spillEnabled / aggregationSpillEnabled / joinSpillEnabled = false） | 避开 Gluten 1.5 sumVector bug |
| 自动 fallback | `spark.gluten.fallbackEnabled=true`（默认） | self-join 等不支持算子自动回退 |

### 6.4 Query 级建议

- **优先 Gluten 化**：q3_join_3tables / q5_subquery / q4_window（加速比 > 1.45×）
- **Gluten 中性 OK**：q2 / q7（1.16-1.32×）
- **不建议 Gluten 化**：q1_agg / q6_self_join（< 1.15×，部分场景反向）
- **必须验证**：上线前用 EXPLAIN 看是否触发 Velox 算子（fallback 部分会失去加速）

1. ✅ **不修改集群任何全局 JDK 配置**（不动 `/etc/profile`、不软链 `/usr/local/jdk`）

4. ✅ **资源对齐**：Native 9G heap，Gluten 5G heap + 8G offHeap，总内存等量

## 九、TL;DR

> **结论**：在 EMR Spark 3.5.3 + Gluten 1.5.0-SNAPSHOT 组合下，5表 91.7GB 数据 / 7 个 TPC-H 风格 Query 测试，**Gluten 平均加速 1.29-1.31×**，Hadoop 2.8.5 与 3.3.4 集群表现持平。最佳场景 q3 三表 join 加速 1.82×，最差 q6 self-join 在 Hadoop 2.x 上反向 0.79×。社区版 Spark 3.5.8 + Gluten 1.6.0 在当前 EMR 集群上 **不可用**（task hang）。
>
> **生产建议**：用 EMR Spark 3.5.3 + Gluten 1.5.0-SNAPSHOT + Kona JDK 11，关闭 Velox spill，OffHeap=8g，仅对 join/window/subquery 类 query 启用 Gluten。
