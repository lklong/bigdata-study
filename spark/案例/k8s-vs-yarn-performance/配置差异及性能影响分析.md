## 二、逐项性能影响分析

### 1. fs.cosn.impl — 影响最大

| 项目 | 说明 |
|------|------|
| **差异** | YARN/5c99 用 `NativeCosFileSystem`（原生 COS SDK），K8s 默认用 `TemrfsHadoopFileSystemAdapter`（EMR 封装层） |
| **影响量** | IO_wait: NativeCosFS 组 avg 7.9s vs TemrfsAdapter 组 avg 9.0s，差 1.1s/task (13%) |
| **日志证据** | 全部 13 个应用 IO_wait 排名与 impl 类型高度对应，TemrfsAdapter 组吞吐 41~45MB/s，NativeCosFS 组 44~50MB/s |
| **原因** | TemrfsAdapter 是 EMR 封装层，包含额外的认证鉴权、监控计量、流量路由等中间逻辑，每次 I/O 操作叠加开销 |

### 2. dynamicAllocation.minExecutors — 影响 YARN ramp-up

| 项目 | 说明 |
|------|------|
| **差异** | YARN minExecutors=1，K8s NativeCosFS 组 minExecutors=100 |
| **影响** | YARN 从 1 个 Executor 慢爬升，Job 3 前期并行度不满 |
| **日志证据** | YARN 日志: `13:14:08 New executor 1 registered` → `13:25:28 New executor 2 registered`（第 2 个到达间隔 11 分钟）→ `13:26:34 executor 81 registered`（66s 内从 2 涨到 81）→ 最后 2 个在 `13:45` 和 `14:16` 才到。K8s 日志: 100 个 Executor 在 2 分钟内全部就绪 |
| **影响量** | YARN Job 3 前 ~70s 只有 1~3 个 Executor 在跑，估计浪费 ~60s wall-clock |

### 3. JDK 版本 — Tencent 1.8.0_282 vs TencentKona 1.8.0_472

| 项目 | 说明 |
|------|------|
| **差异** | YARN 用旧版 Tencent JDK 282，K8s 用新版 TencentKona 472 |
| **影响** | YARN GC avg=188ms (1.6%)，K8s GC avg=234~274ms (2.0~2.2%)，YARN GC 反而更优 |
| **日志证据** | YARN 日志可见 `ParallelGCThreads=8`（显式设置），K8s 未设置。YARN GC pause 目标 200ms (`target pause time: 200.00 ms`) |
| **原因分析** | YARN 的 GC 更优可能是 ParallelGCThreads=8 的效果而非 JDK 版本差异。但 YARN CPU avg=3.9s 高于 K8s 的 3.1~3.3s，说明旧 JDK 的 CPU 执行效率稍低 |

### 4. ParallelGCThreads — YARN 独有

| 项目 | 说明 |
|------|------|
| **差异** | YARN 显式设置 `-XX:ParallelGCThreads=8`，K8s 未设置（G1GC 默认按 CPU 核心数自动计算） |
| **影响** | K8s Executor 只有 1 core，默认 ParallelGCThreads 可能很低，导致 GC 回收效率较差 |
| **日志证据** | YARN GC% = 1.6%，K8s GC% = 2.0~2.2%，差 0.4~0.6 个百分点 |
| **影响量** | 每 task 多约 50ms GC → 34466 tasks / 100 exec × 0.05s ≈ 17s wall-clock，影响较小 |

### 5. fs.cos.userinfo.region — 可能影响路由

| 项目 | 说明 |
|------|------|
| **差异** | YARN/5c99: `zyb-ap-beijing`，K8s 默认: `bj` / `ap-beijing` |
| **影响** | 不同的 region 标识可能导致 COS SDK 路由到不同的接入节点 |
| **日志证据** | 无直接证据，但 IO_wait 差异与 region 配置差异方向一致 |

### 6. Shuffle Manager — 本场景无实际影响

| 项目 | 说明 |
|------|------|
| **差异** | YARN 用默认 SortShuffleManager + External Shuffle Service，K8s 用 Celeborn |
| **影响** | 本次测试 SQL 是纯扫描 (Shuffle=0)，Shuffle Manager 差异对 Job 3 无影响 |
| **日志证据** | 全部 13 个应用 Shuffle Read/Write = 0 |

### 7. JVM committed > max 内存异常 — YARN 独有

| 项目 | 说明 |
|------|------|
| **现象** | YARN 日志: `committed = 2685403136 should be < max = 2684354560`（多次出现） |
| **原因** | JDK bug（JDK-8267417），JVM committed 内存微超 max（差约 1~8MB），在 Executor memory=2560M 时触发 |
| **影响** | 不影响功能和性能，仅打印 ERROR 日志。K8s 使用更新的 Kona JDK 472 未出现此问题 |

## 四、审核意见

### 质疑 1: cosn.impl 差异是否真的是主因？

**挑战**: TemrfsAdapter 和 NativeCosFS 的差异是否仅仅是封装层开销，还是有其他因素（如认证方式、连接池配置）？

**回应**: 从数据看，TemrfsAdapter 组（7 个应用）IO_wait 平均 9.0s，NativeCosFS 组（6 个应用，含 YARN）IO_wait 平均 8.1s。但 **NativeCosFS 无 readahead 的 3 个应用（0408/744c/eb18）IO_wait = 8.2~8.5s**，与 TemrfsAdapter 组最快的 171a(8.4s)/494d(8.7s) 差距不大。说明 **impl 差异的实际影响可能被高估**，部分差异来自 readahead 和 region 配置。

**建议**: 需要一组控制变量实验——同一时间同一节点池，仅切换 cosn.impl，其他配置完全一致，才能精确隔离 impl 的影响。

### 质疑 2: YARN CPU avg=3.9s 显著高于 K8s 的 3.1~3.3s，为什么？

**挑战**: 同一条 SQL 同样的数据，CPU 计算量不应该因为平台不同而差 20%。

**回应**: YARN 使用 Tencent JDK 1.8.0_282，K8s 使用 TencentKona 1.8.0_472。两者是**不同的 JDK 发行版**（Tencent JDK vs TencentKona），不仅仅是版本号差异，可能有 JIT 编译器优化差异。此外 YARN 显式设置了 `ParallelGCThreads=8`，可能影响 JVM 内部的线程调度。

**证据不足**: 仅从 Event Log 无法区分这是 JDK 差异还是底层 CPU 指令集差异（YARN 和 K8s 跑在不同的物理机上）。

### 质疑 3: YARN GC% 更优是否值得关注？

**挑战**: YARN GC%=1.6%，K8s GC%=2.0~2.2%，差 0.4~0.6%，这算不算问题？

**回应**: 两者都在健康范围内（<5%），绝对差异仅 50~80ms/task，对 Job 3 整体影响 <20s。**不是性能瓶颈**。但如果 K8s 侧也加上 `-XX:ParallelGCThreads=8`，GC 可能进一步优化。

### 质疑 4: COS 请求失败次数与性能有关吗？

**挑战**: 744c 有 132 次 COS 请求失败但 Job 3 = 4036s（比 YARN 快），说明 COS 失败次数不直接影响性能？

**回应**: 正确。COS 客户端内置了重试机制（maxRetries=200/1000），请求失败后会自动重试。132 次失败分布在 100 个 Executor 上，平均每 Executor 仅 1.3 次，重试开销可忽略。**COS 请求失败次数不是性能的有效指标，IO_wait 吞吐量才是。**

### 质疑 5: 此次测试 SQL 无 Shuffle，结论能否推广到有 Shuffle 的场景？

**挑战**: Shuffle Manager 差异（SortShuffleManager vs Celeborn）在本次纯扫描测试中完全无影响，但实际生产中大部分 SQL 都有 Shuffle。

**回应**: **不能推广**。Celeborn 在有 Shuffle 的场景可能表现更好（远程 Shuffle 避免本地磁盘瓶颈），也可能更差（增加网络开销）。需要用有 Shuffle 的典型 SQL 再做对比。

---

## 五、结论

1. **cosn.impl 是 K8s 比 YARN 慢的主要原因**，但其单独影响可能被高估（需控制变量实验验证）
2. **YARN minExecutors=1 导致 Executor 慢爬升**，K8s NativeCosFS 组设了 minExecutors=100 规避了此问题
3. **JDK 版本差异导致 YARN CPU 高 20%**，但被 YARN 更优的 GC 部分抵消
4. **本次测试 Shuffle=0，Shuffle Manager 差异无法评估**，建议补充 Shuffle 场景测试
5. **COS 请求失败次数不是有效性能指标**，IO_wait 吞吐量才是
