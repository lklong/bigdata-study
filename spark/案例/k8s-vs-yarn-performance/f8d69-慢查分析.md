## 二、根因分析

### 排除项

| 排查方向 | 结论 |
|---------|------|
| SQL 差异 | 排除 — 4 个完全相同 |
| 配置差异 | 排除 — cosn.impl / JDK / JVM 参数完全一致 |
| 数据倾斜 | 排除 — skew 2.7x，且所有 Executor 均匀慢 |
| GC 问题 | 次要 — GC% 2.30% vs 2.12%，差 0.18%，贡献约 10% |
| Executor 丢失 | 次要 — Exec-37 心跳超时移除，但只少 1 个 |

### 根因：K8s 节点算力差异

f8d69 的 Executor **全部** CPU avg = 4.0~4.7s，744c 的全部 CPU avg = 3.0~3.2s。**不是个别 Executor 慢，是整体 CPU 性能差 35%**。

| 对比 | f8d69 节点池 | 744c 节点池 |
|------|-------------|-------------|
| 主要 IP 段 | **10.127.40.x** (集中) | 10.127.37.x / 42.x / 43.x / 45.x (分散) |
| CPU avg/task | **4.0~4.7s** | **3.0~3.2s** |
| 节点分布 | 101 个 unique 节点 | 100 个 unique 节点 |

**结论**: f8d69 被 K8s 调度到了 CPU 性能较弱的节点池（`10.127.40.x` 为主），导致每个 Task 的 CPU 计算时间增加 35%，Run Time 增加 15%，Job 3 整体慢 607s。

### 因果链

```
f8d69 调度到 10.127.40.x 节点池 (CPU 较弱)
  → 每个 Task CPU 时间 4.3s (vs 其他 3.1s, +35%)
  → GC 也受影响，avg 308ms (vs 247ms, +25%)
  → Task avg 13.4s (vs 11.7s, +15%)
  → 34466 tasks / 100 exec × 1.7s 差 = +586s wall-clock
  → 实际 Job 3 差 607s，吻合
```

## 三、建议

| # | 建议 | 说明 |
|---|------|------|
| 1 | **检查 `10.127.40.x` 节点池机型** | 可能是低配机型或 CPU 被其他 Pod 争抢 |
| 2 | **K8s nodeSelector/affinity 指定高性能节点池** | 避免被调度到慢节点 |
| 3 | **设置 resource request/limit 的 CPU** | `spark.kubernetes.executor.request.cores=1` 可能不够，考虑 limit=1 |
| 4 | **关注 Executor 心跳超时** | f8d69 Exec-37 和 eb18 Exec-78 均因心跳超时移除，可能与节点负载高有关 |
