# 同配置任务慢的原因逐个分析

> 相同配置的任务理论上耗时应一致，以下逐个分析组内偏慢任务的原因。

---

## 一、TemrfsAdapter 组（7 个 K8s 应用，配置完全一致）

组内基准: **494dd578 = 4273s** (最快)

| App ID | Job 3 | 组内差异 | 日期时段 |
|--------|-------|---------|---------|
| 494dd578 | 4273s | 基准 | 4/14 18:18 |
| d87771f2 | 4412s | 慢 139s | 4/14 20:00 |
| 171a31c6 | 4420s | 慢 147s | 4/7 13:29 |
| 264048 | 4435s | 慢 163s | 4/14 17:03 |
| 2f7380f0 | 4441s | 慢 169s | 4/14 12:25 |
| 769606ae | 4520s | 慢 247s | 4/14 14:13 |
| 834aa7f7 | 4521s | 慢 248s | 4/14 15:39 |

### 逐个分析

**494dd578 — 4273s（组内最快）**
- 运行时段 18:18，傍晚业务低谷期
- 无 ERROR、无 Executor 移除
- **结论**: COS 后端负载最低时段，I/O 延迟最小

**d87771f2 — 4412s（慢 139s, 3.3%）**
- 运行时段 20:00，晚间
- Executor-5 因心跳超时 166000ms 被移除，少 1 个 Executor
- **原因**: 1) Executor 丢失导致少量 Task 需要重新调度约 30~50s; 2) 晚间 COS 负载略高于傍晚

**171a31c6 — 4420s（慢 147s, 3.4%）**
- 运行日期 4/7（其他 6 个在 4/14），跨天对比存在网络/COS 服务端差异
- 节点段仅 10.127.37 和 10.127.40 两个网段，节点分布较集中
- 无 ERROR、无异常 Executor 移除
- **原因**: 1) 不同日期 COS 后端状态不同; 2) 节点分布集中可能导致局部网络拥塞

**264048 — 4435s（慢 163s, 3.8%）**
- 运行时段 17:03，下午偏晚
- 无 ERROR、无异常
- **原因**: 下午时段 COS 负载中等偏高

**2f7380f0 — 4441s（慢 169s, 3.9%）**
- 运行时段 12:25，中午
- 无 ERROR、无异常
- **原因**: 中午时段 COS 负载中等

**769606ae — 4520s（慢 247s, 5.8%）**
- 运行时段 14:13，下午高峰
- Driver 日志有 `ERROR DefaultCosHttpClient: failed to execute http request` — COS HTTP 请求超时
- **原因**: 1) 下午 14 点是 COS 日间负载高峰; 2) 出现过 COS HTTP 超时说明此时段 COS 后端响应慢

**834aa7f7 — 4521s（慢 248s, 5.8%）**
- 运行时段 15:39，下午高峰
- Executor-47 因心跳超时 172007ms 被移除
- **原因**: 1) 下午 15~16 点仍是 COS 负载高峰; 2) Executor-47 可能在高负载节点上运行，心跳被 GC 或 I/O 阻塞

### 本组结论

7 个应用配置完全一致，组内极差 248s (5.8%)，差异完全来自 **COS 后端日间负载波动**:

| 时段 | 应用 | Job 3 | 说明 |
|------|------|-------|------|
| 傍晚 18:18 | 494d | 4273s | COS 负载最低，最快 |
| 晚间 20:00 | d877 | 4412s | 负载中等 + 掉 1 Exec |
| 下午 17:03 | 2640 | 4435s | 负载中等偏高 |
| 中午 12:25 | 2f73 | 4441s | 负载中等 |
| **下午 14:13** | **7696** | **4520s** | **负载高峰 + COS HTTP 超时** |
| **下午 15:39** | **834a** | **4521s** | **负载高峰 + Exec 心跳超时** |

## 二、NativeCosFileSystem 无 readahead 组（4 个 K8s 应用，配置完全一致）

组内基准: **0408ee0e = 4030s** (最快)

| App ID | Job 3 | 组内差异 | 日期时段 | CPU avg | Task avg | Exec 移除 |
|--------|-------|---------|---------|---------|----------|----------|
| 0408ee0e | 4030s | 基准 | 4/15 21:40 | 3.2s | 11.6s | 0 |
| 744c908b | 4036s | 慢 6s | 4/15 17:33 | 3.1s | 11.7s | 0 |
| eb182010 | 4143s | 慢 113s | 4/15 20:22 | 3.2s | 12.0s | 1 |
| f8d69dea | 4643s | 慢 614s | 4/15 15:44 | **4.3s** | **13.4s** | 1 |

### 逐个分析

**0408ee0e — 4030s（组内最快）**
- 运行时段 21:40，夜间低负载
- 100 个 Executor 全部正常，无 ERROR
- CPU avg 3.2s，Task avg 11.6s，一切正常
- **结论**: 最佳运行状态基准

**744c908b — 4036s（慢 6s, 0.1%）**
- 运行时段 17:33，下午偏晚
- 100 个 Executor 全部正常
- CPU avg 3.1s（甚至略低于 0408），Task avg 11.7s
- 任务结束后清理阶段出现 `ERROR DefaultCosHttpClient: failed to execute http request`，但不影响 Job 3
- **原因**: 与 0408 基本一致，6s 差异属于正常随机波动，可忽略

**eb182010 — 4143s（慢 113s, 2.8%）**
- 运行时段 20:22，晚间
- **Executor-78 因心跳超时 151036ms 被移除**，实际只有 99 个 Executor 完成任务
- CPU avg 3.2s，与正常值一致，说明剩余 Executor 算力没问题
- 启动阶段出现 `WARN RetryingMetaStoreClient: MetaStoreClient lost connection`，HMS 连接断开后重连成功
- 任务结束后出现 COS HttpClient 超时
- **原因**:
  1. **Executor-78 丢失** — 少了 1% 算力，剩余 99 个 Executor 需要多跑约 345 个 Task (34466/100)，每 Task 12s → 约多 41s
  2. **Executor-78 正在执行的 Task 需要重新调度** — 额外 12~24s
  3. **HMS 连接断开/重连的短暂延迟** — 可能影响 Stage 启动几秒
  4. 合计约 60~70s 可解释，剩余 40s 可能是 COS 时段波动

**f8d69dea — 4643s（慢 614s, 15.2%）**
- 运行时段 15:44，下午
- **CPU avg = 4.3s**，比正常值 3.1~3.2s **高 35%**
- **CPU 总量 = 41.4h**，比正常值 30.0h **高 38%**
- **所有 101 个 Executor 均匀慢**，不是个别 Executor 问题
- Executor-37 因心跳超时 137939ms 被移除

**详细节点分析**:

| 指标 | f8d69 | 0408/744c（正常） |
|------|-------|-------------------|
| CPU avg | 4.3s | 3.1~3.2s |
| CPU 总量 | 41.4h | 30.0~30.2h |
| GC avg | 308ms | 247~249ms |
| Task avg | 13.4s | 11.6~11.7s |
| 节点段 | **10.127.40.x 为主** | 10.127.37~45 分散 |
| 最慢 Executor CPU | 4.7s | 3.2s |
| 最快 Executor CPU | 4.0s | 3.0s |

**根因**: f8d69 被 K8s 调度到 **10.127.40.x 网段为主的节点池**，这批节点的 CPU 性能显著弱于其他网段。证据:
1. 全部 Executor CPU 均匀偏高（最快的 4.0s 仍高于正常最慢的 3.2s）— 排除个别热点
2. 正常应用节点分散在 37/40/41/42/43/45 多个网段，f8d69 集中在 40 网段
3. CPU 多出的 35% 不是 GC 造成的（GC 占比只多 0.2%），是纯 CPU 计算时间增加
4. 可能原因: a) 10.127.40.x 是低配机型; b) 该网段有其他高负载 Pod 争抢 CPU; c) CPU 限流 (cgroup/CFS throttling)

**影响量化**:
- CPU 多: (4.3-3.2) × 34466 tasks = 37912s ≈ 10.5h 额外 CPU 时间
- 每 Task 多: 13.4-11.6 = 1.8s
- 100 Executor 并行: 34466/100 × 1.8s ≈ 620s wall-clock
- 实际差异 614s，完全吻合

---

## 三、汇总：同配置慢的三类原因

| 原因 | 影响量 | 涉及应用 | 可控性 |
|------|--------|---------|--------|
| **K8s 节点算力差异** | 慢 614s (15%) | f8d69 | 可控 — 设置 nodeSelector/affinity |
| **COS 日间负载波动** | 慢 100~248s (2~6%) | 7696, 834a, 2f73 等 | 不可控 — 避开高峰时段 |
| **Executor 心跳超时丢失** | 慢 50~113s (1~3%) | eb18, d877, 834a | 可控 — 调大心跳超时阈值 |
