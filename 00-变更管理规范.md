# 大数据集群变更管理规范

> 生产变更必须遵循的流程，防止人为事故

---

## 一、变更等级

| 等级 | 定义 | 审批 | 窗口 |
|------|------|------|------|
| **S 级** | 影响全集群（NN/RM/ZK/KDC 重启/升级） | 总监审批 | 业务低峰 + 值班 |
| **A 级** | 影响单组件（HS2/Kyuubi/Flink 重启/配置变更） | 负责人审批 | 业务低峰 |
| **B 级** | 影响单应用（Spark 参数调优/队列调整） | 自审 | 随时 |

## 二、变更 Checklist

### 变更前
- [ ] 明确变更内容、影响范围、回滚方案
- [ ] 备份当前配置: `cp conf/xxx.xml conf/xxx.xml.bak.$(date +%Y%m%d)`
- [ ] 通知相关方（用户/值班/上下游）
- [ ] S/A 级: 已获审批
- [ ] 有验证方法（怎么确认变更生效且无副作用）

### 变更中
- [ ] 按计划执行，不做计划外变更
- [ ] 灰度优先（先改 1 个节点验证，再全量）
- [ ] 实时监控关键指标（RPC 延迟/错误率/队列状态）
- [ ] 遇到异常立即回滚，不要"再试试"

### 变更后
- [ ] 验证变更生效: 检查配置/日志/指标
- [ ] 观察 30 分钟确认无异常
- [ ] 通知相关方变更完成
- [ ] 记录变更日志（who/when/what/why/result）

## 三、高危操作清单

| 操作 | 风险 | 安全做法 |
|------|------|---------|
| `hdfs dfs -rm -r /` | 删除全部数据 | 必须指定具体路径，开回收站 |
| `yarn rmadmin -refreshQueues` | 队列配置生效 | 先在测试环境验证配置 |
| NN/RM/ZK 重启 | 服务中断 | 滚动重启，先 Standby 后 Active |
| Kerberos keytab 轮换 | 认证中断 | 先部署新 keytab 再 invalidate 旧的 |
| Ranger 策略批量修改 | 权限错乱 | 先导出备份，小批量变更 |
| HDFS balancer | 占用带宽 | 限速 + 低峰执行 |
| Flink Savepoint + 升级 | 状态丢失 | 确认 Savepoint 成功再升级 |

## 四、回滚规范

```
回滚原则:
  1. 回滚方案在变更前就要准备好
  2. 发现异常 5 分钟内决定是否回滚
  3. 不要纠结"再等等看"——宁可回滚重来
  4. 回滚后确认服务恢复正常

回滚方法:
  配置变更: cp conf/xxx.xml.bak conf/xxx.xml → 重启服务
  软件升级: 切回旧版本目录 → 重启
  数据变更: 从 HDFS 快照恢复 / Iceberg time travel 回滚
  Flink 作业: 从 Savepoint 恢复
```

## 五、变更日志模板

```
日期: 2026-04-08
操作人: xxx
变更内容: 调整 YARN root.prod 队列 max-capacity 从 80% 到 90%
变更原因: prod 队列资源不足导致 ETL 延迟
影响范围: YARN prod 队列用户
变更等级: B
回滚方案: 改回 80% 并 refreshQueues
执行结果: 成功
验证方式: yarn queue -status root.prod 确认 max-capacity=90%
观察期: 30 分钟无异常
```
