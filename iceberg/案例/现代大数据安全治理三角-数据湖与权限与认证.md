## 二、三层之间的协作机制

### 2.1 Kerberos 与 Ranger 的协作 —— 身份传递

**关键问题**：Kerberos 认证的是"谁"，Ranger 鉴权针对"用户"。两者怎么传递？

**核心协议**：`auth_to_local` 规则
```
hadoop.security.auth_to_local =
  RULE:[1:$1@$0](.*@CORP.COM)s/@.*//      # alice@CORP.COM → alice
  RULE:[2:$1/$2@$0](.*@CORP.COM)s/@.*//   # hdfs/host@CORP.COM → hdfs/host
  DEFAULT
```

**传递路径**：
1. SaslRpcServer 验证 client AP_REQ → 提取 `alice@CORP.COM`
2. 通过 `KerberosName.getShortName()` + auth_to_local 规则 → `alice`
3. 把 `alice` 注入 `UserGroupInformation`
4. Hadoop 业务系统 (HDFS NameNode / Hive HS2) 调 Ranger Plugin
5. Ranger 通过 `UserGroupInformation.getCurrentUser().getShortUserName()` 拿到 `alice`
6. Ranger 用 `alice` + 用户所属 groups（来自 LDAP/SSSD）做策略匹配

**生产坑点**：
- auth_to_local 规则错 → Ranger 看到的是 `alice@CORP.COM` 全名而非 `alice` → 策略全部 miss → 隐式 deny
- LDAP 同步延迟 → 新用户加 group 但 Ranger Plugin 缓存还是旧 group list

### 2.2 Ranger 与 Iceberg 的协作 —— 表/列/行级鉴权

**关键问题**：Iceberg 表的列权限、行过滤、列脱敏怎么实现？

**Iceberg 不直接做鉴权**，依赖外部引擎（Spark/Trino/Hive）的 Ranger Plugin：

```
[查询: SELECT name, salary FROM hr.employees]
   │
   ▼
[Trino Coordinator]
TrinoRangerPlugin → RangerAccessRequest:
  resource: { catalog: "iceberg", database: "hr", table: "employees", column: "salary" }
  user: "alice"
  accessType: "select"
   │
   ▼
RangerPolicyEngineImpl.evaluatePolicies(...)
  - Resource policy: alice can SELECT hr.employees.{name, age, dept_id}
  - Column "salary" not in alice's allowed columns → DENY
   │
   ▼
返回 isAllowed=false
   │
   ▼
[Trino] 抛 AccessDeniedException 给用户
```

**进阶：Row Filter / Data Mask**
- Ranger 支持 `RowFilterPolicy` —— 给 alice 加 row filter `WHERE department='Engineering'`
- Trino/Spark 在 SQL plan 中**自动注入** filter，alice 只能看到工程部数据
- Ranger 支持 `DataMaskPolicy` —— `salary` 列对 alice 做 `[MASK_HASH]`

### 2.3 Kerberos 与 Iceberg 的协作 —— 数据湖底层访问

**关键问题**：Iceberg 数据存在 HDFS/S3，怎么保证只有授权 client 能写？

**协作链**：
```
Spark 写 Iceberg 表
   │
   │ ① Driver 用 Kerberos keytab 登录拿 TGT (alice@CORP.COM)
   ▼
Driver SQL: INSERT INTO iceberg_tbl ...
   │ ② Driver 通过 ranger 鉴权（select on source, write on target）
   ▼
Driver 把 task 分发给 Executor
   │ ③ Driver 通过 HDFS DT (Delegation Token) 把"代理身份"传给 Executor
   ▼
Executor 用 DT 写 HDFS 数据文件
   │ ④ Executor 不需要 keytab，DT 短期有效
   ▼
Driver 调 SnapshotProducer.commit()
   │ ⑤ Driver 用 alice 身份写新 metadata.json 到 HDFS
   ▼
Iceberg Catalog (HMS/JDBC) 原子切换 metadata 指针
   │ ⑥ HMS 调用 Ranger 鉴权 alice 是否能 ALTER TABLE
   ▼
完成
```

**关键点**：
- **Delegation Token 解决了 Executor 大规模分发问题** —— 不用每个 executor 都拿 keytab，driver 用 keytab 拿 DT 再分发，DT 有限期（默认 24h）+ 可 renew
- **OCC retry 与 Kerberos 续期独立** —— commit 重试 30min 内，Kerberos TGT 不能过期。所以**长 retry + 长任务必须开 reloginFromKeytab**

## 三、安全治理三大设计共性

### 3.1 不可变性（Immutability）
- **Kerberos**：TGT 一旦签发不可修改，过期才换新（票据本身不可变）
- **Ranger**：Policy 历史保留版本（policy_version 自增），变更生成新版本
- **Iceberg**：TableMetadata 不可变，每次更新产生新版本

→ **不可变是审计、追溯、回滚的物理基础**。如果对象会被 in-place 修改，就无法恢复历史。

### 3.2 拉模式 + 缓存（Pull + Cache）
- **Kerberos**：Client pull TGT，缓存到 Subject；Service 缓存 keytab key
- **Ranger**：Plugin 周期 pull policy（默认 30s），本地缓存 + 增量更新（PolicyDelta）
- **Iceberg**：Reader 拉 metadata.json + manifests，本地缓存 plan 阶段

→ **拉模式让 server 端无状态化**，缓存让性能可控。Push 模式（如 Zk Watcher）在大规模场景下扩展性差。

### 3.3 短路决策（Short-Circuit）
- **Kerberos**：`checkTGTAndReloginFromKeytab` 先看 TGT 是否快过期，没到 80% 直接 return
- **Ranger**：`!result.getIsAccessDetermined() || !result.getIsAuditedDetermined()` 已决策的策略不评估
- **Iceberg**：commit 时 `if (updated.changes().isEmpty()) return;` 无变更不写 metadata

→ **性能敏感系统普遍用短路优化**。每个 if 看似平淡，省下来的开销在百万 QPS 下指数级放大。

### 3.4 重试 + 退避（Retry + Backoff）
- **Kerberos**：`hadoop.kerberos.min.seconds.before.relogin=60` 防 relogin 风暴
- **Ranger**：Plugin 拉 admin 失败用本地 cache 兜底，下次再拉
- **Iceberg**：commit OCC 冲突时指数退避重试（默认 4 次 / 30min）

→ **分布式系统失败是常态**。退避重试是必备防御。

### 3.5 双轨决策（独立决策路径）
- **Kerberos**：Authentication（你是谁）与 Authorization（你能干什么）完全分离
- **Ranger**：Access 决策与 Audit 决策独立 —— Access 已决可短路，Audit 仍要继续找 audit policy
- **Iceberg**：Schema evolution / Partition evolution / Snapshot 演进各自独立 metadata 字段

→ **复杂系统用正交关注点拆分**，避免一处变更影响多处逻辑。

## 四、生产中的"安全治理三角"故障矩阵

| 现象 | 涉及层 | 根因 | 排查路径 |
|------|------|------|---------|
| `Cannot get tgt` | Kerberos | TGT 过期未续期 | UGI 日志看 relogin；加 checkTGT |
| `auth_to_local rule X not matched` | Kerberos→Ranger 桥 | 规则错 | hadoop org.apache.hadoop.security.HadoopKerberosName 测 |
| `Access denied: User alice does not have ...` | Ranger | 策略缺失或 deny | Ranger UI 查 audit；DEBUG plugin |
| `Permission denied (HDFS)` 但 Ranger 没记录 | Ranger 未生效 | Plugin 没启用或 cache 异常 | 看 NameNode log 是否有 Ranger 评估痕迹 |
| `CommitFailedException: Cannot commit` | Iceberg OCC | 多写并发 | 调大 commit.retry.num-retries |
| `Cannot ALTER TABLE: Permission denied` | Ranger × Iceberg | catalog 用 hive principal 但 alice 没 alter 权限 | 加 hive 库 alter 策略 |
| Iceberg 表元数据膨胀 | Iceberg | 无 expire_snapshots | 配 metadata 自动清理 |
| Audit 记录显示陌生 IP | Ranger | DT 被偷？或代理转发？| 查 Audit 详情 contextOrigins |
| `org.apache.hadoop.security.AccessControlException` | Kerberos→Ranger | Kerberos 通了但 Ranger 拒 | 同时看 NN audit 与 Ranger audit |
| Spark 长任务运行 24h 后 fail | Kerberos | TGT 过期 | --principal/--keytab + spark.kerberos.relogin.period |

## 五、深度认知：三层共同的"无状态服务"思想

三大组件的核心架构都遵循"**服务侧无状态 + 客户端持有凭证 + 服务侧验证凭证**"模式：

| 层 | 服务 | 凭证 | 验证 |
|----|------|------|------|
| Kerberos | KDC | TGT (用 KDC master key 加密) | 服务端 keytab 验证 ServiceTicket 签名 |
| Ranger | RangerAdmin | Policy version + Tag version | Plugin 本地评估，无须 admin 介入 |
| Iceberg | Catalog (HMS/JDBC) | metadata.json 路径 | Catalog 原子 CAS 切换路径 |

→ **服务端无状态** = Kerberos KDC 不记忆每个 client 的 session、Ranger Admin 不记忆 plugin 的评估、Iceberg Catalog 不记忆每个 reader 的版本
→ **凭证自包含**所有信息（principal / policy / metadata），客户端凭它独立完成业务
→ **服务端只在凭证创建/更新时介入**

这种设计**无限横向扩展**：KDC 集群化、Ranger Admin 多副本、Catalog 高可用，都不影响业务侧每秒百万次的认证/鉴权/读写决策。

## 六、本次综述的认知突破

1. **三大组件都用"不可变 + 多版本"做审计基础** —— 之前一直觉得"为什么 Iceberg 不能 in-place 修改？"，原来是为了和 Kerberos TGT、Ranger policy version 一样，**让任何时刻的状态都可追溯**。
2. **Audit 在三层都是独立路径** —— Kerberos KDC log、Ranger audit、Iceberg snapshot history 各自独立，需要 SIEM 系统聚合。**生产中"全链路可追溯"必须三层 audit 都开**。
3. **`auth_to_local` 是被低估的关键配置** —— 80% 的"权限不一致"问题都源于它。规则错就让所有 Ranger 策略失效，但报错信息很隐蔽（隐式 deny）。
4. **OCC + retry 不是 Iceberg 独有** —— Kerberos relogin 限速、Ranger plugin pull 重试、Iceberg commit 重试，都是同一个范式。**"失败重试 + 退避"是大数据安全栈的肌肉记忆**。
5. **服务端无状态 = 横向扩展的前提** —— 这条规律不止安全栈，整个大数据栈（HDFS NN、YARN RM、Hive HMS）都遵循。理解了这条，就懂为什么 Hadoop 能撑万节点。
