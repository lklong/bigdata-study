## 二 · 逐阶段耗时差异归因

### 阶段 1：`four` CTAS（16 顶点）

| 指标 | 异常 | 正常 | 关闭 skew |
|---|---|---|---|
| Map 1 task 数 | 5945 | 5945 | **5691** |
| 耗时 | **3725.68s** | 1650.31s | 1680.38s |

**观察**：
- Map 1 task 数异常/正常相同（5945），关闭 skew 略少（5691，-4.3%）—— 这 **4.3% 的差异来自业务日的数据增量**（maxDateTime 不同导致上游切片数差 254 个）
- 没有 FAILED 重试，纯系统耗时
- 异常/正常 = 2.26×，正常/关闭skew ≈ 1× —— 多出来的 ~2075s（35 min）**不是任务本身的问题，是集群系统级慢化**

**归因**：  
✅ **集群级背景慢化**（凌晨 04:59 的 YARN 资源竞争 / HDFS IO 压力 / 同时段调度并发）。  
❌ 与 SQL、参数、数据量无关。

---

### 阶段 2：`four0` CTAS（9 顶点）

| 指标 | 异常 | 正常 | 关闭 skew |
|---|---|---|---|
| Map 1 task 数 | 1518 | 1518 | 1518 |
| 耗时 | **446.57s** | 254.21s | 258.60s |
| **Map 1 FAILED** | **7** | 0 | 0 |
| **Map 1 KILLED** | **153** | 0 | 0 |
| Reducer 2 KILLED | 18 | 0 | 0 |
| Reducer 4 KILLED | 6 | 0 | 0 |
| Reducer 5 KILLED | 1 | 0 | 0 |

**观察**：
- Task 数完全一致（1518）
- **异常那次 Map 1 有 7 次 attempt 失败 + 153 个被 KILLED**（正常 / 关闭 skew 全零）
- 异常 / 正常 = 1.76×，比 four 的 2.26× 低
- 正常 vs 关闭 skew 几乎一致（254 vs 259，差 1.7%）→ 集群条件相近

**归因**：  
⚠️ **任务级失败 + 集群级背景**两者叠加。Map 1 的 7 FAILED 触发了 Tez 重试（AssertionError / HIVE-20990 已在其他分析中定位），带来部分额外耗时。  
但比值 1.76× 远小于 four 的 2.26×，说明 **这里失败重试的额外耗时没 four 的系统级慢化占比大**。

### 阶段 3：`four1` CTAS 🔥（问题焦点）

| 指标 | 异常 | 正常 | 关闭 skew |
|---|---|---|---|
| **DAG 顶点数** | **10** | **10** | **7** ← 拓扑不同 |
| Map 1 task 数 | 1153 | 1153 | 1153 |
| Reducer 2 task 数 | 1009 | 1009 | 1009 |
| Reducer 3 task 数 | 1009 | 1009 | — |
| Reducer 10 task 数 | 2 | 2 | — |
| 耗时 | **751.50s** 🔥 | **110.62s** | **85.31s** ✅ |
| **Map 1 FAILED** | **30** | 0 | 0 |
| Map 1 KILLED | 1 | 0 | 0 |
| **Reducer 2 FAILED** | **10** | 0 | 0 |
| Reducer 2 KILLED | 26 | 0 | 0 |
| **Reducer 10 FAILED** | **1** | 0 | 0 |

**观察**（三项叠加）：
1. **DAG 拓扑差异**：开 skew = 10 顶点，关 skew = 7 顶点（skew 拆出 3 个额外顶点用于两阶段 GROUP BY）
2. **Task 失败密集发生**：异常那次 Map 1 + Reducer 2 + Reducer 10 总共 **41 次 FAILED + 27 次 KILLED**
3. **耗时 751s - 基线 ~110s = 约 641s 多出来**

**归因（分三层拆解）**：

| 差异来源 | 对耗时的影响 | 证据 |
|---|---|---|
| ① 集群级背景慢化 | ≈ 100 s（比值 1.8×，与其他阶段一致） | four 的 2.26× / four0 的 1.76× |
| ② SQL 参数：开 skew 的 10 顶点拓扑 | ≈ +25 s（正常 110s − 关闭 skew 85s） | 正常 vs 关闭 skew |
| ③ **HIVE-20990 触发大量 task 失败重试** | ≈ +516 s ← **罪魁** | 异常 41 FAILED × ~12s 启动重试开销 = 量级相符 |

**✅ 主因：HIVE-20990 导致 four1 的 Map 1 / Reducer 2 批量触发 AssertionError，Tez 自动重试了 41 个失败的 task attempt，耗时从基线 ~100-110s 放大到 751s**

## 三 · 归因汇总矩阵

| 阶段 | Task 数差异 | FAILED 重试 | 集群背景慢化（≈ 1.5~2.3×） | 主因 |
|---|---|---|---|---|
| four | 相同 | 0 | 2.26× | **纯集群背景** |
| four0 | 相同 | **7** | 1.76× | 集群背景 + 少量重试 |
| **four1** | 相同 | **41** 🔥 | 1.8×（基线）| **HIVE-20990 + 集群背景** |
| four2 | 相同 | 0 | 1.52× | **纯集群背景** |
| four3 | 相同 | 0 | 2.00× | **纯集群背景** |
| insert ads | 差异 | 0 | 1.65× | **纯集群背景** |

**核心结论**：
1. 除 four1 外的 5 个阶段，耗时差异的主因是**集群级背景慢化**，典型慢化倍数 1.5~2.3×
2. **只有 four1 偏离基线**：异常那次慢到 6.8×（正常）/ 8.8×（关闭skew），超出背景慢化 3 倍以上
3. 超出部分的**唯一解释**是 41 次 task attempt 失败重试带来的延迟（对应 HIVE-20990 的 AssertionError 反复触发）

## 五 · 针对每阶段的优化建议（与数据异常问题解耦）

### 建议 1（🟢 高优）：four 阶段深挖，降低 27~62 分钟的瓶颈
- 正常情况下 four 占总耗时 74%
- 5945 个 Map task 是整个流水线最大的 task 数
- 建议方向：
  - 检查 `hive.exec.reducers.bytes.per.reducer=6000000`（6MB）是否过小 → 2025 个 Map 切片合理性
  - 是否能提升 container size（当前 9216MB）让 CPU-bound 的 Map 更快
  - 是否有不必要的 vectorization / orc split overhead

### 建议 2（🟡 中优）：four1 SQL 改写规避 HIVE-20990
- 异常那次的 641s 额外耗时 = HIVE-20990 触发 41 次重试
- 治标方案：`set hive.vectorized.if.expr.mode=good;`
- 治本方案：把 `coalesce(sys.sys_name, sys1.sys_name, '')` 抽到子查询预物化（详见之前的数据异常报告）
- 副作用：目前 skew 优化对 four1 没性能收益（关 skew 反而快 23%），可一并关闭

### 建议 3（🟢 高优）：排查凌晨 04:59~06:25 的集群背景慢化
- 五个阶段（除 four1）都慢 1.5~2.3×，**非常一致**
- 不是本流水线自己的问题，是该时段的集群负载问题
- 排查方向：
  - YARN RM UI：该时段 default 队列使用率 / Pending 任务数
  - Grafana：HDFS RPC / DataNode 读写吞吐
  - NN IO / NameNode RPC Queue
  - 同时段并发调度任务列表

### 建议 4（🟢 可立即）：关闭 four1 上无收益的 skew 参数
- 正常（开 skew）：110.62s
- 关闭 skew：85.31s（-23%）
- 从耗时角度看，这两个参数属于负优化
- 操作：把 four1 前的两行 set 改为 false 或注释

## 六 · 给老板的一句话

> 三次跑耗时差异的总账：**除 four1 外，五个阶段普遍慢 1.5~2.3 倍 = 集群背景慢化**（凌晨 04:59~06:25 这段 YARN/HDFS 负载高），与 SQL 无关；**four1 阶段异常那次额外慢 6 倍以上 = HIVE-20990 偶发触发 41 次 task attempt 失败重试**，Tez 自动重试带来约 500 秒额外耗时。要提速：(1) 彻底治本要挖 four 阶段（占总 74%）；(2) 规避 four1 的 bug 加 `set hive.vectorized.if.expr.mode=good;` 或 SQL 改写；(3) 凌晨调度窗口的集群负载需要独立排查。

## 七 · 附录：证据采集方法

- **DAG 完成事件**：按 `VERTICES: NN/NN [==>>] 100% ELAPSED TIME: T s` 首次出现的行定位
- **vertex 级详情**：向上回溯到最近的 `VERTICES MODE STATUS` 表头，抓取各 Map/Reducer 行
- **FAILED / KILLED 列**：来自 vertex 行的第 7、第 8 数字列
- **业务日参数**：来自 `SchedulerInstanceDTO` 中的 `maxDateTime` 字段
- **所有行号已在文中标注**，可直接回到原日志核对
