> 目标版本：Hive 2.3.x / Hive 3.x | 共计 9+ 个问题点

## 二、详细分析

### DL-001 TxnHandler SQL 死锁重试机制

- **严重程度**: P1
- **影响版本**: Hive 2.3.x、Hive 3.x
- **涉及文件**:
  - `standalone-metastore/metastore-server/src/main/java/org/apache/hadoop/hive/metastore/txn/retry/SqlRetryHandler.java:45-178`
  - `standalone-metastore/metastore-server/src/main/java/org/apache/hadoop/hive/metastore/txn/TxnHandler.java:165,254`
- **错误堆栈特征**:
  ```
  WARN  SqlRetryHandler - Deadlock detected in XXX. Will wait XXX ms try again up to X times.
  ERROR SqlRetryHandler - Too many repeated deadlocks in XXX, giving up.
  ```
- **根因分析**:
  Metastore 事务操作使用 READ_COMMITTED 隔离级别以减少死锁，但在高并发场景（多个 Compaction 线程 + 多个客户端并发操作 ACID 表）仍会触发 SQL 死锁。重试机制最多 10 次（`ALLOWED_REPEATED_DEADLOCKS=10`），重试间隔递增。
  ```java
  // SqlRetryHandler.java:45
  private static final int ALLOWED_REPEATED_DEADLOCKS = 10;
  // SqlRetryHandler.java:163-178
  private boolean checkDeadlock(SQLException e, SqlRetryCallProperties properties) {
      if (databaseProduct.isDeadlock(e)) {
          if (properties.getDeadlockCount() > 0) {
              long waitInterval = deadlockRetryInterval * (ALLOWED_REPEATED_DEADLOCKS - properties.getDeadlockCount());
              LOG.warn("Deadlock detected in {}. Will wait {} ms try again up to {} times.",
                  properties.getCaller(), waitInterval, properties.getDeadlockCount());
              // 递增等待重试...
  ```
  死锁检测依赖 `DatabaseProduct.isDeadlock()`，支持 MySQL/Postgres/Oracle/Derby 的死锁 SQLState 匹配。可通过 `TXN_RETRYABLE_SQLEX_REGEX` 自定义匹配模式。
- **因果链**: `高并发 ACID 操作` → `SQL 行级锁冲突` → `数据库死锁` → `重试 10 次` → `重试耗尽 → DDL/DML 失败`
- **社区状态**: 设计如此（重试机制本身是修复方案），但 10 次重试上限可能不够
- **参数规避**:
  ```xml
  <!-- 自定义可重试的 SQL 异常模式 -->
  <property>
    <name>metastore.txn.retryable.sqlex.regex</name>
    <value>^Deadlock detected, roll back,.*08177.*,.*08178.*</value>
  </property>
  <!-- 增加 Metastore 后端数据库连接池大小 -->
  <property>
    <name>datanucleus.connectionPool.maxPoolSize</name>
    <value>30</value>
  </property>
  ```
- **验证方法**: 检查 Metastore 日志中 `Deadlock detected` 的频率

### DL-002 HiveSessionImpl MetaStoreClient 死锁

- **严重程度**: P1
- **影响版本**: Hive 3.x
- **涉及文件**: `service/src/java/org/apache/hive/service/cli/session/HiveSessionImpl.java:407-408`
- **错误堆栈特征**: HS2 线程 dump 中出现 `HiveMetaStoreClient$SynchronizedHandler` 的 BLOCKED 状态
- **根因分析**:
  代码注释明确指出存在死锁风险：
  ```java
  // HiveSessionImpl.java:407-408
  // the risk of deadlock on HiveMetaStoreClient#SynchronizedHandler can happen.
  // Refresh the thread-local Hive to avoid the deadlock.
  ```
  `HiveMetaStoreClient` 使用 `SynchronizedHandler`（动态代理）对所有方法加同步锁。当 HS2 多线程并发访问同一个 MetaStoreClient（通过 SessionState 共享）时，如果一个线程持有 MetaStoreClient 锁并等待编译锁，而另一个线程持有编译锁并等待 MetaStoreClient 锁，就会死锁。
- **因果链**: `多线程共享 MetaStoreClient` → `SynchronizedHandler 全方法同步` → `与编译锁交叉持有` → `死锁`
- **社区状态**: 代码中通过 `refreshThreadLocalHive` 部分规避，但根本问题（MetaStoreClient 非线程安全的设计）未解决
- **参数规避**:
  ```xml
  <!-- 避免同一 session 并发执行多个查询 -->
  <property>
    <name>hive.server2.parallel.ops.in.session</name>
    <value>false</value>
  </property>
  ```
- **验证方法**: `jstack <pid>` 检查是否有 `SynchronizedHandler` 相关的 BLOCKED 线程

### DL-003 HS2 编译锁超时

- **严重程度**: P1
- **影响版本**: Hive 2.1.0+（HIVE-12431 引入编译锁超时支持）
- **涉及文件**: `ql/src/java/org/apache/hadoop/hive/ql/lock/CompileLockFactory.java:54`
- **错误堆栈特征**:
  ```
  ERROR - Waiting to acquire compile lock timed out after XXX seconds
  HiveSQLException: Error while compiling statement: ...
  ```
- **根因分析**:
  HS2 使用全局/session 级编译锁来序列化 SQL 编译过程。高并发查询场景下，编译锁成为严重瓶颈。
  ```java
  // CompileLockFactory.java:54
  HiveConf.ConfVars.HIVE_SERVER2_COMPILE_LOCK_TIMEOUT, TimeUnit.SECONDS
  ```
- **社区状态**:
  - **HIVE-12431** (Closed/Fixed in **Hive 2.1.0**) — 引入编译锁超时支持
  - Hive 3.x 引入并行编译支持（`hive.driver.parallel.compilation`）
- **修复版本**: Hive 2.1.0（超时支持）、Hive 3.x（并行编译）
- **参数规避**:
  ```xml
  <!-- Hive 2.3.x -->
  <property>
    <name>hive.server2.compile.lock.timeout</name>
    <value>60</value>  <!-- 秒，默认0=无限等待 -->
  </property>

  <!-- Hive 3.x 开启并行编译 -->
  <property>
    <name>hive.driver.parallel.compilation</name>
    <value>true</value>
  </property>
  <property>
    <name>hive.driver.parallel.compilation.global.limit</name>
    <value>3</value>
  </property>
  ```
- **验证方法**: 监控日志中 `Waiting to acquire compile lock` 的频率和耗时

### DL-004 LLAP BuddyAllocator 死锁

- **严重程度**: P2
- **影响版本**: Hive 2.3.x (LLAP)、Hive 3.x (LLAP)
- **涉及文件**: `llap-server/src/java/org/apache/hadoop/hive/llap/cache/BuddyAllocator.java:535,1006,1040`
- **根因分析**: 多线程 eviction 场景下，一个线程持有自己的 victim 列表锁并等待 free list 锁，另一个线程反向持有。代码中有专门的解锁逻辑避免死锁：
  ```java
  // BuddyAllocator.java:535
  /* free lists, or unlocked due to some lock being held and deadlock potential. */
  // BuddyAllocator.java:1006
  // list. That may deadlock with another thread locking its own victims
  // BuddyAllocator.java:1040
  /* See the comment at the call site; this is to prevent deadlocks. */
  ```
- **社区状态**: 代码中已有死锁预防逻辑，但在极端高并发下仍可能出现
- **参数规避**: 增大 LLAP 缓存大小减少 eviction 频率

### DL-005 WorkloadManager 自死锁

- **严重程度**: P2
- **影响版本**: Hive 3.x
- **涉及文件**: `ql/src/java/org/apache/hadoop/hive/ql/exec/tez/WorkloadManager.java:1700`
- **根因分析**: 显式触发 WM 操作时可能与内部锁交叉导致自死锁
- **参数规避**: 避免通过 API 频繁触发 WM 操作，依赖自动触发机制

### DL-006 TaskExecutorService 同步限制

- **严重程度**: P2
- **影响版本**: Hive 2.3.x (LLAP)、Hive 3.x (LLAP)
- **涉及文件**: `llap-server/src/java/org/apache/hadoop/hive/llap/daemon/impl/TaskExecutorService.java:1265`
- **根因分析**: 调度方法不能加 synchronized，否则与内部调用形成死锁。当前通过注释约束。

### DL-007 LeaseLeaderElection 死锁

- **严重程度**: P2
- **影响版本**: Hive 3.x（HMS HA）
- **涉及文件**: `standalone-metastore/metastore-server/src/main/java/org/apache/hadoop/hive/metastore/leader/LeaseLeaderElection.java:319`
- **根因分析**: Leader 选举操作在某些场景下会触发死锁
- **参数规避**: 确保 ZooKeeper 连接稳定，HMS HA 节点网络延迟低

### DL-008 LrfuCachePolicy 锁竞争

- **严重程度**: P3
- **影响版本**: Hive 2.3.x (LLAP)、Hive 3.x (LLAP)
- **涉及文件**: `llap-server/src/java/org/apache/hadoop/hive/llap/cache/LowLevelLrfuCachePolicy.java:141`
- **根因分析**: 缓冲区入口时堆访问锁竞争，影响 LLAP IO 性能

### DL-009 ServiceStateChangeListener 死锁风险

- **严重程度**: P3
- **影响版本**: Hive 2.3.x、Hive 3.x
- **涉及文件**: `service/src/java/org/apache/hive/service/ServiceStateChangeListener.java:38`
- **根因分析**: 子类如果在回调中调用其他同步方法，可能形成死锁

## 三、运维诊断命令

```bash
# 检查 HS2 是否有死锁
jstack $(pgrep -f HiveServer2) | grep -A5 "Found one Java-level deadlock"

# 检查 BLOCKED 线程
jstack $(pgrep -f HiveServer2) | grep -c "BLOCKED"

# 检查编译锁等待
grep "Waiting to acquire compile lock" /var/log/hive/hiveserver2.log | tail -20

# 检查 Metastore 死锁重试
grep "Deadlock detected" /var/log/hive/hive-metastore.log | tail -20

# 检查 Metastore 后端数据库连接
grep "Too many repeated deadlocks" /var/log/hive/hive-metastore.log
```
