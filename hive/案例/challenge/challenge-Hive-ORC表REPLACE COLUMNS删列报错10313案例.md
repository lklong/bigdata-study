## 一、Challenge 1：第一版分析没用线上代码

### 现象

eric 第一次提问"解释下这个报错"，我直接拉了 `D:\bigdata\gitproject\hive`（社区 master）的源码就给出了完整结论：

- 引用文件：`AlterTableReplaceColumnsOperation.java`
- 错误码：`return code 10313`

### 问题

eric 当场打回："用 txproject/hive 分析"。

### 根本失误

违反了 eric 立的**两条铁律**（memory 34843788）：
> 1. txproject 是腾讯线上代码，gitproject 是社区代码
> 2. 分析问题前先问"看线上还是看社区"，**默认应当先看线上**

我应该在第一次回答前主动澄清"你是想看线上 EMR 实际跑的代码（3.1.3）还是社区主干（master）"，而不是直接默认社区。线上和社区的代码组织差异可能很大，结论也会不同。

### 切到 txproject 后发现的真实差异

| 维度 | 我第一版（社区 master） | 实际线上（txproject 3.1.3） |
|---|---|---|
| 校验代码位置 | `AlterTableReplaceColumnsOperation#doAlteration` | `DDLTask.java` 4172-4213 |
| 类组织 | DDL 已按 Operation 拆类 | 还是巨石类 `DDLTask` |
| 用户可见返回码 | `return code 10313` | **`return code 1`** |
| SerDe 白名单 | 含 IcebergSerDe | 含 DynamicSerDe，不含 Iceberg |

**最关键的差异是返回码**：线上看到的是 `return code 1` 而不是 `10313`。如果 eric 拿着我第一版的结论去线上 grep 日志找 "10313"，会**完全找不到**，排障会卡住。

### 反思

下次涉及任何线上排障类问题，**必须先确认源码仓库**：
1. 默认假设"先看线上代码"，去 `D:\bigdata\txproject\<组件>`
2. 必要时再交叉对比社区代码补充上下文
3. 对比表必须明确标出**用户实际看到的输出差异**（如返回码、日志格式），不能只比较代码逻辑

## 二、Challenge 2：解释颗粒度的迭代

### 现象

eric 第一次问"解释下"，我给了完整的根因分析（带源码、推导链、解决方案）。
eric 第二次又问"`ALTER TABLE ... REPLACE COLUMNS` 这个是什么意思"。

### 反思

第一次回答**直接跳到了"为什么报错"**，跳过了"语法是什么意思"这一层。eric 二次发问其实是在补这一课——他想先吃透**这条 SQL 本身的语义**，再理解"为什么 ORC 不让这么干"。

正确的解释顺序应该是：

```
语法语义（这条 SQL 想干什么）
  ↓
执行层面（Hive 实际做了什么 / 不做什么）
  ↓
为什么会报错（源码校验逻辑）
  ↓
解决方案
```

我把 1、2 两层都隐含在第三层里了，导致信息密度太高、新人或者非排障语境的同事看不懂"是什么"。

### 沉淀

讲解类问题（不是排障类），**必须先讲清楚语义和执行层面**，再讲源码校验。
排障类问题，可以直接从"为什么报错"切入，因为用户已经被错误信息推到这一层了。

## 四、方法论进化

### 进化点 1：分析任何线上问题前的"源码仓库三问"

```
Q1：你想看线上代码还是社区代码？
Q2：你的集群是什么版本？（3.1.3 / 4.0 / master）
Q3：是否需要做双仓对照（确认腾讯有无 patch）？
```

如果 eric 没明说，**默认走线上代码**，并主动告知用了哪个仓库哪条分支。

### 进化点 2：解释类问题的"三层递进"模板

层 1（What）：这个语法/操作是什么意思
层 2（How）：它在系统层面实际做了什么、不做什么
层 3（Why）：为什么会有这个限制 / 为什么会报错

排障类问题可以从层 3 切入；纯学习类问题必须从层 1 起步。

### 进化点 3：报错信息的"用户视角"对照

写排障文档时，**必须列出用户实际在终端看到的报错文本**，而不是只引用源码里的错误码常量。3.1.3 看到的是 `return code 1`，master 看到的是 `return code 10313`，这种差异在 grep 日志时会决定生死。
