# HiveServer2 查询结果不对 — 排障实战案例

## 背景

| 字段 | 值 |
|------|-----|
| **案例编号** | CASE-S04-001 |
| **创建时间** | 2026-04-04 |
| **问题概述** | Hive 查询返回结果不正确（多行/少行/值错/列错/为空），涵盖向量化、JOIN、CBO、PPD、ACID 等子系统 |
| **问题类型** | 数据正确性 |
| **严重程度** | P0（数据错误直接影响业务决策） |
| **影响范围** | 所有通过 HS2 提交的查询 |
| **涉及组件** | HiveServer2、Tez/MR/Spark 执行引擎、向量化引擎、CBO (Calcite)、ORC/Parquet 存储层 |

**问题环境**：
- **Hive 版本**: 3.x 社区版
- **执行引擎**: Tez / MapReduce / Spark（混合使用）
- **问题表现**: 查询返回的行数或值与预期不符，无报错

---

## 严重程度

- **P0 级**（致命 — 数据直接错误）: 8 个高频 Bug
- **P1 级**（严重 — 特定场景触发）: 12 个高频 Bug
- **P2 级**（中度 — 低频或已修复）: 9 个高频 Bug
- **Hive 3.x 需回合**: 18 个 Critical/Blocker 级 Bug（修复在 4.x 但影响 3.x）

## 最致命的 5 个问题

| # | Bug | 现象 | 根因 | 影响范围 |
|---|-----|------|------|---------|
| 1 | 向量化 NULL 变 true | Boolean 列 NULL 值查出来变 true | `ColumnBuffer.addValue()` 检查 `nulls` 数组但 ORC 存储 `isNull[i]=true` 时仍写入默认值 false | 所有 ORC 表 Boolean 列 |
| 2 | NOT IN + NULL 丢全部数据 | `WHERE id NOT IN (SELECT id FROM t2)` 当 t2 有 NULL 时返回 0 行 | Anti Join 转换未处理 NULL 语义 (HIVE-26135) | 所有 NOT IN 子查询 |
| 3 | JOIN emit interval 截断 | JOIN 结果随机少行 | `hive.join.emit.interval=1000` 在特定场景截断 outer join 匹配 | 大表 OUTER JOIN |
| 4 | ACID compaction 丢数据 | Major compaction 后 SELECT 少行 | MRCompactor 多 FileSinkOperator 并发写丢数据 (HIVE-28700) | ACID 事务表 |
| 5 | Decimal64 精度丢失 | Decimal 列计算结果精度不对 | `ConvertDecimal64ToDecimal` 转换丢失精度 (HIVE-20004) | Decimal 列向量化计算 |

## 分析维度

| 维度 | 状态 | 关键发现 |
|------|------|----------|
| 向量化执行 | ❌ 严重 | 14 个 Bug：NULL 处理、IF/CASE、聚合、CAST、字符串/日期函数、PTF+DISTINCT |
| JOIN 操作 | ❌ 严重 | 9 个 Bug：emit interval 截断、N-way join 合并、Bucket/SMB Join、Map Join 列混淆 |
| CBO 优化器 | ⚠️ 中度 | 5 个 Bug：相关子查询聚合、EXISTS 重写、IN 条件组合、非确定性函数 PPD |
| NOT IN / 子查询 | ❌ 严重 | 4 个 Bug：NULL 语义、类型强转、EXISTS+LIMIT、物化视图交互 |
| PPD / 存储格式 | ⚠️ 中度 | 4 个 Bug：ORC Schema Evolution、Parquet 缺失列、Timestamp PPD、压缩文本 split |
| ACID / 事务 | ❌ 严重 | 5 个 Bug：compaction 丢数据、LOAD DATA 结果错、UPDATE 列错乱、重复 ROW__ID |
| 数据类型精度 | ⚠️ 中度 | 4 个 Bug：Decimal 精度、Float toString 中转、Timestamp 时区、CHAR 空格 |
| JDBC 客户端 | ⚠️ 中度 | ColumnBuffer 源码缺陷导致 NULL/Float 值失真 |

## 根因分析

### 因果链

```
[根因层 — 代码缺陷]
├── 向量化引擎：ColumnVector NULL 标记与默认值逻辑不一致
├── JOIN 算子：joinEmitInterval 截断 + N-way join 合并条件丢失
├── CBO (Calcite)：子查询 decorrelation 未处理边界情况
├── PPD：谓词下推到存储层后与 Schema Evolution 不兼容
├── ACID：compaction 并发写 + ROW__ID 生成不唯一
└── JDBC 客户端：ColumnBuffer 双重创建 RowSet + Float toString 精度丢失

    ↓↓↓

[中间影响]
├── 向量化/CBO/PPD：查询计划在编译期已经错误 → 执行时无法自愈
├── JOIN/ACID：运行时数据流被截断或错乱 → 结果集行数/值不可预测
└── JDBC：正确的服务端结果经 Thrift 传输到客户端后被二次损坏

    ↓↓↓

[最终现象 — 5 种结果错误]
├── 数据多了（重复行/多余行）
├── 数据少了（丢行/缺行）
├── 数据值错了（精度/NULL/聚合不对）
├── 数据列错了（列错位/列数据串了）
└── 数据集为空（该有结果返回 0 行）
```

### 核心排障思路

```
[一键关闭所有优化]
SET hive.vectorized.execution.enabled=false;
SET hive.vectorized.execution.reduce.enabled=false;
SET hive.cbo.enable=false;
SET hive.auto.convert.join=false;
SET hive.join.emit.interval=0;
SET hive.optimize.ppd=false;
SET hive.optimize.index.filter=false;
SET hive.compute.query.using.stats=false;
SET hive.optimize.constant.propagation=false;
SET hive.optimize.semijoin.conversion=false;

→ 结果正确了？ → 逐个开启，二分定位到哪个子系统
→ 还不对？ → 数据本身有问题 / ACID 表需检查 compaction / JDBC 客户端缺陷
```

## 操作路径（按优先级排序）

### 🔴 紧急操作（立即执行 — 确认结果是否真的错了）

#### 操作 1：一键对比法

```bash
# 在 beeline 中执行以下 SET 语句，关闭所有可能引入错误的优化：
beeline -u "jdbc:hive2://<host>:<port>/default" -n <user> <<'EOF'
SET hive.vectorized.execution.enabled=false;
SET hive.vectorized.execution.reduce.enabled=false;
SET hive.cbo.enable=false;
SET hive.auto.convert.join=false;
SET hive.join.emit.interval=0;
SET hive.optimize.ppd=false;
SET hive.optimize.index.filter=false;
SET hive.compute.query.using.stats=false;
SET hive.optimize.constant.propagation=false;
SET hive.optimize.semijoin.conversion=false;

-- 执行你的问题查询
<YOUR_QUERY_HERE>;
EOF
```

- **预期效果**：如果关闭所有优化后结果正确，说明是优化器/引擎 Bug，不是数据问题
- **风险提示**：关闭优化后查询可能变慢（尤其大表 JOIN），仅用于验证
- **回滚方案**：不需要回滚，SET 只影响当前 session

#### 操作 2：逐个开启二分定位

```bash
# 在操作 1 的基础上，逐个开启优化，找到导致结果错的那个：
# 1. 先开向量化
SET hive.vectorized.execution.enabled=true;
-- 执行查询，如果结果又错了 → 就是向量化 Bug

# 2. 如果向量化没问题，继续开 CBO
SET hive.cbo.enable=true;
-- 执行查询...

# 3. 依次开启 auto.convert.join、ppd、index.filter 等
# 哪个开了结果就错 → 锁定那个子系统
```

- **预期效果**：精确定位到哪个优化开关引入了错误
- **风险提示**：每次只开一个，避免交叉影响

---

### 🟡 短期修复（当天完成 — 配置调优绕过 Bug）

#### 操作 3：针对性关闭有问题的优化

根据操作 2 定位的子系统，在 `hive-site.xml` 中持久化关闭：

```xml
<!-- 示例：如果确认是向量化 Bug -->
<property>
  <name>hive.vectorized.execution.enabled</name>
  <value>false</value>
  <description>关闭向量化绕过 HIVE-XXXXX Bug，待升级后开启</description>
</property>

<!-- 示例：如果确认是 join emit interval 截断 -->
<property>
  <name>hive.join.emit.interval</name>
  <value>0</value>
  <description>关闭 emit interval 截断，绕过 OUTER JOIN 丢行 Bug</description>
</property>
```

- **预期效果**：持久化绕过 Bug，但可能牺牲部分查询性能
- **风险提示**：关闭向量化可能导致查询速度下降 2-5 倍；关闭 CBO 可能导致 JOIN 顺序不优
- **回滚方案**：恢复 hive-site.xml 原值，重启 HS2

#### 操作 4：使用验证 SQL 确认是否命中已知 Bug

```bash
# 将本案例附带的验证脚本上传到 Hive 环境执行
# 脚本会自动建表、插数据、对比结果，输出命中的 Bug 列表
bash scripts/verify-result-bugs.sh <hs2_host> <hs2_port> <user>
```

## 验证方法

### 验证 Bug 定位是否准确

```bash
# 1. 用默认参数执行问题 SQL，记录结果 A
beeline -u "jdbc:hive2://<host>:<port>/default" -n <user> \
  -e "YOUR_QUERY" > /tmp/result_default.txt

# 2. 关闭所有优化执行，记录结果 B
beeline -u "jdbc:hive2://<host>:<port>/default" -n <user> \
  -e "SET hive.vectorized.execution.enabled=false; ... YOUR_QUERY" > /tmp/result_safe.txt

# 3. 对比
diff /tmp/result_default.txt /tmp/result_safe.txt
# 有 diff → 优化器 Bug
# 无 diff → 数据本身有问题或 JDBC 客户端缺陷
```

### 验证修复是否生效

```bash
# 执行验证脚本，检查之前命中的 Bug 是否已修复
bash scripts/verify-result-bugs.sh <host> <port> <user>
# 预期：所有测试 PASS
```

## 预防措施

### 监控告警

| 监控项 | 阈值 | 告警级别 | 采集方式 |
|--------|------|---------|----------|
| 查询结果行数突变 | 同一查询结果行数变化 > 10% | WARNING | 业务侧对账 |
| ACID compaction 失败 | 失败次数 > 0 | CRITICAL | `SHOW COMPACTIONS` 监控 |
| HS2 升级后回归测试 | 任一验证 SQL 结果变化 | CRITICAL | CI/CD 流水线 |

### 定期巡检

- **每周**：执行验证 SQL 脚本，确认核心查询结果稳定
- **每月**：检查社区 JIRA，关注新发现的结果正确性 Bug
- **升级前**：必须执行全量验证 SQL 回归测试
