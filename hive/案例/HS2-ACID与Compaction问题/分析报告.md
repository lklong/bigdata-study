## 二、详细分析

### AC-001 CompactionInfo 序列化重置 Bug (HIVE-21056)

- **严重程度**: P2
- **影响版本**: Hive 3.0+
- **涉及文件**: `standalone-metastore/metastore-server/src/main/java/org/apache/hadoop/hive/metastore/txn/entities/CompactionInfo.java:45`
- **错误堆栈特征**: Compaction 信息丢失字段、Compaction 状态不一致
- **根因分析**:
  `CompactionInfo` 被序列化为 `CompactionInfoStruct` 时，如果 `CompactionInfo` 中新增了字段，可能遗漏在序列化函数中。代码注释明确指出：
  ```java
  // CompactionInfo.java:45
  // being resetted. This will be fixed at HIVE-21056.
  ```
  HIVE-21056 提出让 `CompactionInfo` 继承 `CompactionInfoStruct` 以防止此类 Bug。
- **因果链**: `CompactionInfo 新增字段` → `序列化函数未同步更新` → `字段丢失/被重置` → `Compaction 行为异常`
- **社区状态**:
  - **HIVE-21056** — 状态：**Patch Available / Unresolved**（2018年12月创建，至今未合并）
  - 优先级：Major
  - 提议方案：让 CompactionInfo 继承 CompactionInfoStruct
- **自研方案**: 如遇到 Compaction 信息丢失，可手动检查 `COMPACTION_QUEUE` 表中的数据一致性
- **参数规避**: 无直接参数规避，建议监控 `SHOW COMPACTIONS` 输出的完整性
- **验证方法**: `SHOW COMPACTIONS;` 检查是否有字段值异常（null 或被重置）

### AC-002 alter query 时 writeId 不正确 (HIVE-28803)

- **严重程度**: P2
- **影响版本**: Hive 3.x（trunk 代码中已标注）
- **涉及文件**: `standalone-metastore/metastore-server/src/main/java/org/apache/hadoop/hive/metastore/directsql/MetaStoreDirectSql.java:591`
- **错误堆栈特征**: alter table 后统计信息异常、writeId 不匹配告警
- **根因分析**:
  ```java
  // MetaStoreDirectSql.java:591
  // HIVE-28803 is changing the writeId from -1 to 0 during alter query which is undesirable
  ```
  在 alter 查询期间，writeId 从 -1 被错误地修改为 0，这会导致 ACID 表的写入 ID 跟踪异常。
- **因果链**: `alter query 修改 writeId` → `writeId 从 -1 变为 0` → `后续 ACID 操作的 writeId 校验异常` → `数据一致性风险`
- **社区状态**: 代码中已标注 HIVE-28803，具体修复进展需跟踪
- **参数规避**: 避免在 ACID 表上频繁执行 alter 操作；alter 后手动更新统计信息：
  ```sql
  ANALYZE TABLE <table> COMPUTE STATISTICS;
  ```

### AC-003 Compaction 在分桶表上不支持 Rebalancing

- **严重程度**: P2
- **影响版本**: Hive 3.x
- **涉及文件**: `itests/hive-unit/src/test/java/org/apache/hadoop/hive/ql/txn/compactor/TestCrudCompactorOnTez.java:447`
- **错误堆栈特征**:
  ```
  Cannot execute rebalancing compaction on bucketed tables.
  ```
- **根因分析**: Rebalancing Compaction（重平衡压缩）不支持分桶表。这是设计限制而非 bug，因为分桶表的文件分布有固定要求。
- **参数规避**: 对分桶表使用 major/minor compaction 而非 rebalancing：
  ```sql
  ALTER TABLE <table> COMPACT 'major';
  ```

### AC-004 Compaction 在结构化字段/关键字字段表上失败

- **严重程度**: P2
- **影响版本**: Hive 3.x 早期版本
- **涉及文件**: `itests/hive-unit/src/test/java/org/apache/hadoop/hive/ql/txn/compactor/TestCrudCompactorOnTez.java:579,3070`
- **根因分析**:
  - `testCompactionShouldNotFailOnPartitionsWithBooleanField` — Boolean 分区字段可能导致 Compaction 失败
  - `testCompactionShouldNotFailOnStructField` — 结构化字段表的 Compaction 问题
  - `testCompactionShouldNotFailOnKeywordField` — 字段名为 SQL 关键字时 Compaction 失败
- **社区状态**: 测试名中的 "ShouldNotFail" 表明这些是已修复的回归测试
- **修复版本**: Hive 3.x 后续版本
- **参数规避**: 如遇 Compaction 失败，检查表的列名是否包含 SQL 关键字，用反引号转义

### AC-005 事务中止阈值与 Compaction 清理

- **严重程度**: P3
- **影响版本**: Hive 3.x
- **涉及文件**: `ql/src/java/org/apache/hadoop/hive/ql/txn/compactor/AcidTableOptimizer.java:49-51`
- **根因分析**:
  当中止事务数超过阈值 `HIVE_COMPACTOR_ABORTEDTXN_THRESHOLD` 或中止事务存在时间超过 `HIVE_COMPACTOR_ABORTEDTXN_TIME_THRESHOLD` 时，自动触发清理。如果清理速度跟不上中止速度，delta 文件会持续堆积。
- **参数规避**:
  ```xml
  <!-- 中止事务数阈值（默认1000） -->
  <property>
    <name>hive.compactor.abortedtxn.threshold</name>
    <value>500</value>  <!-- 降低阈值更快触发清理 -->
  </property>
  <!-- 中止事务时间阈值 -->
  <property>
    <name>hive.compactor.abortedtxn.time.threshold</name>
    <value>3600000</value>  <!-- 1小时 -->
  </property>
  <!-- Compaction Worker 线程数 -->
  <property>
    <name>hive.compactor.worker.threads</name>
    <value>4</value>  <!-- 增加 Worker 加速清理 -->
  </property>
  <!-- Initiator 线程数 -->
  <property>
    <name>hive.compactor.initiator.on</name>
    <value>true</value>
  </property>
  ```

### AC-006 Insert Overwrite 后 Minor Compaction 导致 Major Compaction 失败

- **严重程度**: P2
- **影响版本**: Hive 3.x 早期版本
- **涉及文件**: `itests/hive-unit/src/test/java/org/apache/hadoop/hive/ql/txn/compactor/TestCrudCompactorOnTez.java:2398`
- **根因分析**: 执行 INSERT OVERWRITE 后紧接 Minor Compaction，会导致后续的 Major Compaction 失败。这是因为 Minor Compaction 的输出格式与 INSERT OVERWRITE 产生的 base 文件不兼容。
- **社区状态**: 已通过测试覆盖，后续版本已修复
- **参数规避**: INSERT OVERWRITE 后直接执行 Major Compaction 而非 Minor

## 三、ACID 问题诊断命令

```bash
# 查看 Compaction 状态
beeline -e "SHOW COMPACTIONS;"

# 查看当前活跃事务
beeline -e "SHOW TRANSACTIONS;"

# 查看锁信息
beeline -e "SHOW LOCKS;"

# 检查 delta 文件堆积
hdfs dfs -ls -R /user/hive/warehouse/<db>/<table>/ | grep delta | wc -l

# 检查中止事务数
beeline -e "SELECT COUNT(*) FROM TXNS WHERE TXN_STATE='a';"  -- 直接查 Metastore DB

# 手动触发 Compaction
beeline -e "ALTER TABLE <db>.<table> COMPACT 'major';"

# 检查 Compaction 历史
beeline -e "SHOW COMPACTIONS;" | grep -E "failed|refused"
```

## 四、Hive 2.3.x vs 3.x ACID 差异

| 特性 | Hive 2.3.x | Hive 3.x |
|------|-----------|----------|
| ACID 默认状态 | 关闭 | **开启** |
| Full ACID | 支持（需手动开启） | 默认支持 |
| INSERT-only ACID | 不支持 | 支持（Managed 表默认） |
| Compaction | 基础 | 增强（Rebalancing 等） |
| standalone-metastore | 不存在 | 独立模块 |
| TxnHandler | 在 metastore 包内 | 独立 txn 子包 |

**建议**：Hive 2.3.x 如不需要 ACID 功能，建议关闭以避免潜在问题：
```xml
<property><name>hive.support.concurrency</name><value>false</value></property>
<property><name>hive.txn.manager</name><value>org.apache.hadoop.hive.ql.lockmgr.DummyTxnManager</value></property>
```
