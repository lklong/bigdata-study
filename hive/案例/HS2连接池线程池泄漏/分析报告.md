# HS2/HMS 连接泄漏与线程池问题集

> 目标版本：Hive 2.3.x / Hive 3.x | 共计 5+ 个问题点

---

## 一、问题总览

| 问题 | 位置 | 严重程度 | 典型现象 |
|------|------|----------|----------|
| SQLOperation 子线程 HMS 连接泄漏 | HS2 Operation | P1 | HMS 连接数持续增长 |
| TaskRunner HMS 连接泄漏 | HS2 Task | P1 | 同上 |
| backgroundOperationPool RejectedExecution | HS2 SessionManager | P1 | 异步查询提交失败 |
| backgroundOperationPool 无容量上限 | HS2 SessionManager | P2 | 线程数无限增长 |
| Metastore 连接池配置问题 | HMS Server | P2 | HMS 连接池耗尽 |

---

## 二、详细分析

### CL-001 SQLOperation 子线程 HMS 连接泄漏

- **严重程度**: P1
- **影响版本**: Hive 2.3.x、Hive 3.x
- **涉及文件**: `service/src/java/org/apache/hive/service/cli/operation/SQLOperation.java:344-347`
- **错误堆栈特征**: HMS 日志中连接数持续增长、`Too many connections` 错误
- **根因分析**:
  异步查询在后台线程池中执行时，子线程可能创建新的 `Hive` 对象（内含 HMS Thrift 连接）。如果不在 finally 中调用 `Hive.closeCurrent()`，该连接将泄漏。代码中已有修复注释：
  ```java
  // SQLOperation.java:344-347
  // If new hive object is created by the child thread, then we need to close it
  // as it might have created a hms connection. Call Hive.closeCurrent() that
  // closes the HMS connection, causes HMS connection leaks otherwise.
  Hive.closeCurrent();
  ```
  **Hive 3.x 已在此处添加了 `Hive.closeCurrent()` 调用**，但 Hive 2.3.x 的部分版本可能缺少此修复。
- **因果链**: `子线程创建新 Hive 对象` → `Thrift HMS 连接打开` → `未在 finally 中关闭` → `HMS 连接泄漏` → `数据库连接池耗尽`
- **社区状态**: Hive 3.x 中已添加 `Hive.closeCurrent()` 调用
- **参数规避**:
  ```xml
  <!-- 使用远程 Metastore，限制 HMS 端连接池 -->
  <property>
    <name>datanucleus.connectionPool.maxPoolSize</name>
    <value>30</value>
  </property>
  <!-- HMS 端连接超时 -->
  <property>
    <name>javax.jdo.option.ConnectionDriverName</name>
    <value>com.mysql.cj.jdbc.Driver</value>
  </property>
  ```
- **验证方法**: 监控 HMS 后端数据库的连接数：`SHOW PROCESSLIST;` (MySQL)

### CL-002 TaskRunner HMS 连接泄漏

- **严重程度**: P1
- **影响版本**: Hive 2.3.x、Hive 3.x
- **涉及文件**: `ql/src/java/org/apache/hadoop/hive/ql/exec/TaskRunner.java:87`
- **根因分析**: 与 CL-001 同源，TaskRunner 在执行 Task 时也可能创建新的 Hive 对象。代码注释：
  ```java
  // TaskRunner.java:87
  // HMS connection leaks otherwise.
  ```
- **社区状态**: Hive 3.x 已添加关闭逻辑

### CL-003 backgroundOperationPool RejectedExecutionException

- **严重程度**: P1
- **影响版本**: Hive 2.3.x、Hive 3.x
- **涉及文件**:
  - `service/src/java/org/apache/hive/service/cli/operation/SQLOperation.java:285`
  - `service/src/java/org/apache/hive/service/cli/operation/hplsql/HplSqlOperation.java:77`
- **错误堆栈特征**:
  ```
  HiveSQLException: The background threadpool cannot accept new task for execution, please retry the operation
  java.util.concurrent.RejectedExecutionException
  ```
- **根因分析**:
  当 HS2 的后台操作线程池满时（所有线程都在执行查询），新的异步查询提交会被拒绝。
  ```java
  // SQLOperation.java:285
  } catch (RejectedExecutionException rejected) {
      setState(OperationState.ERROR);
      throw new HiveSQLException("The background threadpool cannot accept" +
          " new task for execution, please retry the operation", rejected);
  }
  ```
- **因果链**: `并发查询过多` → `线程池满` → `RejectedExecutionException` → `查询失败`
- **参数规避**:
  ```xml
  <!-- 增加后台操作线程池大小 -->
  <property>
    <name>hive.server2.async.exec.threads</name>
    <value>200</value>  <!-- 默认100，根据并发量调整 -->
  </property>
  <!-- 等待队列大小 -->
  <property>
    <name>hive.server2.async.exec.wait.queue.size</name>
    <value>200</value>  <!-- 默认100 -->
  </property>
  <!-- 空闲线程存活时间 -->
  <property>
    <name>hive.server2.async.exec.keepalive.time</name>
    <value>60</value>  <!-- 秒 -->
  </property>
  ```
- **验证方法**: 监控日志中 `RejectedExecutionException` 的出现频率

### CL-004 backgroundOperationPool 无容量上限保护

- **严重程度**: P2
- **影响版本**: Hive 2.3.x、Hive 3.x
- **涉及文件**: `service/src/java/org/apache/hive/service/cli/session/SessionManager.java:235`
- **根因分析**:
  ```java
  // SessionManager.java:235
  backgroundOperationPool.allowCoreThreadTimeOut(true);
  ```
  线程池使用 `allowCoreThreadTimeOut(true)` 但核心线程数可能配置过大。没有绝对的线程数硬限制，高并发下可能创建大量线程。
- **参数规避**:
  ```xml
  <property>
    <name>hive.server2.async.exec.threads</name>
    <value>100</value>  <!-- 根据实际并发需求设置合理值 -->
  </property>
  ```

---

### CL-005 Metastore 连接池配置与边界情况

- **严重程度**: P2
- **影响版本**: Hive 2.3.x、Hive 3.x
- **涉及文件**:
  - `ql/src/test/org/apache/hadoop/hive/metastore/txn/TestTxnHandlerWithOneConnection.java`
  - `ql/src/test/org/apache/hadoop/hive/metastore/txn/TestTxnHandlerNoConnectionPool.java`
- **根因分析**: 测试验证了连接池大小为 1 或无连接池时 TxnHandler 的行为。在生产中如果连接池配置过小，高并发下容易出现连接等待超时。
- **参数规避**:
  ```xml
  <!-- Metastore 后端数据库连接池（BoneCP/HikariCP/DBCP） -->
  <property>
    <name>datanucleus.connectionPool.maxPoolSize</name>
    <value>30</value>
  </property>
  <property>
    <name>datanucleus.connectionPoolingType</name>
    <value>HikariCP</value>  <!-- 推荐使用 HikariCP -->
  </property>
  <!-- MySQL 连接超时 -->
  <property>
    <name>javax.jdo.option.ConnectionURL</name>
    <value>jdbc:mysql://host:3306/hive_meta?connectTimeout=5000&amp;socketTimeout=30000</value>
  </property>
  ```

## 三、连接泄漏诊断命令

```bash
# 检查 HS2 到 HMS 的 Thrift 连接数
netstat -an | grep <HMS_PORT> | grep ESTABLISHED | wc -l

# 检查 HMS 后端数据库连接数
mysql -e "SHOW PROCESSLIST;" | grep hive_meta | wc -l

# 检查 HS2 线程数
jstack $(pgrep -f HiveServer2) | grep "^\"" | wc -l

# 检查后台操作线程池状态
jstack $(pgrep -f HiveServer2) | grep "HiveServer2-Background-Pool" | wc -l

# 检查 RejectedExecution 日志
grep "RejectedExecutionException\|background threadpool cannot accept" /var/log/hive/hiveserver2.log | tail -10
```

```xml
<!-- Session 管理 -->

<property><name>hive.server2.session.check.interval</name><value>300000</value></property>

<property><name>datanucleus.connectionPool.maxPoolSize</name><value>30</value></property>
<property><name>datanucleus.connectionPoolingType</name><value>HikariCP</value></property>
```
