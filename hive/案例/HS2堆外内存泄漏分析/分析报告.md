# HiveServer2 堆外内存泄漏深度源码分析

## 1. 问题概述

- **问题描述**：HiveServer2 进程 RSS 持续增长，远超 -Xmx 设定值，表明存在大量堆外（off-heap / native）内存不释放
- **问题类型**：资源泄漏 / 内存管理缺陷
- **严重程度**：P0（服务可用性风险 — 最终导致 OOM Killer）
- **影响范围**：HS2 长时间运行后必现
- **泄漏点总数**：**27 个**（P0: 5 / P1: 6 / P2: 16）

## 2. 分析维度总览

| 维度 | 状态 | 关键发现 |
|------|------|----------|
| DirectByteBuffer / Native 内存 | ❌ 严重 | LLAP BuddyAllocator 无 close 方法、MappedByteBuffer 无 unmap、CleanerUtil 单点故障 |
| ThreadLocal 泄漏 | ❌ 严重 | 11+ 个 ThreadLocal 无 remove、GlobalWorkMapFactory Map 无限增长 |
| 线程池 / 线程泄漏 | ⚠️ 中度 | ThreadWithGarbageCleanup 依赖 finalize()、多个 static 线程池无 shutdown |
| Thrift / Netty 网络层 | ⚠️ 中度 | SSL 缓冲区、SASL 原生上下文、Jetty DirectByteBuffer |
| ClassLoader / Metaspace | ❌ 严重 | Beeline addLocalJar 从不关闭旧 CL、UDF Registry 钉住 CL |
| Deflater/Inflater 原生内存 | ⚠️ 中度 | InputJobInfo.java 中 Deflater 未调用 end() |
| Session 关闭路径 | ⚠️ 中度 | detachSession 顺序问题、DriverState 未清理 |

## 3. 因果链

```
[根因] ThreadLocal 无 remove + DirectByteBuffer 无显式释放 + ClassLoader 无 close
    → [中间影响] 线程池复用线程积累大对象、DirectByteBuffer 依赖 GC Finalizer、Metaspace 持续增长
        → [最终现象] RSS 远超 Xmx，持续增长直到 OOM Killer 杀死进程
```

**置信度**: 高（源码直接证据）

## 4. 根因分析（按严重程度排序）

### 🔴 P0 级 — 致命泄漏点

#### OHL-001: GlobalWorkMapFactory ThreadLocal — 执行计划 Map 无限增长
- **文件**: `ql/src/java/org/apache/hadoop/hive/ql/exec/GlobalWorkMapFactory.java:35,105-110`
- **根因**: 在 HiveServer2 查询模式下（`isHiveServerQuery()`），每次查询都通过 `ThreadLocal<Map<Path, BaseWork>>` 存入执行计划数据，但**没有任何 `remove()` 或 `clear()` 调用**
- **影响**: BaseWork 可达数十MB，随线程池复用，Map 只增不减 → 持续消耗堆内存（间接增加 Metaspace 和 native 内存压力）
- **复现条件**: HS2 使用 Binary/HTTP 模式接收查询 + 线程池复用（默认行为）
- **源码分析**:
  ```java
  // GlobalWorkMapFactory.java:35
  private static final ThreadLocal<Map<Path, BaseWork>> threadLocalWorkMap = 
      ThreadLocal.withInitial(HashMap::new);
  
  // 查询时 put 到 Map 中
  // 但整个文件中没有任何 remove() 调用！
  ```

#### OHL-002: ThreadWithGarbageCleanup 依赖 finalize() 清理 RawStore
- **文件**: `service/src/java/org/apache/hive/service/server/ThreadWithGarbageCleanup.java:48-52`
- **根因**: HS2 三大线程池（Background Pool、HTTP Handler Pool、Binary Handler Pool）使用此 ThreadFactory，依赖 `finalize()` 来关闭 PersistenceManager 连接。**finalize() 在 Java 9+ 已废弃且不保证执行**
- **影响**: 每个未 finalize 的线程残留一个 RawStore → PersistenceManager → JDBC 连接 → 原生内存
- **源码分析**:
  ```java
  // ThreadWithGarbageCleanup.java:48-52
  @Override
  protected void finalize() throws Throwable {
    // 依赖 finalize() 释放资源 — 不可靠！
    cleanRawStore();
  }
  ```

#### OHL-003: LLAP BuddyAllocator — DirectByteBuffer/MappedByteBuffer 无释放路径
- **文件**: `llap-server/src/java/org/apache/hadoop/hive/llap/cache/BuddyAllocator.java:879-914`
- **根因**: `preallocateArenaBuffer()` 分配每个最大 1GB 的 DirectByteBuffer 或 MappedByteBuffer，BuddyAllocator **没有实现 close()/destroy() 方法**，`LlapIoImpl.close()` 只关闭 MXBean 和线程池，**不释放 BuddyAllocator 的离堆内存**
- **影响**: LLAP 模式下，分配的 GB 级离堆内存在 daemon 关闭后无法归还
- **源码分析**:
  ```java
  // BuddyAllocator.java:879-914
  private ByteBuffer preallocateArenaBuffer(int arenaSize) {
    if (isDirect) {
      return ByteBuffer.allocateDirect(arenaSize); // 最大 1GB!
    } else {
      return isMapped ? mapBuffer(arenaSize) : ByteBuffer.allocate(arenaSize);
    }
  }
  // 整个类没有 close()/destroy() 方法！
  ```

#### OHL-004: CleanerUtil 单点故障 — 所有 DirectByteBuffer 释放链条断裂
- **文件**: `common/src/java/org/apache/hive/common/util/CleanerUtil.java:75-87`
- **根因**: 当 `UNMAP_SUPPORTED = false` 时（特定 JVM 环境/安全策略），SimpleAllocator、EncodedReaderImpl、LlapRecordReaderUtils 中**所有 DirectByteBuffer 都无法主动释放**
- **影响**: 依赖 GC Finalizer 释放，在高负载下延迟极大

#### OHL-005: AuthorizationPreEventListener — 4 个 static ThreadLocal 永不清理
- **文件**: `ql/src/java/org/apache/hadoop/hive/ql/security/authorization/AuthorizationPreEventListener.java:74-112`
- **根因**: 4 个 `static final ThreadLocal`（tConfig/tAuthenticator/tAuthorizers/tConfigSetOnAuths）**没有任何 remove() 调用**
- **影响**: 每个 MetaStore 线程永久持有完整 HiveConf + Auth Provider 对象链

### 🟡 P1 级 — 严重泄漏点

#### OHL-006: Hive ThreadLocal 中 allowClose=false 阻止 MetaStore 连接释放
- **文件**: `ql/src/java/org/apache/hadoop/hive/ql/metadata/Hive.java:285-312`
- **根因**: `HiveSessionImpl` 设置 `setAllowClose(false)` 后，`ThreadLocalHive.remove()` 中 `close(false)` 不会真正关闭 MetaStore Thrift 连接
- **影响**: 每个 HS2 会话的 MetaStore 连接在会话存活期间无法释放

#### OHL-007: SessionState.close() 清理顺序问题
- **文件**: `ql/src/java/org/apache/hadoop/hive/ql/session/SessionState.java:1914-1978`
- **根因**: `detachSession()` 在第一个 finally 块中执行，但后续的 `closeCUDFLoaders()`、`hiveDb.close()` 在 detach 之后。异常路径下 UDF ClassLoader 不会被关闭
- **影响**: 异常路径下 UDF ClassLoader 泄漏 → Metaspace 增长

#### OHL-008: Binary 模式 CLOSE_SESSION_ON_DISCONNECT=false
- **文件**: `service/src/java/org/apache/hive/service/cli/thrift/ThriftBinaryCLIService.java:163-169`
- **根因**: 当此配置为 false 时，客户端断连后 session 不关闭，所有关联的 SSL 缓冲区、SessionState、HiveConf 持续积累
- **影响**: session 数与历史连接数成正比增长

#### OHL-009: Beeline addLocalJar — ClassLoader 永不关闭
- **文件**: `beeline/src/java/org/apache/hive/beeline/Commands.java:180-183`
- **根因**: 每次 `!addlocal jar` 创建新 URLClassLoader 并设为上下文 CL，**旧 ClassLoader 从不关闭**
- **影响**: 每次 addLocalJar 泄漏一个 ClassLoader → Metaspace 增长

#### OHL-010: Deflater 原生内存未释放
- **文件**: `hcatalog/core/src/main/java/org/apache/hive/hcatalog/mapreduce/InputJobInfo.java:173`
- **根因**: `new Deflater(Deflater.BEST_COMPRESSION)` 通过 JNI 分配原生内存，但 `def.end()` 从未被显式调用
- **影响**: 每次调用泄漏数 KB 原生内存

#### OHL-011: SSL/TLS — 每连接 SSLEngine DirectByteBuffer
- **文件**: `service/src/java/org/apache/hive/service/cli/thrift/ThriftHttpCLIService.java:149-185`
- **根因**: 每个 SSL 连接的 SSLEngine 分配 16-64KB DirectByteBuffer，空闲连接的缓冲区延迟释放

## 6. 推荐代码修复清单

### Fix 1: GlobalWorkMapFactory — 添加 ThreadLocal remove()

**文件**: `ql/src/java/org/apache/hadoop/hive/ql/exec/GlobalWorkMapFactory.java`

```java
// 在 Driver.releaseResources() 或查询结束回调中添加:
public static void clearThreadLocalWorkMap() {
    Map<Path, BaseWork> map = threadLocalWorkMap.get();
    if (map != null) { map.clear(); }
    threadLocalWorkMap.remove();
}
```
**调用位置**: `Driver.java` 的 `releaseResources()` 方法末尾

### Fix 2: ThreadWithGarbageCleanup — 移除 finalize() 依赖

**文件**: `service/src/java/org/apache/hive/service/server/ThreadWithGarbageCleanup.java`

```java
// 改为在 ThreadPoolExecutor 的 afterExecute 中清理
@Override
protected void afterExecute(Runnable r, Throwable t) {
    super.afterExecute(r, t);
    Thread thread = Thread.currentThread();
    if (thread instanceof ThreadWithGarbageCleanup) {
        ((ThreadWithGarbageCleanup) thread).cleanRawStore();
    }
}
```

### Fix 3: BuddyAllocator — 实现 Closeable 接口

**文件**: `llap-server/src/java/org/apache/hadoop/hive/llap/cache/BuddyAllocator.java`

```java
public class BuddyAllocator implements EvictionAwareAllocator, BuddyAllocatorMXBean, Closeable {
    @Override
    public void close() throws IOException {
        for (Arena arena : arenas) {
            if (arena.data != null) {
                if (arena.data.isDirect()) { CleanerUtil.freeBuffer(arena.data); }
                if (arena.data instanceof MappedByteBuffer) { CleanerUtil.freeBuffer(arena.data); }
            }
        }
    }
}
```
**同时修改**: `LlapIoImpl.close()` 中添加 `buddyAllocator.close()` 调用

### Fix 4: CleanerUtil — 添加 fallback 释放机制

**文件**: `common/src/java/org/apache/hive/common/util/CleanerUtil.java`

```java
public static void freeBuffer(ByteBuffer buffer) {
    if (UNMAP_SUPPORTED) {
        CLEANER.freeBuffer(buffer);
    } else {
        try {
            if (buffer.isDirect()) {
                Method cleanerMethod = buffer.getClass().getMethod("cleaner");
                cleanerMethod.setAccessible(true);
                Object cleaner = cleanerMethod.invoke(buffer);
                if (cleaner != null) {
                    Method cleanMethod = cleaner.getClass().getMethod("clean");
                    cleanMethod.invoke(cleaner);
                }
            }
        } catch (Exception e) {
            LOG.warn("Failed to free DirectByteBuffer via fallback, relying on GC", e);
        }
    }
}
```

## 8. 预防措施

1. **监控告警**: RSS 超过 `Xmx * 1.5` 时告警
2. **定期 NMT 快照**: 每天自动执行 `VM.native_memory summary` 并对比
3. **Code Review 规则**: ThreadLocal 必须有 remove()、DirectByteBuffer 必须有 freeBuffer()、Deflater/Inflater 必须在 finally 中 end()、URLClassLoader 必须有 close()
4. **升级 Java 版本**: Java 17+ 对 DirectByteBuffer 的 Cleaner 机制更可靠
