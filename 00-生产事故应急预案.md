# 大数据生产事故应急预案

> 遇到故障 → 按预案执行 → 分钟级恢复

---

## 一、应急等级定义

| 等级 | 影响 | 响应时间 | 恢复目标 |
|------|------|---------|---------|
| **P0 — 集群不可用** | 所有任务失败/数据丢失风险 | 5 分钟 | 30 分钟恢复 |
| **P1 — 核心功能受损** | ETL 延迟/查询失败 | 15 分钟 | 2 小时恢复 |
| **P2 — 性能下降** | 任务变慢/资源紧张 | 1 小时 | 24 小时恢复 |

---

## 二、P0 级应急预案

### 2.1 HDFS NameNode 双挂（集群不可写）

```
影响: 整个 HDFS 不可用 → 所有依赖 HDFS 的组件瘫痪
应急:
  1. 检查 NN 进程 → 尝试重启 Active NN
  2. Active 重启失败 → 手动 failover: hdfs haadmin -failover nn1 nn2
  3. 双 NN 都起不来 → 检查 JournalNode (3 个中 2 个必须存活)
  4. JN 也挂 → 从最新 fsimage 恢复 (最后手段)
  5. 恢复后: hdfs fsck / 检查数据完整性
回滚: 无
预防: NN 部署在不同机架; JN 独立部署
```

### 2.2 YARN RM 双挂（无法提交任务）

```
应急:
  1. 检查 RM 进程 → 重启 Active RM
  2. 检查 ZK → RM HA 依赖 ZK
  3. Work-Preserving Recovery: RM 重启后运行中的应用不受影响
  4. 若 Recovery 失败 → AM 重新启动 → 任务可能需要重跑
回滚: 无
预防: RM HA + Work-Preserving Recovery 必开
```

### 2.3 ZK 集群不可用（多组件受影响）

```
影响: HDFS HA / YARN HA / HBase / Kafka / Kyuubi 全部受影响
应急:
  1. 检查 ZK 节点存活数 (3 节点需 2 个, 5 节点需 3 个)
  2. 重启挂掉的 ZK 节点 → systemctl restart zookeeper
  3. 如果数据损坏 → 从其他存活节点同步: zkServer.sh start-foreground
  4. 全部损坏 → 从快照恢复 (最后手段)
回滚: 无
预防: 5 节点集群; ZK 独立部署; SSD 存事务日志
```

### 2.4 Kerberos KDC 挂了（认证全断）

```
影响: 新的认证请求全部失败; 已有 Ticket 的服务在 Ticket 过期前正常
应急:
  1. 重启 KDC: systemctl restart krb5kdc kadmin
  2. KDC 数据损坏 → 从 Slave KDC 恢复
  3. 临时: 延长 Ticket 有效期争取恢复时间
注意: 不要恐慌——已有 Ticket 的服务在 10h 内不受影响
预防: 部署 Slave KDC; 定期备份 KDC 数据库
```

## 三、P1 级应急预案

### 3.1 HiveServer2 全部挂掉

```
应急:
  1. 检查 HS2 进程 + OOM 日志
  2. 重启 HS2: hive --service hiveserver2 &
  3. 如果 OOM → 调大堆内存后重启
  4. 检查 ZK 上的 HS2 注册: /hiveserver2
  5. 通知用户重连
预防: 多 HS2 实例 + ZK 服务发现; 设连接数上限
```

### 3.2 Flink 作业全部失败

```
应急:
  1. 检查 JM 日志 → 确认是 JM 挂了还是 TM 挂了
  2. JM 挂 → 重启 JM → 作业从 Checkpoint 自动恢复
  3. TM 全挂 → 检查 YARN/K8s 是否有节点故障
  4. Checkpoint 损坏 → 从最近的 Savepoint 恢复
  5. 检查消费延迟 → 恢复后追赶数据
预防: 定期做 Savepoint; 算子必须设 UID
```

### 3.3 Spark 任务大面积失败

```
应急:
  1. yarn application -list → 查看失败应用 diagnostics
  2. 常见原因: HDFS 不可用 / NM 掉线 / 队列资源不足
  3. 底层服务恢复后 → 重新提交任务
  4. 如果是代码 Bug → 回滚到上个版本
预防: 任务有重试机制; 关键 ETL 有备用路径
```

## 四、P2 级应急预案

### 4.1 NameNode RPC 慢（查询/写入变慢）

```
应急:
  1. 检查 CallQueueLength → 开启 FairCallQueue
  2. 检查 FSNamesystemLock 等待 → 定位大目录操作
  3. 检查 GC 日志 → NN JVM 调优
  4. 临时: 杀掉占大量 RPC 的应用
预防: FairCallQueue 必开; 大目录治理; NN 堆内存充足
```

### 4.2 YARN 队列资源紧张

```
应急:
  1. 查看队列使用率: yarn queue -status queueName
  2. 杀掉低优先级任务: yarn application -kill appId
  3. 临时调大队列 max-capacity
  4. 通知用户延迟提交
预防: 队列规划留 buffer; 设应用数上限; 开抢占
```

### 4.3 磁盘即将满

```
应急:
  1. 清理 Trash: hdfs dfs -expunge
  2. 清理 /tmp 下的临时文件
  3. 清理 Spark shuffle 文件: find /data -name "blockmgr-*" -mtime +3 -delete
  4. 清理旧日志: find /var/log -name "*.log.gz" -mtime +30 -delete
  5. Iceberg: expire_snapshots + remove_orphan_files
预防: 磁盘 > 70% 告警; 定期清理策略; 容量规划
```

  4. 恢复后 24 小时内完成复盘
